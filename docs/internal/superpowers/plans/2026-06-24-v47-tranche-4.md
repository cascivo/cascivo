# v47 — Tranche 4: Edit & Controlled-Sync Integration + Cache Rework

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make **edits cheap**. With T2/T3 in place, the view path already tokenizes only the visible window via
`tokenizeRange` over a `useRef`-held `LineStateIndex`. T4 wires that index to **every edit path** so an edit
invalidates the index **only from the first changed line** and the next render lazily re-threads until the grammar
state reconverges — instead of rebuilding from line 0. Then rework the bounded per-line memo (`MAX_CACHE = 5000`)
so no path depends on a 5,000-entry cap for steady-state speed or correctness. Result: keystroke cost on a
50k-line document is bounded by the **changed suffix**, not the document length (today ~587 ms). Addresses
master-plan findings **keystroke latency** and the **`MAX_CACHE` cliff**.

**Architecture:** All edits funnel through `commit(next, sel, coalesce)` in `code-editor.tsx` — typing (`onChange`),
indent/dedent (`commitFromCommand`), find/replace (`replaceCurrent`/`replaceAllNow`), and the imperative
`applyEdit`. A separate external/controlled-value `useSignalEffect` detects when `text` changed to something not
self-emitted (parent reset / remote pull) and runs `diff(prevText, next)` + `rebaseSelection`. Undo/redo go through
`applySnapshot`. T4 adds one invalidation site per path: convert the change's char offset (from `diff(prev, next)`
in `sync.ts`) to a **line number** and call `index.invalidateFrom(line)`. The index stores one small end-state per
line; `tokenizeRange` re-threads from the invalidated line until a recomputed end-state equals the stored one
(reconvergence) or the window bottom is reached — token arrays are only built for the window. With the index now
driving correctness/threading, the `MAX_CACHE` cap is what causes the cliff: **remove it** — the index supersedes
the memo for the visible window, so memory is O(distinct visible lines) (one end-state per line in the index, plus
token arrays only for the rendered rows). Programmatic `value` swaps re-seed via `invalidateFrom(0)`.

**Tech Stack:** owned TypeScript in `@cascivo/editor` (no new runtime deps); `@cascivo/core` signals; the
`LineStateIndex` held in a `useRef` (mutable infra, like the v46 history handle); `sync.ts`'s `diff` for the
char→line conversion; the test-only lines-tokenized counter from T1; Vitest + Testing Library. Signals only —
`useSignalEffect` for DOM, `useRef` for the index/handle, `useSignals()` first; no `useState`/`useEffect`.

### Task 1: Map every edit entry point + the diff→line conversion

**Files:** `packages/editor/src/editor/code-editor/code-editor.tsx`, `../../engine/tokenize.ts`, `./sync.ts`

- [ ] **Step 1:** Enumerate the text-change sites in `code-editor.tsx`: `commit(next, sel, coalesce)` (the single
      funnel for typing `onChange`, `commitFromCommand` indent/dedent, `replaceCurrent`/`replaceAllNow`, and the
      imperative `applyEdit`), `applySnapshot` (undo/redo), and the external-change `useSignalEffect`. Confirm each
      knows the *previous* text (`prevTextRef.current`) at the moment of change.
- [ ] **Step 2:** Confirm the T2/T3 index lives in a `useRef` (`indexRef`) and exposes `invalidateFrom(line)`. The
      next render's `tokenizeRange` re-threads from the lowest invalidated line until reconvergence. No edit path
      may rebuild from line 0 except a controlled full swap.
- [ ] **Step 3:** Define the **char→line** conversion: for a `Change` from `diff(prev, next)`, the first changed
      line is `prev.slice(0, change.from).split('\n').length - 1`. This is the line passed to `invalidateFrom`.
      Keep it a tiny local helper (or inline) — no new module.

### Task 2: Wire `invalidateFrom` into every edit path

**Files:** `packages/editor/src/editor/code-editor/code-editor.tsx`

- [ ] **Step 1:** In `commit`, before `setText(next)`, compute `change = diff(prevTextRef.current, next)`, convert
      to the first changed line, and call `indexRef.current.invalidateFrom(line)`. This covers typing, indent/dedent,
      find/replace, and `applyEdit` — they all route through `commit`. (Order matters: `prevTextRef.current` is
      still the old text here, since `commit` updates it after.)
- [ ] **Step 2:** In the external-change `useSignalEffect`, it already computes `change = diff(prevTextRef.current,
      next)`. Add `indexRef.current.invalidateFrom(firstChangedLine(change))` so a remote pull / parent reset
      re-threads only from the changed line — not the whole document.
- [ ] **Step 3:** In `applySnapshot` (undo/redo), the snapshot text can differ from the current text anywhere.
      Compute the change against the live text (`prevTextRef.current` before overwrite) and `invalidateFrom` the
      first changed line so restored highlighting matches the restored text.
- [ ] **Step 4:** Programmatic controlled `value` swap (whole-document replacement): when `diff` reports a change
      at line 0 (or the texts share no prefix), `invalidateFrom(0)` is the natural result of Step 2 — verify it
      re-seeds the index lazily (window-scoped) rather than walking all lines eagerly. No separate `reset` call
      unless the index API requires one; if so, prefer `invalidateFrom(0)` for a uniform path.

### Task 3: Rework / remove `MAX_CACHE` in `tokenize.ts`

**Files:** `packages/editor/src/engine/tokenize.ts`

- [ ] **Step 1:** Remove the `MAX_CACHE = 5000` cap and its eviction branch in `tokenize()`. The index now owns
      cross-line threading and the window owns how many token arrays exist; the per-line memo only needs to reuse
      tokens for lines within the current window. Drop the approximate-LRU `delete`/`set` recency dance if it no
      longer serves a bounded cache.
- [ ] **Step 2:** Keep memory bounded by **scope, not by cap**: `tokenizeRange` only ever builds token arrays for
      `[from, to)` (the window), and the index stores one small end-state per line. Document this in the cache
      doc-comment — the cap is gone because the index supersedes it; memory is O(distinct visible lines), not
      O(document). Update `clearTokenizeCache` if its contract changed.
- [ ] **Step 3:** Confirm `tokenizeDocument` still works unchanged (it remains the T2 oracle and the public
      whole-document API). Its threading loop and per-line memo calls are untouched aside from the cap removal.
