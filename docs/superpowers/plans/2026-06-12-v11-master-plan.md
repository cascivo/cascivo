# v11 Master Plan — The Open Ecosystem (Registry Protocol, Directory, Differentiators)

> **For agentic workers:** This is the umbrella document. Implement tranche by tranche:
> `2026-06-12-v11-tranche-1.md` … `2026-06-12-v11-tranche-7.md`, in order. Each tranche uses
> superpowers:subagent-driven-development or superpowers:executing-plans.

**Goal:** Execute `docs/ROADMAP-V11.md` — open the registry into a protocol third parties
can publish to (versioned item schema, static per-item build, multi-registry CLI with
namespaces/GitHub addressing/lockfile/three-way-merge updates), launch a curated + verified
directory with docs page and starter template, migrate overlays to the Popover API + CSS
anchor positioning, queue the gap components through the factory, add Standard Schema form
validation, ship DTCG token export/import, `llms-full.txt`, a generated a11y conformance
page, and the "why cascade" positioning page.

**Architecture:** One new package (`@cascivo/registry` — shared schema, types,
validation, build logic used by CLI, MCP, scripts and CI). Everything else extends existing
surfaces: `packages/cli` (commands + config + lockfile), `packages/mcp` (registry fetch +
new tools), `scripts/registry` + `scripts/llms` (generation), `apps/docs` (Directory,
conformance, agent-guide pages), `packages/components` (overlay migration, data-table
virtualization, form adapter), `packages/tokens` (DTCG export), `factory-backlog.json`
(gap component specs).

**Tech stack:** unchanged — React 18+, modern CSS with `--cascivo-*` tokens, Preact signals,
vitest + @testing-library/react, vp (vite+) toolchain. New CLI-only dep allowed for
three-way merge (e.g. `diff3`-style algorithm) — CLI is not a runtime package.

### cascade today (codebase inventory)

- **`registry.json`** (repo root, 106 entries): `{ version, generatedAt, components[] }`;
  entries carry `name`, `type: 'component'|'layout'|'block'|'chart'|'section'`,
  `description`, `category`, `version` (package-level, NOT per-item), `files` (GitHub raw
  URLs; empty for charts), `install?`, `dependencies`, `tags`, `meta` (full
  `ComponentMeta`: states/variants/sizes/props/tokens/accessibility/examples). Generated by
  `scripts/registry/generate.ts` (`pnpm registry:generate`, `REGISTRY_BASE_URL` env).
- **CLI** (`packages/cli/src/`): commands `init`, `add`, `list`, `update`, `theme`,
  `generate`, `doctor`. Config `cascade.config.ts` = `{ registry: string, outputDir,
theme }` (`utils/config.ts`, env overrides `CASCADE_REGISTRY`, `CASCADE_OUTPUT_DIR`).
  `add` fetches the single configured registry URL, `findComponent` does exact + suffix
  match, files written to `outputDir/<name>/`, deps auto-installed (`utils/exec.ts`).
  `update` = fetch latest + `diffLines` + confirm + overwrite — **two-way against current
  main, no base version, no lockfile**. **No namespaces, no multi-registry, no item
  versioning anywhere.**
- **MCP** (`packages/mcp/src/`): tools `list_components`, `get_component`,
  `search_components`, `add_to_project` (shells `npx -y cascade add`), `create_theme`
  (`theme.ts` → `generateThemeCss`, oklab color-mix), `validate_view`, `scaffold_view`,
  deprecated `scaffold_page`. Registry loaded **from filesystem only**
  (`registry.ts: resolveRegistryPath` — arg, `CASCADE_REGISTRY_PATH`, bundled, monorepo
  root). No HTTP fetch, no third-party registries.
- **Docs** (`apps/docs`, Preact): nav derives from registry (`src/nav.ts: buildNav()`),
  pages in `src/pages/` (Home, ComponentPage, ChartsPage, LayoutsPage, AiPage, Benchmarks,
  PlaygroundPage). `scripts/llms/generate.ts` → `public/llms.txt` + `public/llms/<name>.md`
  (106 files). **No llms-full.txt, no directory page, no a11y conformance page.** Deployed
  to Cloudflare Pages (`.github/workflows/cf-pages.yml`) at cascade-ui.dev; the view schema
  is already published at `cascade-ui.dev/view.v1.json` — precedent for schema hosting.
- **Components**: 60 components incl. `form` (`createForm`/`useForm`, signal fields,
  sync/async validation — **no Standard Schema support, no i18n messages**), `data-table`
  (sort/filter/pagination/selection, `content-visibility` — **no windowing**),
  `date-picker`/`time-picker` (native input wrappers — **no calendar grid, no range**),
  `combobox`, overlays (`tooltip`, `popover`, `dropdown`, `hover-card`, `menu`,
  `context-menu`, `command-menu`, `modal`, `sheet`, `alert-dialog`, `toast`). **Missing

  authors, no registry template.**

### shadcn registry mechanics (what we adopt, verified against ui.shadcn.com docs)

- Two schemas with published URLs (`/schema/registry.json`, `/schema/registry-item.json`);
  index supports `items` or `include` (composition); item names unique per registry.
- Item fields worth mirroring: `type` taxonomy, `author`, `dependencies` (npm, pinnable),
  `registryDependencies` (bare = official, `@ns/name`, URL, `owner/repo/item#ref` —
  resolved recursively across registries), per-file `target` paths, `docs` post-install
  message, `categories`, `meta`.
- `shadcn build` → flat static per-item JSON in `public/r/` — any static host is a
  registry; Vercel ships a registry-template repo.
- Namespace config: `registries` map in `components.json` — string URL template with
  `{name}` placeholder, or object `{ url, headers, params }` with `${ENV}` expansion; the
  substrate for private/paid registries. 401/403/429 server messages surfaced.
- GitHub addressing: `owner/repo/item#ref` (branch/tag/SHA) against `registry.json` at repo
  root; public repos only.
- Directory: PR'd JSON in the shadcn repo + CI validation; published at `/r/registries.json`
  (~300 namespaces, mid-2026); listed namespaces pre-configured in the CLI; flat-structure +
  no-inline-content requirements; explicit "review code yourself" disclaimer (their trust
  gap = our verified-tier opening).
- Their structural weakness we attack: **no item versioning** — "the latest shadcn CLI
  version has nothing to do with component versioning"; `diff` + hand-merge is the only
  update path (discussions #7170, #1467).

### User-research headlines that shaped tranche scope

- State of React 2025: styling/customization difficulty #1 (34 mentions); Tailwind 17 pain
  mentions; form complexity 9; 33% use **no** component library.
- Complex components are paywalled elsewhere (MUI X Pro grid/date-range) — free coverage is
  an adoption lever.
- Popover API Baseline since 2025-01; anchor positioning Interop 2025, **not** yet
  Baseline in Firefox/Safari → progressive enhancement mandatory.
- DTCG tokens first stable 2025.10 (Figma/Style Dictionary/Tokens Studio implementing).
- EAA enforced 2025-06-28 (WCAG 2.1 AA) — per-component conformance is unowned territory.
- AI-prototyping users' #1 frustration: "can't bring my own design system" to v0/Lovable.

## Cross-cutting rules (every tranche)

1. **Back-compat is a test, not a hope.** The monolithic `registry.json` shape does not
   change; `apps/docs/src/data.ts`, `packages/render/src/validate.ts`, and pre-v11
   `cascade.config.ts` files (no `registries` key) must keep working — covered by tests in
   T1/T2.
2. **One schema source.** Nothing outside `@cascivo/registry` re-declares registry
   types; CLI/MCP/scripts/CI import it.
3. **House component rules bind all component work** (T4/T5): signals not hooks,
   `useSignalEffect` for DOM side effects, CSS pseudo-classes for visual states, tokens
   only, logical properties, i18n builtin catalog for chrome strings, manifests updated,
   `packages/react` exports maintained.
4. **Progressive enhancement, never regression** (T4): every overlay must pass its existing
   test suite unmodified in the fallback path; `@supports` gates all anchor-positioning
   CSS.
5. **Network code is testable offline.** CLI/MCP registry fetch goes through one injectable
   fetch wrapper; tests use local fixture registries (no live GitHub calls in CI).
6. **Security posture for third-party code:** `cascade add` from a non-directory registry
   prints a "review before use" notice with the resolved file list; directory-verified
   namespaces note their CI validation. Never auto-execute fetched code at install time.
7. **Workspace alias rule** (CLAUDE.md): if `@cascivo/registry` exports point at
   `./dist/`, add source aliases to docs/landing/storybook vite configs.
8. **Gate before committing** (CLAUDE.md): `pnpm exec vp check` → `pnpm build` →
   `pnpm exec vp run -r check` → `pnpm test` → `pnpm registry:generate && pnpm
readme:generate && pnpm llms:generate` → `pnpm exec vp check --fix` →
   `git diff --exit-code`.
9. **Surgical changes** — the CLI especially is small and tested; extend, don't rewrite.

## Edge cases / risks registry

1. **Schema design lock-in (T1):** `schemaVersion` field from day one; additive evolution
   only; validators accept unknown fields with a warning, never a hard fail.
2. **Three-way merge correctness (T2):** the merge base must be reconstructable — lockfile
   stores the installed version, and per-item versioned JSON must remain fetchable
   (`r/<name>@<version>.json` emitted alongside latest). If a base version is gone
   (third-party registry pruned it), degrade to today's two-way diff with an explicit
   warning.
3. **Lockfile drift:** user edits files after install (the whole point of owned code) —
   hashes in the lock detect drift; `update` uses local-as-ours/upstream-as-theirs/
   base-as-base regardless, hashes only inform messaging (`local edits detected — merging`).
4. **GitHub raw rate limits + availability (T2/T3):** cache directory fetches
   (`~/.cascade/cache`), exponential backoff, clear offline error; CI directory validation
   retries and is allowed to soft-fail on network-only errors (separate status from schema
   failure).
5. **Typosquatting in the directory (T3):** namespace review is part of the PR checklist;
   CI flags Levenshtein-close namespaces against existing ones.
6. **Anchor positioning support matrix (T4):** Chromium-only as of research date; Firefox/
   Safari fall back. Eyeball matrix in T4 must cover one non-supporting engine; if vite+/
   Rolldown CSS minification mangles `anchor-name`, pin/patch or scope the syntax in
   plain CSS files (no preprocessing surprises expected — verify).
7. **Popover + focus management (T4):** native popover changes focus/light-dismiss
   semantics vs current implementations — run the full a11y test suites; menu/context-menu
   keyboard tests are the canary.
8. **Windowed data-table a11y (T5):** windowing must keep `aria-rowcount`/`aria-rowindex`
   correct and not break selection; `content-visibility` path remains the default,
   `virtualized` is opt-in.
9. **Standard Schema typing (T5):** the `~standard` interface is TS-heavy; keep the
   adapter's public surface minimal (`schema?: StandardSchemaV1` on field/form config) and
   test against all three libraries as dev deps of the components package only.
10. **DTCG fidelity (T6):** cascade tokens include non-DTCG concepts (e.g. composite
    shadows, oklch strings) — map what the spec covers, emit `$extensions["dev.cascade"]`
    for the rest; round-trip test: export → import → theme CSS compiles.
11. **MCP HTTP fetch (T3):** keep filesystem-first resolution for the bundled registry
    (offline default); HTTP only for explicitly configured/directory registries; timeouts +
    size caps on fetched JSON.
12. **Factory backlog collisions (T5):** 28 items sit in `review` — new v11 specs must not
    duplicate in-flight names; check before queueing.
