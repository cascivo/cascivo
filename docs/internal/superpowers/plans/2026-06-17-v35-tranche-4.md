# v35 — Tranche 4: Render-Blocking CSS

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Eliminate or reduce render-blocking CSS resources to improve FCP/LCP, particularly on mobile (FCP 3.23s, LCP 3.48s). Two resources block rendering:

| Resource               | Size     | Blocking delay (desktop / mobile) |
| ---------------------- | -------- | --------------------------------- |
| `index-SawEnPAu.css`   | 16 508 B | 233ms / 782ms                     |
| `spinner-BN0steAw.css` | 858 B    | 73ms / 182ms                      |

Both are injected as `<link crossorigin href="...">` tags by Vite, without any non-blocking strategy.

**Strategy:**

- **Spinner CSS** (858 B): inline into `<head>` — eliminates the HTTP round-trip and the blocking behavior with zero FOUC risk.
- **Main CSS** (`index-SawEnPAu.css`, 16KB): add `<link rel="preload" as="style">` to overlap its download with JS parsing. Full deferral (media=print trick) risks FOUC for above-fold components; do not use it without critical-CSS extraction (out of scope — no Playwright at build time). If `rel="preload"` alone is insufficient, document the limitation.

### Task 2: Inline spinner CSS into `index.html`

- [ ] **Step 1:** Read the spinner CSS source file identified in Task 1. Copy its content (minify by hand or use the output from `pnpm exec vp run @cascivo/landing#build` which should have minified it).

- [ ] **Step 2:** In `apps/landing/index.html`, find the `<link ... href="/src/landing.css" ...>` block. Add a `<style>` tag in `<head>` containing the spinner CSS content. Place it before the main entry script tag.

  Example placement:

  ```html
  <head>
    <!-- ... existing tags ... -->
    <style>
      /* spinner CSS inlined here */
    </style>
    <!-- ... script tags ... -->
  </head>
  ```

- [ ] **Step 3:** Configure Vite to stop emitting `spinner.css` as a separate CSS chunk. Options:
  - If the spinner CSS comes from a CSS module import, Vite auto-chunks it. Check if `build.cssCodeSplit: false` consolidates it (may have side effects — evaluate).
  - Alternatively, import the spinner CSS directly in `index.html` via a `<style>` containing the raw CSS, and remove the JS-side import. This requires finding the import site.
  - Simplest safe option: accept that Vite still emits the chunk but also add the inline `<style>` in `index.html`. The browser will use the inline style (faster) and ignore or cache the external one. The Lighthouse violation disappears because the render-blocking `<link>` is removed from `index.html` (the prerendered shell) OR because the inline style satisfies the need before the `<link>` is reached.

  **Preferred approach:** Remove the `<link>` for `spinner-BN0steAw.css` from the HTML template and include the spinner styles inline in `<head>`. If the Vite-emitted chunk is still referenced by JS, it will still load (cached) but won't block the initial render.

- [ ] **Step 4:** Build and verify:
  ```bash
  pnpm exec vp run @cascivo/landing#build
  ```
  Inspect `apps/landing/dist/index.html`. Confirm `spinner-BN0steAw.css` is no longer a render-blocking `<link rel="stylesheet">` in the `<head>`. Confirm the spinner still renders correctly in the Suspense fallback.

### Task 3: Add `rel="preload"` for main CSS

The main CSS (`index-SawEnPAu.css`) must remain a stylesheet `<link>`, but preloading it overlaps the download with JS evaluation, reducing the effective blocking delay.

- [ ] **Step 1:** In the built `dist/index.html`, note the exact filename of `index-SawEnPAu.css`. This is a content-hashed name that changes on rebuild. Vite typically injects `<link rel="modulepreload">` for JS chunks; it does not automatically add `<link rel="preload" as="style">` for CSS.

- [ ] **Step 2:** Add a Vite plugin or `transformIndexHtml` hook in `apps/landing/vite.config.ts` that adds `<link rel="preload" as="style">` for the main CSS chunk. The plugin needs to run in the `closeBundle` / `transformIndexHtml` phase when hashed filenames are known.

  Example `transformIndexHtml` hook:

  ```ts
  transformIndexHtml(html, ctx) {
    if (!ctx.bundle) return html
    // Find the main CSS chunk filename
    const cssChunk = Object.keys(ctx.bundle).find(k => k.endsWith('.css') && k.includes('index'))
    if (!cssChunk) return html
    const preload = `<link rel="preload" as="style" href="/${cssChunk}">`
    return html.replace('<head>', `<head>\n    ${preload}`)
  }
  ```

  Adjust path prefix (`/assets/` may be needed depending on `base` config).

- [ ] **Step 3:** Build and verify:
  ```bash
  pnpm exec vp run @cascivo/landing#build
  ```
  Inspect `dist/index.html`. Confirm a `<link rel="preload" as="style" href="/assets/index-*.css">` appears in `<head>`, before the blocking `<link rel="stylesheet">`.
