# v57 — Tranche 4: Tailwind v4 Interop — Doc + Thin Opt-in Bridge

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close the report's "theme collision with Tailwind" gap. cascivo themes key on `[data-theme='dark']` inside
`@layer cascivo.theme`; Tailwind v4 keys dark mode on a `.dark` class and emits its own (often unlayered) `--color-*`
tokens — so the two systems use different dark-mode selectors and two unmapped token namespaces, with **zero**
guidance today. Ship a single interop page plus a thin, **opt-in** CSS bridge so a Tailwind project keeps its `.dark`
toggle and reads cascivo tokens — without cascivo moving off `[data-theme]` or depending on Tailwind (M-4, Master
Decision 5).

**Architecture:** one doc + one opt-in stylesheet; no change to either token system.
- **`docs/USING-WITH-TAILWIND.md`** (mirrors `docs/USING-WITH-PREACT.md`): (1) the canonical cross-system `@layer`
  order so neither side's tokens get clobbered; (2) a one-line **dark-mode bridge** rule so `.dark` also activates
  cascivo's dark theme; (3) a **token bridge** via Tailwind v4 `@theme inline` mapping `--cascivo-*` semantics →
  Tailwind `--color-*`, so `bg-accent`-style utilities resolve to cascivo tokens; (4) a "keep both / pick one"
  decision guide.
- **`@cascivo/themes/tailwind.css`** (new, opt-in): ships the bridge so consumers `@import` one file instead of
  copy-pasting. Contains the `.dark → [data-theme='dark']` activation and the `@theme inline` token map. Pure CSS,
  loaded *after* the cascivo theme imports; a no-op for non-Tailwind projects who never import it.

**Tech Stack:** `packages/themes/src/` (CSS only), `docs/`. No JS, no Tailwind dependency, no token-system change.
