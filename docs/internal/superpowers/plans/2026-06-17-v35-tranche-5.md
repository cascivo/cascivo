### Task 1: Investigate the CSS minification gap

**Context:** `landing-C4hGJ04s.css` is the output of `apps/landing/src/landing.css` (the non-blocking progressive-enhancement stylesheet). It is 10,983 bytes with ~2,413 bytes saveable (22% potential reduction). Rolldown's built-in CSS minifier should be handling this. The gap suggests minification is either disabled for this bundle or the output CSS contains whitespace/comments that Rolldown doesn't strip.

- [ ] **Step 1:** Check the current `build.cssMinify` setting in `apps/landing/vite.config.ts`. If absent, Vite defaults to `'esbuild'` for the CSS minifier.

- [ ] **Step 2:** Check if vite+ (`vp`) overrides `cssMinify`. Run:

  ```bash
  pnpm exec vp run @cascivo/landing#build
  cat apps/landing/dist/assets/landing-*.css | wc -c
  ```

  Compare the output size to the Lighthouse-reported 10,983 bytes. If the built file is already smaller, the MIME fix in T1 may have changed which CSS is considered "wasted."

- [ ] **Step 3:** If `landing-C4hGJ04s.css` is still not fully minified, try setting `build.cssMinify: 'lightningcss'` in `vite.config.ts` (Vite 5+ / vite+ supports Lightning CSS as the CSS minifier). Lightning CSS is significantly more aggressive than esbuild's CSS minification.

  ```ts
  export default defineConfig({
    // ...
    build: {
      cssMinify: 'lightningcss',
    },
    // ...
  })
  ```

  If vite+ (`vp`) does not support `'lightningcss'` yet (check the alpha changelog at https://viteplus.dev), fall back to verifying esbuild is actually running. Check whether the 22% gap is from comments in the CSS that esbuild preserves.

- [ ] **Step 4:** After any changes, run `pnpm exec vp run @cascivo/landing#build`. Verify `landing-*.css` output size has decreased. Run `pnpm ready`.

### Task 2: Source maps — document the decision

**Context:** The `valid-source-maps` audit flags that `index-BMCbo8TB.js` (175KB) is missing a source map. This is a diagnostic-only audit — it does not affect the best-practices score in Lighthouse's current scoring rubric. The audit is advisory.

Enabling `build.sourcemap: true` in Vite would fix this, but:

- Doubles the transferred JS size (public source maps)
- Exposes full source code to anyone inspecting the site
- The correct production approach is private source-map upload to an error monitoring service (Sentry, Datadog, etc.)

- [ ] **Step 1:** Add a comment in `apps/landing/vite.config.ts` near the `build` section explaining the decision:

  ```ts
  // Source maps: not enabled for public deployment — doubles JS transfer size and exposes source.
  // For production debugging, upload source maps to an error monitoring service (Sentry/Datadog).
  // build.sourcemap: true  ← enable locally for debugging; do not commit
  ```

- [ ] **Step 2:** Do **not** set `build.sourcemap: true` unless a source-map upload pipeline is in place. The `valid-source-maps` audit is informational and does not affect the best-practices score.

### Task 3: Profile unused JavaScript

**Context:** `index-BMCbo8TB.js` (175KB total, 20.7KB unused = 11.8%). Lighthouse flags this as score=0 for `unused-javascript`, which means the 20.7KB exceeds the audit threshold (~20KB). The landing app already uses extensive lazy loading (15+ lazy imports in `App.tsx`). This task assesses whether further splitting is viable without complexity cost.

- [ ] **Step 1:** Run a bundle analysis. Vite has a `vite-bundle-visualizer` plugin or built-in stats. If neither is configured, use the rollup stats output:

  ```bash
  pnpm exec vp run @cascivo/landing#build 2>&1 | grep "index\|main" | head -20
  ```

  Alternatively, check `apps/landing/dist/assets/index-*.js` size and look for large dependencies that load eagerly.

- [ ] **Step 2:** Identify what is in `index-BMCbo8TB.js` that is not used on first load. Key candidates:
  - Any eagerly-imported large library (charting, syntax highlighting, etc.)
  - `@cascivo/components` modules that are imported at the top level instead of inside a `lazy(() => ...)`
  - The `registry.json` import in `RelayConsole.tsx` (loaded even when the RelayConsole section isn't visible)

- [ ] **Step 3:** For each candidate ≥ 5KB unused, evaluate whether lazy-splitting it would reduce the main bundle below 20KB unused (the audit threshold). Only split if:
  - The split is clean (no circular dependencies introduced)
  - The lazy boundary makes semantic sense (e.g., the component is below-the-fold)
  - The change is ≤ 10 lines

- [ ] **Step 4:** Apply any splits. Run `pnpm ready`. Re-run a local Lighthouse performance audit to verify the `unused-javascript` score improves.

  If no clean split is achievable, document the finding and accept the current score. The 11.8% unused rate in a 175KB bundle is within normal SPA range.
