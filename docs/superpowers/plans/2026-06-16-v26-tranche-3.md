# v26 — Tranche 3: View Transitions for Page Navigation

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add cross-document view transitions so navigating between landing pages (`/`, `/accessibility`,
`/performance`, `/guides`, `/examples`) produces a smooth CSS cross-fade in browsers that support
the View Transitions API Level 2. Browsers without support show the existing hard cut — no errors,
no fallback JavaScript.

**Architecture:** Two file changes only. A CSS `@view-transition` at-rule in `landing.css` opts the
document in. A `<meta name="view-transition">` in `index.html` is the required HTML signal. Optional:
`view-transition-name` attributes on the shell header and main content landmark so the browser can
animate named elements independently.

**Tech Stack:** CSS `@view-transition { navigation: auto }`, `view-transition-name` CSS property,
`<meta name="view-transition" content="same-origin">`. No JavaScript changes. No router changes.
Progressive enhancement only — the experience degrades gracefully to a hard cut.

### Task 2: Add `<meta name="view-transition">` to index.html

- [ ] **Step 1: Add the meta tag**

  In `apps/landing/index.html`, in the `<head>`, add after the existing viewport meta:

  ```html
  <meta name="view-transition" content="same-origin" />
  ```

  This tag tells Chrome (and future browsers) that same-origin navigations are eligible for
  cross-document view transitions even before the page fully loads.

### Task 3: Add `@view-transition` and named transition targets to landing.css

- [ ] **Step 1: Add the CSS opt-in**

  In `apps/landing/src/landing.css`, at the very top of the file (before the first rule block
  but after any existing `@layer` declarations if present), add:

  ```css
  /* ── Cross-document view transitions ────────────────────── */
  @view-transition {
    navigation: auto;
  }

  @media (prefers-reduced-motion: reduce) {
    ::view-transition-group(*),
    ::view-transition-old(*),
    ::view-transition-new(*) {
      animation: none;
    }
  }
  ```

  The `@media (prefers-reduced-motion)` block ensures users who opt out of motion see no
  animation — the transition still fires but resolves instantly.

- [ ] **Step 2: Add view-transition-name to the shell header**

  The `ShellHeader` from `@cascivo/components/shell-header` renders a `<header role="banner">`.
  In `landing.css`, target the header element and give it a stable name:

  ```css
  [role='banner'] {
    view-transition-name: shell-header;
  }
  ```

  This allows the header to appear to "stay in place" during page transitions (the browser will
  morph old → new header element rather than cross-fading the full root).

- [ ] **Step 3: Add view-transition-name to the main content area**

  The skip-nav target wraps `<main>`. Find (or add) a targeting rule:

  ```css
  main {
    view-transition-name: page-content;
  }
  ```

  With both `shell-header` and `page-content` named, the browser cross-fades the header in
  place and slides the page content.

- [ ] **Step 4: Add a slide animation for page-content**

  By default, named elements also cross-fade. For page content, a subtle slide looks more
  intentional than a fade. Add:

  ```css
  @keyframes slide-in-from-right {
    from { transform: translateX(1.5rem); opacity: 0; }
    to   { transform: translateX(0);     opacity: 1; }
  }

  @keyframes slide-out-to-left {
    from { transform: translateX(0);      opacity: 1; }
    to   { transform: translateX(-1.5rem); opacity: 0; }
  }

  ::view-transition-old(page-content) {
    animation: slide-out-to-left 200ms ease both;
  }

  ::view-transition-new(page-content) {
    animation: slide-in-from-right 200ms ease both;
  }

  @media (prefers-reduced-motion: reduce) {
    ::view-transition-old(page-content),
    ::view-transition-new(page-content) {
      animation: none;
    }
  }
  ```

  The `both` fill-mode preserves the first/last frame state so there is no flicker at the
  start and end of the transition.

### Task 4: Check for conflicts with existing view-transition usage

The landing already uses `view-transition-name: hero-stage` for the theme-change cross-fade
(in `theme.ts` and `landing.css`). Ensure:

- [ ] **Step 1: Confirm `hero-stage` is not affected by the cross-document transition**

  Scan `landing.css` for `hero-stage` usages:

  ```sh
  grep -n "hero-stage" apps/landing/src/landing.css
  ```

  Confirm the existing `::view-transition-old(hero-stage)` / `::view-transition-new(hero-stage)`
  rules are scoped to that named element and will not interfere with the new `page-content` and
  `shell-header` transitions.

- [ ] **Step 2: Confirm no name collisions**

  The names `shell-header`, `page-content`, `hero-stage` must all be unique across all elements
  on any given page (a browser constraint — duplicate `view-transition-name` values on the same
  page cause the transition to be skipped for those elements).

  ```sh
  grep -rn "view-transition-name" apps/landing/src/landing.css
  ```

  Expected: `hero-stage`, `shell-header`, `page-content` — each appears exactly once.

### Task 5: Verify the transition works

- [ ] **Step 1: Build a preview**

  Cross-document view transitions require actual page navigations (not hot-module reload).
  Build and preview:

  ```sh
  pnpm exec vp run @cascivo/landing#build
  pnpm exec vp run @cascivo/landing#preview
  ```

  (Or `pnpm exec vp preview --port 4180` inside `apps/landing`.)

- [ ] **Step 2: Navigate between pages in Chrome**

  Open the landing in Chrome 111+ (which supports cross-document view transitions). Navigate:
  - Home → Accessibility (click header link or footer link)
  - Accessibility → Performance
  - Performance → Home (click "cascivo" brand)

  Observe: the header should stay visually fixed; the page content should slide in from the
  right. The transition should last approximately 200 ms.

- [ ] **Step 3: Verify reduced-motion users see no animation**

  In Chrome DevTools → Rendering → Emulate CSS `prefers-reduced-motion: reduce`. Navigate
  between pages. Confirm the transition fires (no hard cut) but resolves instantly — no
  sliding, no fading, just an immediate switch.

- [ ] **Step 4: Verify graceful degradation in a browser without support**

  Test in Firefox (no cross-document view transitions as of 2026-06). Navigate between pages.
  Expect a hard cut — no errors in the console, no broken layouts, no flicker. Just the normal
  non-animated navigation.
