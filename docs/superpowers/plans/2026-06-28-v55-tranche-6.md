# v55 — Tranche 6: Seed Templates + Verification & Safety

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Seed the marketplace and **prove the whole loop end-to-end** (M-8), then lock down the **safety/verification**
model (M-9). Publish 3–5 first-party templates (dashboard, landing, auth, settings, docs) built from existing
components + `@cascivo/example-kit` patterns; document the `verified` review checklist; propagate advisories through
templates; and run `cascivo doctor`/`audit` over bundled template source in CI. First-party templates *derive from*
example-kit — they do **not** relocate it (Master Decision 7).

**Architecture:** two pieces.
- **Seed templates** (`templates/*`): five `type:'template'` registry items, each a real page composing shipped
  components (using example-kit's mock-data/simulation patterns for fixtures), with `light`/`dark` screenshots, a
  `*.template.ts` manifest, and a built `public/r/*.json`. Listed under a first-party `templates/cascivo-registry.json`
  whose directory entry advertises `provides:[…,'template']` (T4).
- **Verification & safety** (`CONTRIBUTING-TEMPLATES.md` + `marketplace.yml`): a `verified` review checklist (source
  reviewed for malicious code; CSS house rules; a11y spot-check; maintainer agrees to respond to security reports —
  mirroring the registry's `verified` meaning); advisories propagation (a template carrying a flagged component surfaces
  the advisory); CI runs `cascivo doctor` + `audit` over each template's bundled source (Decisions 4–5).

**Tech Stack:** the seed templates use the shipped component registry + `@cascivo/example-kit` patterns; the T1–T5
schema/CLI/catalog/gallery; `cascivo doctor`/`audit`; CI. No new runtime dependency.

### Task 1: Build the seed templates

**Files:** `templates/{dashboard,landing,auth,settings,docs}/*`

- [ ] **Step 1:** For each template, author a real page composing shipped components (e.g. dashboard = AppShell + KPI
      cards + a chart + a data table; auth = a centered card + form inputs). Use `@cascivo/example-kit` mock-data /
      simulation **patterns** for fixtures (derive, don't import the private package into installed output). Signals
      only; i18n strings; a11y; mobile sweep.
- [ ] **Step 2:** Write each `*.template.ts` `TemplateMeta` (intent/framework/category/screenshots/fileRoles/pages) and
      capture `light`/`dark` screenshots. `registryDependencies` = the exact shipped components used.
- [ ] **Step 3:** Assemble `templates/cascivo-registry.json` (the five items) and run `cascivo registry build` (T3) →
      `public/r/*.json`. Each must pass `validateTemplate` (screenshots + license present).

### Task 2: List + verify end-to-end

**Files:** `directory/registries.json`

- [ ] **Step 1:** Mark the first-party directory entry `provides:[…,'template']` so the T4 catalog builder picks up the
      seed templates; run `build-marketplace` and confirm they appear in `marketplace.json` + the gallery.
- [ ] **Step 2:** End-to-end smoke: in a throwaway app, `cascivo create --template @cascivo/dashboard` (and `cascivo
      add` each of the others) installs the component closure + the page/fixture files, the app boots into the page, and
      `cascivo doctor` reports no violations.
- [ ] **Step 3:** Test: each seed template installs cleanly; the lockfile records it; `cascivo update` sees it.

### Task 3: Safety + verification model

**Files:** `.github/workflows/marketplace.yml`, `docs/CONTRIBUTING-TEMPLATES.md`

- [ ] **Step 1:** CI: run `cascivo doctor` + `cascivo audit --ai` over each template's bundled source (the first-party
      templates gate on it; third-party PRs surface results for the reviewer). Templates are owned copy-paste source —
      never executed remotely (Decision 4).
- [ ] **Step 2:** Document the `verified` checklist in `CONTRIBUTING-TEMPLATES.md` (source reviewed for malicious code;
      CSS house rules / `@layer` scoping; a11y spot-check; keyboard nav; maintainer responsiveness) — mirroring the
      registry's `verified` meaning; unverified templates show with a warning in the gallery.
- [ ] **Step 3:** Advisories propagation: document + test that a template whose `registryDependencies` include a
      component with an active `advisory` surfaces that advisory at `view`/`add` time and in the gallery card.
