# v49 — Tranche 5: Link Integrity + Domain Unification

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stop link rot from being a manual follow-up. The Footer mixes two domains — `cascivo.com` (storybook, repo)
and `cascivo.dev` (`@cascivo/charts`, `@cascivo/layouts`) — and a code comment literally concedes "if a target ever
404s, that's a deploy-config follow-up, not a landing change." There is no CI link-check, so a broken internal route
or a dead ecosystem URL ships silently. This tranche unifies the ecosystem domain (or documents the split with the
owning deploy) and adds a link-check that fails the build on a broken internal route. Lens: **architect.**

**Architecture:** `Footer.tsx` `COLUMNS` carry internal hrefs (`/docs`, `/examples`, `/showcase`, `/guides`,
`/create`, `/blocks`, `/ai`, `/accessibility`, `/performance`, `/modern-css`, `/docs/why`, `/docs/benchmarks`,
`/llms.txt`, `/registry.json`) and external ones (`https://storybook.cascivo.com`, `github.com/cascivo/cascivo`,
`https://cascivo.dev/charts`, `https://cascivo.dev/layouts`). Internal routes are the keys of `ROUTES` in
`App.tsx` plus the dynamic `/blocks/*` and `/examples/<slug>` routes and the deploy-served `/docs`, `/storybook`,
`/llms.txt`, `/registry.json`. `Header.tsx` and `CtaBand.tsx` also carry internal hrefs. `.github/workflows` has no
link-check job today.

**Tech Stack:** a dependency-free Node script (`scripts/check-links.mjs`, Node built-ins) that statically extracts
hrefs from the marketing source (or from a generated route manifest) and asserts internal targets are known routes;
optionally pings a small pinned allow-list of external targets. Wired into CI + a root `package.json` script. No new
runtime deps.

### Task 1: Decide and apply the canonical ecosystem domain (A4)

**Files:** `apps/site/src/marketing/sections/Footer.tsx`

- [ ] **Step 1:** Determine the real, deployed home of `@cascivo/charts` and `@cascivo/layouts`. If they live on the
      same site as storybook/repo, move the links to that domain (`cascivo.com` or the canonical apex). If they
      genuinely deploy to `cascivo.dev`, **keep** the URLs but replace the vague "404 is a follow-up" comment with a
      precise one naming the owning deploy/app for each external target.
- [ ] **Step 2:** Audit every external href in the Footer (`storybook.cascivo.com`, `cascivo.dev/*`, the repo
      `tree/main/...` deep links) and confirm each resolves today. Fix or remove any that don't.
- [ ] **Step 3:** Remove the "if a target ever 404s, that's a deploy-config follow-up, not a landing change" comment
      once the link-check (Task 2) guards these — the fragility is no longer unguarded.

### Task 2: Build the link-check script (A4)

**Files:** `scripts/check-links.mjs`

- [ ] **Step 1:** Statically collect internal hrefs from the marketing source (Footer/Header/CtaBand/Hero and any
      section with links). Build the set of known internal routes from `ROUTES` in `App.tsx` plus the dynamic route
      prefixes (`/blocks/`, `/examples/`) and the deploy-served paths (`/docs`, `/storybook`, `/llms.txt`,
      `/registry.json`, `/docs/why`, `/docs/benchmarks`). Assert every internal href maps to a known route/prefix;
      fail non-zero naming any that don't.
- [ ] **Step 2:** Maintain a small pinned allow-list of external targets to ping (HEAD request, short timeout,
      tolerant of transient failures so CI isn't flaky — e.g. retry once, only hard-fail on a definitive 404). Keep
      this list tiny and high-value (the ecosystem URLs, storybook). Treat external checks as a separate, optionally
      non-blocking phase if flakiness is a concern — the internal-route check is the hard gate.
- [ ] **Step 3:** Keep the script dependency-free (Node built-ins, `node:fs`, `node:https`); no new devDeps. Clear,
      actionable failure messages ("Footer link /foo has no matching route").

### Task 3: Wire it into CI + scripts (A4)

**Files:** root `package.json`, `.github/workflows/<ci>.yml`

- [ ] **Step 1:** Add `"links:check": "node scripts/check-links.mjs"` to root `package.json`.
- [ ] **Step 2:** Add a "Link check" step to the appropriate CI workflow (next to the existing
      `breakpoint:check`/`fallback:check` checks). The internal-route assertion is blocking; decide whether the

- [ ] **Step 3:** Run `pnpm links:check` locally and confirm it passes on the current tree (after Task 1 fixes) and
