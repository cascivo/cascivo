# v26 — Tranche 5: Screenshot Pipeline — Playwright Real Captures

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the placeholder wireframe PNGs currently committed in
`apps/landing/public/screenshots/<slug>/` with real Playwright screenshots of each example app.
Add a `pnpm screenshots:capture` convenience script and a GitHub Actions workflow to regenerate
screenshots on a schedule.

**Architecture:** The capture logic already exists in `scripts/gen-demo-screenshots.mjs --capture`.
This tranche: (1) verifies it works end-to-end, (2) adds the npm script alias, (3) adds a CI
workflow, (4) optionally improves the capture (waiting for animations, dark-mode variant).

**Tech Stack:** Node.js, Playwright (`@playwright/test`), GitHub Actions.

### Task 1: Verify `--capture` works end-to-end

- [ ] **Step 1: Confirm Chromium is installed**

  ```sh
  pnpm exec playwright install chromium --with-deps
  ```

  This installs the Chromium browser that Playwright will use. The `--with-deps` flag installs
  system dependencies on Linux CI environments.

- [ ] **Step 2: Build and assemble the landing with demo apps**

  The `--capture` mode navigates to `http://localhost:<PORT>/demos/<slug>/`. The demo apps
  must be assembled into the landing `dist/` first. Run the full landing build:

  ```sh
  pnpm exec vp run @cascivo/landing#build
  ```

  After the build, confirm `apps/landing/dist/demos/` exists with subdirectories for each
  demo slug (`deploy`, `pay`, `flow`, `track`, `pulse`).

  If `dist/demos/` is empty or missing, run the demo assembly step:

  ```sh
  pnpm exec vp run @cascivo/landing#assemble-demos 2>/dev/null || \
  node apps/landing/scripts/assemble-demos.mjs 2>/dev/null || \
  echo "Check the landing build script for the assemble step"
  ```

- [ ] **Step 3: Run capture in dry-run check**

  ```sh
  SHOT_PORT=4191 node scripts/gen-demo-screenshots.mjs --capture
  ```

  The script starts a preview server on port 4191, navigates each demo, and saves PNGs to
  `apps/landing/public/screenshots/<slug>/`. Watch for:
  - Server start output (the script sleeps 4 s to let the server initialise)
  - Per-slug output: `[screenshots] deploy: 4 real screenshots`
  - Final output: no error messages

  If the script fails:
  - Check if the preview server started (port conflict?). Try a different `SHOT_PORT`.
  - Check if `/demos/<slug>/` loads correctly in the preview.
  - Check if `pnpm exec playwright install chromium` was run.

- [ ] **Step 4: Verify output quality**

  After capture, inspect the generated PNGs:

  ```sh
  ls -lh apps/landing/public/screenshots/deploy/
  ```

  Expected: four files (`light-desktop.png`, `dark-desktop.png`, `light-mobile.png`,
  `dark-mobile.png`). Each should be substantially larger than the ~4–5 KB placeholder
  wireframes — real screenshots of a React dashboard are typically 200–800 KB.

  Open one in an image viewer to confirm it shows the actual dashboard UI, not the placeholder.

### Task 2: Improve capture quality (if needed)

- [ ] **Step 1: Check for incomplete renders**

  If the screenshots show loading states or blank regions, increase the wait time in
  `scripts/gen-demo-screenshots.mjs`. Find:

  ```js
  await page.waitForTimeout(1200) // let seeded data settle
  ```

  Increase to 2500 if the demo apps take longer to initialise their simulation data.

- [ ] **Step 2: Check for theme not applying**

  If dark-mode screenshots look identical to light-mode, the `storageState` seed may not
  be working. In the capture loop, find the `localStorage` seeding:

  ```js
  localStorage: [{ name: themeKey, value: JSON.stringify(theme) }],
  ```

  Verify `themeKey` for each demo matches the actual localStorage key each example app uses.
  Open one demo manually in a browser, set its theme to dark, and inspect localStorage to
  confirm the key name (e.g., `deploy.theme` vs `cascade-theme`).

  If the key is different, update `DEMOS` in `gen-demo-screenshots.mjs`:

  ```js
  const DEMOS = {
    deploy: { accent: [0, 112, 243], themeKey: 'the-actual-key' },
    …
  }
  ```

- [ ] **Step 3: Ensure reduced-motion is set**

  Confirm the capture context includes `reducedMotion: 'reduce'`:

  ```js
  const ctx = await browser.newContext({
    viewport: { width: w, height: h },
    reducedMotion: 'reduce',
    storageState: { … },
  })
  ```

  `reducedMotion: 'reduce'` stops auto-advancing carousels and animations so the screenshot
  captures a stable first frame.

### Task 3: Add `screenshots:capture` npm script

- [ ] **Step 1: Open `apps/landing/package.json`**

  Find the `"scripts"` section.

- [ ] **Step 2: Add the script**

  ```json
  "screenshots:capture": "node ../../scripts/gen-demo-screenshots.mjs --capture",
  "screenshots:placeholder": "node ../../scripts/gen-demo-screenshots.mjs"
  ```

  `screenshots:capture` runs Playwright captures (requires Chromium). `screenshots:placeholder`
  regenerates the wireframe placeholders (no browser needed).

- [ ] **Step 3: Add a root-level alias (optional but convenient)**

  In the root `package.json`, find the `"scripts"` section. Add:

  ```json
  "screenshots:generate": "node scripts/gen-demo-screenshots.mjs",
  "screenshots:capture": "node scripts/gen-demo-screenshots.mjs --capture"
  ```

  (The root already has `"screenshots:generate"` per the earlier audit — rename/update it
  to be consistent.)

### Task 4: Add GitHub Actions workflow for scheduled screenshot capture

- [ ] **Step 1: Check existing CI workflows**

  ```sh
  ls .github/workflows/
  ```

  Note existing workflow file names so the new one does not conflict. Common names:
  `ci.yml`, `deploy.yml`, `test.yml`.

- [ ] **Step 2: Create `.github/workflows/screenshots.yml`**

  ```yaml
  name: Regenerate demo screenshots

  on:
    schedule:
      # Every Sunday at 02:00 UTC
      - cron: '0 2 * * 0'
    workflow_dispatch:
      # Also triggerable manually from the Actions tab

  jobs:
    capture:
      runs-on: ubuntu-latest
      steps:
        - uses: actions/checkout@v4

        - uses: pnpm/action-setup@v4
          with:
            run_install: false

        - uses: actions/setup-node@v4
          with:
            node-version: '22'
            cache: 'pnpm'

        - name: Install dependencies
          run: pnpm install --frozen-lockfile

        - name: Install Chromium
          run: pnpm exec playwright install chromium --with-deps

        - name: Build landing (with demo apps assembled)
          run: pnpm exec vp run @cascivo/landing#build

        - name: Capture screenshots
          run: node scripts/gen-demo-screenshots.mjs --capture

        - name: Commit updated screenshots
          uses: stefanzweifel/git-auto-commit-action@v5
          with:
            commit_message: 'chore(screenshots): regenerate demo screenshots [skip ci]'
            file_pattern: 'apps/landing/public/screenshots/**/*.png'
  ```

  The `[skip ci]` suffix prevents the screenshot commit from triggering another CI run.
  `stefanzweifel/git-auto-commit-action` handles the `git add` + `git commit` + `git push`
  automatically. If this action is not already in use, the alternative is to add explicit
  git steps:

  ```yaml
  - name: Commit updated screenshots
    run: |
      git config user.name "github-actions[bot]"
      git config user.email "github-actions[bot]@users.noreply.github.com"
      git add apps/landing/public/screenshots/
      git diff --staged --quiet || git commit -m "chore(screenshots): regenerate [skip ci]"
      git push
  ```

- [ ] **Step 3: Add a comment at the top of the workflow**

  ```yaml
  # Regenerates the landing's demo screenshots weekly.
  # Triggered automatically on Sundays or manually via workflow_dispatch.
  # Run locally: node scripts/gen-demo-screenshots.mjs --capture
  ```

### Task 6: Gate

- [ ] **Step 1: Run the full gate**

  ```sh
  pnpm exec vp check
  pnpm build
  pnpm exec vp run -r check
  pnpm test
  pnpm regen && pnpm exec vp check --fix && git diff --exit-code
  pnpm breakpoint:check
  ```

  All six must exit 0.

  Note: the drift check (`pnpm regen && git diff --exit-code`) should pass even with the new
  screenshot PNGs, because `regen` does not touch binary assets in `public/screenshots/`.
  If it fails due to the PNG files being included in the diff, add the `public/screenshots/`
  path to the regen exclusion list or `.gitattributes` (binary files should not be diffed).
