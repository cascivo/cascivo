**Goal:** Let a chart print its own values â€” a bar that shows `42` above it, a line point that shows its y, a pie
slice that shows its share. Today no chart can (`grep -ri 'labelList\|valueLabel\|dataLabel\|showValue'
packages/charts/src` â†’ zero hits). This adds an opt-in `labels` prop on Bar/Line/Area/Pie, the equivalent of Recharts'
`LabelList`/`Label`. Lens: **feature parity** (C-F2).

**Architecture:** every chart already computes a pixel position per datum â€” bars compute `barStart`/width/height,
lines map points to `(x, y)`, pie computes slice centroids. A shared `chrome/data-label.tsx` helper takes
`{ x, y, value, position }` and renders a clamped, formatted `<text>` so each chart only supplies positions. Labels
are `aria-hidden` â€” the value is **already** in the a11y tree via the fallback `<table>` and the `aria-live` tooltip
(see `chart-frame.tsx`), so re-announcing would double up. Placement is collision-aware: bars label above (or inside,
flipping to inside when the bar is too short to clear the top); line/area label above the point; pie labels at the
centroid (outside for thin slices). Formatting reuses each chart's existing tooltip formatter when present, else
`toLocaleString`.

**Tech Stack:** SVG `<text>` + the charts' existing per-datum geometry; `@cascivo/i18n` formatting defaults; tokens
`--cascivo-color-foreground` / `--cascivo-color-foreground-muted` for label fill. No engine change.
