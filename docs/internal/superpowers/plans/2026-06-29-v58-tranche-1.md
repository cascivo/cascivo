# v58 — Tranche 1: Semantic Typography Tokens + Variant-Matrix Family (post 1's "ui.default")

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Give typography the same "exactly one correct token" property that color already has. Today an agent (or
engineer) sizing a button label chooses from **eight** raw scale tokens (`--cascivo-text-xs..4xl`) with no canonical
answer; the color axis, by contrast, has a deterministic role→token *variant matrix*. Add a **semantic typography
layer** — purpose-named tokens that each resolve to exactly one primitive — and surface it as a `typography` family
in the variant matrix, so the answer to "what is a button label's type token?" is exactly one: `--cascivo-text-ui`.
(M-1, Master Decision 1.)

**Architecture:** a thin semantic layer over the existing primitive type scale, mirroring how the semantic *color*
layer sits over the primitive color scale — plus a one-family extension to the variant-matrix generator.

- **Semantic type tokens** in `@layer cascivo.tokens` (or the semantic block), each `var()`-ing exactly one
  primitive: `--cascivo-text-ui: var(--cascivo-text-sm)`, `--cascivo-text-body: var(--cascivo-text-base)`,
  `--cascivo-text-heading-lg: var(--cascivo-text-3xl)`, … No new raw sizes — these are *intent aliases* onto the
  existing scale, so the design stays on the canonical scale and agents get one name per purpose.
- **Variant matrix**: `scripts/variants/generate.ts` already maps color `role → slot → token`; add a `typography`
  entry to `families` mapping each role (`ui`, `body`, `heading-lg`, …) to its token name. Type has no per-theme
  variation, so `byTheme` for these resolves to the same primitive everywhere — that is fine and expected.
- **Exemplar migration**: Button's label moves from a raw size to `--cascivo-text-ui`, documented as the canonical
  example. (Wholesale migration of every component is **out of scope** for T1 — introduce the layer, prove it on the
  exemplar, and let later work adopt it; this keeps the tranche surgical.)

**Tech Stack:** `packages/tokens/src/index.css`, `scripts/variants/generate.ts` (+ test), the token manifest
generator (`scripts/tokens/generate-manifest.mjs`, regen only), `apps/site/src/pages/TokensPage.tsx`, and Button
(`packages/components/src/button/`). Pure CSS + TypeScript. No new dependency.

### Task 1: Define the semantic typography layer

**Files:** `packages/tokens/src/index.css`

- [ ] **Step 1:** Decide the role set and the single primitive each maps to. Proposed (each role → exactly one
      existing primitive size; no new sizes):

      | Role token                   | → primitive            | Purpose                                   |
      | ---------------------------- | ---------------------- | ----------------------------------------- |
      | `--cascivo-text-display`     | `--cascivo-text-4xl`   | Hero / marketing display                  |
      | `--cascivo-text-heading-lg`  | `--cascivo-text-3xl`   | Page title (h1)                           |
      | `--cascivo-text-heading-md`  | `--cascivo-text-2xl`   | Section heading (h2)                      |
      | `--cascivo-text-heading-sm`  | `--cascivo-text-xl`    | Subsection heading (h3)                   |
      | `--cascivo-text-body`        | `--cascivo-text-base`  | Default body copy                         |
      | `--cascivo-text-body-sm`     | `--cascivo-text-sm`    | Secondary / dense body copy               |
      | `--cascivo-text-ui`          | `--cascivo-text-sm`    | **Control & button labels (the canonical "ui.default")** |
      | `--cascivo-text-label`       | `--cascivo-text-sm`    | Form field labels                         |
      | `--cascivo-text-caption`     | `--cascivo-text-xs`    | Captions, helper text, metadata           |
      | `--cascivo-text-code`        | `--cascivo-text-sm`    | Inline code / mono (pair with `--cascivo-font-mono`) |

      Note where several roles intentionally share a primitive (`ui`/`label`/`body-sm`/`code` → `sm`): that is
      correct — different *purposes* can resolve to the same size; the value is that each purpose has **one** name.

- [ ] **Step 2:** Add the tokens in a clearly-commented `/* ── Typography roles (semantic) ─────── */` block, each as
      `var()` onto its primitive (so the canonical scale stays the single source of size values). Where a role
      implies a companion (e.g. headings pair with `--cascivo-font-semibold` + `--cascivo-leading-tight`), **do not**
      bundle those into the size token — keep size tokens scalar; document the recommended pairing in the cookbook
      (T6) and the `TokensPage`. (Bundling weight/leading into a size token would make it unusable as a `font-size`
      value; keep one token = one property.)

- [ ] **Step 3:** Confirm the new names pass `pnpm breakpoint:check` (they are not width literals) and do not collide
      with any existing `--cascivo-text-*` primitive (the roles are `-ui`/`-body`/`-heading-*`/`-display`/`-label`/
      `-caption`/`-code`; the primitives are `-xs..-4xl` — no overlap).
