# v44 — Tranche 3: Editor Tokens + Theming (light / dark / warm)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make `@cascivo/editor` theme exactly like every other cascivo surface — via the token system, with no
editor-specific theme API. Add `--cascivo-editor-*` chrome tokens and a semantic `--cascivo-editor-syntax-*`
palette to `@cascivo/tokens`, map them in all three first-party themes (`light.css`, `dark.css`, `warm.css`) with
WCAG-AA contrast, and switch the T2 editor CSS to consume those tokens (it referenced the names already). Applied
via `data-theme`, scoped to any container — exactly the rest of the system.

**Architecture:** Follow the three-level token flow (primitive → semantic → component). The **chrome** tokens
(`bg`, `fg`, `gutter-bg`, `gutter-fg`, `current-line`, `selection`, `border`) map from existing semantic surface/
text tokens. The **syntax palette** (`keyword`, `string`, `number`, `comment`, `function`, `type`, `operator`,
`punctuation`, `variable`, `tag`, `attr`, `regexp`, `boolean`, `property`) maps from primitive color ramps per
theme, chosen for legibility + AA contrast against `--cascivo-editor-bg` in each theme. The editor's token
`<span>` classes (defined in T2) read `var(--cascivo-editor-syntax-<kind>)`. Per CLAUDE.md: tokens only (no
literals), logical properties, static fallbacks before any progressive declaration (`fallback:check`),
reduced-motion-safe, `breakpoint:check` clean.

**Tech Stack:** CSS custom properties (`@cascivo/tokens`, `@cascivo/themes`); `catalog:generate` for the token
catalog. No JS.

### Task 2: Add the editor chrome + syntax tokens

**Files:** `packages/tokens/src/*`

- [ ] **Step 1:** Add the **chrome** tokens with sensible default (theme-agnostic / light-ish) values referencing
      existing semantic tokens: `--cascivo-editor-bg`, `--cascivo-editor-fg`, `--cascivo-editor-gutter-bg`,
      `--cascivo-editor-gutter-fg`, `--cascivo-editor-current-line`, `--cascivo-editor-selection`,
      `--cascivo-editor-border`.

      `--cascivo-editor-syntax-{keyword,string,number,comment,function,type,operator,punctuation,variable,tag,attr,regexp,boolean,property}`,

- [ ] **Step 3:** Keep the defaults in the base token file so a consumer who imports only `@cascivo/editor`
      (without a theme) still gets readable colors; themes override below.

### Task 3: Map the palette in the three themes

**Files:** `packages/themes/src/light.css`, `dark.css`, `warm.css`

- [ ] **Step 1:** In each theme, set the chrome tokens from that theme's surface/text scale
      (`--cascivo-editor-bg` ≈ a subtle raised surface, `--cascivo-editor-fg` = body text, gutter muted,

- [ ] **Step 2:** In each theme, set the syntax palette from that theme's ramps so every kind has **AA contrast**

- [ ] **Step 3:** Verify legibility by eye in all three themes (a quick story/scratch render) and confirm no
