# v51 — Tranche 6: Complete, Generated Docs Coverage

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close the documentation coverage gap. Today only **8 of 17** charts have a Storybook story and **7 of 17**
have an LLM doc (`apps/site/public/llms/chart/` = area, bar, bubble, combo, line, pie, scatter — boxplot, bullet,
heatmap, histogram, kpi, meter, radar, sparkline, treemap are absent). There is **no per-chart documentation page**,
and each `*.meta.ts` carries a rich `intent` block (`whenToUse`/`whenNotToUse`/`related`) that nothing renders. This
tranche makes coverage **17/17 (+ the two new T5 types)** and surfaces the decision-support data that already exists.
Lens: **docs depth** (C-D1/D2/D3/D4). Runs after the features land so the docs describe the finished surface.

**Architecture:** the manifests are the single source of truth. `*.meta.ts` already holds `name`, `description`,
`props`, `tokens`, `accessibility`, `examples`, and `intent`. The LLM docs in `apps/site/public/llms/chart/*.md`
mirror that structure (see `line-chart.md` — title, install, category, props table, examples, tokens). So per-chart
docs should be **generated from the metas**, not hand-written, and the drift check (`pnpm regen` → `git diff
--exit-code`) guarantees they stay in sync. The per-chart page is a docs-app route that reads a chart's meta and
renders: a live demo (the chart with sensible sample data), the props table (`meta.props`), the when-to-use panel
(`meta.intent`), the tokens, and the a11y notes.

**Tech Stack:** `apps/site` (Preact) docs routes/components; `apps/storybook`; the charts' `*.meta.ts`; the existing
docs/LLM generator (whatever produces `public/llms/chart/*.md` + `public/r/chart-*.json` today). No new framework.
