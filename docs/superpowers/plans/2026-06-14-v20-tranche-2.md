# v20 Tranche 2 — Enforcement & authoring DNA

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development or
> superpowers:executing-plans. Steps use checkbox (`- [ ]`) syntax.
>
> This tranche turns "responsive" from a guideline into a gate. It adds a `breakpoint:check` lint (no
> off-scale width literals), a reusable mobile-overflow + touch-target Playwright harness (generalized
> from the landing app's), a small first enrolled set, and the `CLAUDE.md` authoring rule. Later tranches
> enroll the surfaces they touch. The gate must stay green from day one — hence the allowlist.
>
> Current-state assumptions — RE-VERIFY before starting:
>
> - T1 shipped: `@cascivo/tokens` exports `SCREEN` + `minWidth/maxWidth`; `--cascivo-screen-*` documented.
> - `scripts/checks/` holds `target-size.test.ts`, `media-features.test.ts`, `css-fallback.ts`, wired into
>   the pre-commit/CI check set (re-read how they're registered — likely a `vp` task or a test glob).
> - The only mobile e2e today is `apps/landing/test/mobile.spec.ts` (320/375/390/414 overflow + drawer);
>   it is a good template for the generalized helper.
> - Off-scale width literals exist in the repo today (`30rem`, `28rem`, `480px` in docs; `40rem`, `64rem`
>   are on-scale). The lint must allowlist existing literals so it doesn't flood-fail.

**Goal:** A `breakpoint:check` test flags any `@media`/`@container` width literal not on the canonical
scale (with an allowlist). A reusable harness asserts zero horizontal overflow + minimum tap target at
320/360/390/414 for an enrolled set of surfaces (seed: Button, Input, Modal). `CLAUDE.md` gains a
"Responsive by default" authoring rule. All checks pass.

**Architecture:** new `scripts/checks/breakpoint.test.ts` (+ an allowlist file), new
`scripts/checks/responsive-sweep.ts` (or under the appropriate test app) helper + a first spec,
`CLAUDE.md`. Re-use `SCREEN` from `@cascivo/tokens`.

**Tech stack:** unchanged. The lint is a Node/Vitest scan over CSS files (mirror how `css-fallback.ts`
walks files). The sweep is Playwright (mirror `apps/landing/test/mobile.spec.ts`).

### Task 1: `breakpoint:check` lint

**Files:** Create `scripts/checks/breakpoint.test.ts` and `scripts/checks/breakpoint-allowlist.ts`

- [ ] **Step 1:** Walk `packages/**/src/**/*.css` (mirror `css-fallback.ts`'s file walk). For each
      `@media` and `@container` rule, extract every length used in a width/inline-size condition
      (`min-width`, `max-width`, `width`, `inline-size`, `min-inline-size`, `max-inline-size`).
- [ ] **Step 2:** Assert each extracted width is one of the canonical `SCREEN` values (compare in a
      normalized unit; `64rem`/`1024px` both map to `lg`) OR is listed in `breakpoint-allowlist.ts`. Fail
      with an actionable message naming the file, line, value, and the nearest canonical value.
- [ ] **Step 3:** Seed `breakpoint-allowlist.ts` with the off-scale literals that exist today (re-scan to
      get the exact list — e.g. the docs app's `30rem`/`28rem`/`480px` if those files are in scope, plus
      any component using a non-scale width). Add a comment: "Entries are debt; later tranches migrate
      them onto the scale and delete the entry." Container-query *content* breakpoints that are
      intentionally component-intrinsic (not viewport) may be allowlisted with a reason.
- [ ] **Step 4:** Register `breakpoint.test.ts` in the same check set as `target-size`/`media-features`
      (re-read how those run — add to the glob/task so `vp test` and the pre-commit check pick it up).

### Task 2: Reusable mobile-overflow + touch-target harness

**Files:** Create the shared helper + a first spec (place alongside the existing e2e infra — re-verify
whether components/layouts have a Playwright project or whether one must be added; if adding is heavy,
host the sweep in an existing app's test dir and import stories/components)

- [ ] **Step 1:** Write a helper `assertNoOverflow(page, widths = [320,360,390,414])` that, for each
      width, sets the viewport, waits for layout, and asserts
      `document.documentElement.scrollWidth <= innerWidth` (mirror `apps/landing/test/mobile.spec.ts`).
- [ ] **Step 2:** Write a helper `assertTapTargets(page, selector)` that checks every matched interactive
      element's rendered box is ≥ the coarse target (44px) — emulate a coarse pointer
      (`hasTouch`/`isMobile` Playwright context) so `@media (pointer: coarse)` rules apply.
- [ ] **Step 3:** Add the first enrolled spec covering the **seed set** — Button, Input, Modal (smoke).
      Render each (via Storybook URL or a tiny harness page — re-verify what's available) and run both
      assertions. Use `SCREEN.*.px` for any width math, never literals.
- [ ] **Step 4:** Confirm the seed spec passes against the **current** Button/Input/Modal. If Button's
      32px `sm` fails the 44px coarse-target assertion, scope that assertion to surfaces enrolled in T3+
      (or mark the seed Button case as overflow-only) so T2 stays green; T3 turns the target assertion on
      for primitives. Document this in the spec.

### Task 3: `CLAUDE.md` "Responsive by default" authoring rule

**Files:** Modify `CLAUDE.md`

- [ ] **Step 1:** Under "Component Authoring Rules", add a "Responsive by default (mobile-first)" section:
      base styles target the smallest screen; enhancements layer via `min-width`/min container queries;
      prefer `@container` over `@media`; use **only** canonical `SCREEN` values in width conditions
      (`breakpoint:check` enforces this); interactive controls reach ≥44px under `@media (pointer: coarse)`;
      never hide content with `display:none` to "fit" — relocate it; verify with the mobile sweep.
- [ ] **Step 2:** Add a one-line entry to the "Checklist before committing a component": "Passes the
      mobile-overflow + touch-target sweep at 320/360/390/414; no off-scale breakpoint literals."
- [ ] **Step 3:** Reference the new scripts (`breakpoint:check`, the sweep) from the "Gate Before
      Committing" list so the local-CI-reproduction commands stay complete.

### Task 4: Commit

- [ ] **Step 1:** Run the full gate (`vp check` → build → `vp run -r check` → test → `pnpm regen` +
      `git diff --exit-code`). All green.
- [ ] **Step 2: Commit** — `feat(checks): breakpoint lint + mobile-overflow/touch-target sweep + authoring rule (v20-t2)`

## Tranche exit criteria

1. `breakpoint:check` fails on a deliberately-introduced off-scale literal (verify by temporarily adding
   one) and passes on the current repo via the allowlist; it runs in the standard check set.
2. A reusable `assertNoOverflow` + `assertTapTargets` harness exists and a seed spec (Button/Input/Modal)
   passes at 320/360/390/414 using `SCREEN` constants.
3. `CLAUDE.md` documents the responsive-by-default rule, the commit checklist line, and the new gate
   commands.
4. The growing-gate model is in place: later tranches add surfaces to the sweep and retire allowlist
   entries.
5. Full CLAUDE.md gate exits 0. No `useEffect`/`useState` introduced.
