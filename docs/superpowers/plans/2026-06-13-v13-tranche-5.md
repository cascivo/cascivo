**Goal:** Prove cascade can move style _logic_ — token math and variant→token selection —
into the stylesheet using CSS `@function` and `if(style())`, on a small pilot (Button,
Badge, Alert), **without any regression**: every enhanced declaration is guarded so a
non-supporting browser renders the existing static value, and a gate enforces that fallback.

**Architecture:** `@cascade-ui/tokens` gains `functions.css` defining trivial,
single-purpose `@function` helpers. Pilot component CSS adopts one `@function` use and/or one
`if(style(--variant: …))` use, each preceded by or wrapped in a static fallback. A
`scripts/checks` script parses pilot CSS and fails if any `@function`/`if()` declaration
lacks a guarded static fallback. The v9 parity test must stay green; cross-browser proof is a
screenshot pair recorded in the PR (jsdom can't evaluate these features).

**Tech stack:** unchanged + emerging CSS (`@function`, `if()`) as **progressive enhancement,
accepted-risk** per CLAUDE.md's alpha stance. No build-time polyfill — native or fallback only.

- Modify: `packages/tokens/src/index.css` (import it last, inside the appropriate `@layer`)

- [ ] **Step 1:** Define one or two single-purpose helpers, using defaults on optional args
      and `calc()` to defer numeric validation (master-plan risk #9 — arg-count and
      integer-strictness gotchas). Example density/step helper:

```css
/* progressive enhancement: only consumed where a guarded fallback exists */
@function --cascivo-step(--n <number>, --base: 0.25rem) {
  result: calc(var(--base) * var(--n));
}
```

      Keep functions trivial: no recursion (banned), single `result`, no reliance on variable
      spreading (not yet supported per the source article).

- [ ] **Step 2:** Document at the top of the file: these are progressive enhancement, every
      consumer MUST provide a static fallback, and the supported-browser set (Chrome-leading,

### Task 2: Fallback-audit script

**Files:**

- Create: `scripts/checks/css-fallback.ts` (+ test)
- Modify: the checks runner / `package.json` to include it in `vp check` or the test gate

- [ ] **Step 1:** `auditFallbacks(cssFiles)` — parse each file; for every declaration whose
      value contains a custom `--cascivo-*(` function call or `if(`:
  - require that the **same property** has a static fallback that a non-supporting browser
    will use: either a preceding plain declaration of that property in the same rule (the
    browser keeps the last value it understands and ignores the unknown-function one), OR the
    enhanced declaration sits inside an `@supports` block whose negation/base provides the
    static value.
  - if neither pattern is present → fail with `file:line` and the offending property.
- [ ] **Step 2: Test:** a rule with `padding: 0.5rem; padding: --cascivo-step(2);` → passes;
      a bare `padding: --cascivo-step(2);` with no preceding static value → fails; an
      `@supports`-guarded pair → passes.
- [ ] **Step 3: Commit** — `feat(checks): CSS @function/if() fallback audit`

### Task 4: Pilot — Badge + Alert (`if(style())` variant selection)

**Files:**

- Modify: `packages/components/src/badge/badge.module.css`,
  `packages/components/src/alert/alert.module.css`

- [ ] **Step 1:** Where variant→token mapping currently relies on separate
      `.badge[data-variant="x"]` rules or class switches, pilot a single `if(style())`
      selection on **one** property, keeping the existing rules as the static fallback. Sketch:

```css
.badge {
  /* static fallback: existing per-variant rules remain below and still apply */
  background: var(--cascivo-badge-bg);
  /* progressive: choose the token by variant signal where supported */
  background: if(
    style(--variant: success): var(--cascivo-color-success) ;
      style(--variant: warning): var(--cascivo-color-warning) ; else: var(--cascivo-badge-bg)
  );
}
/* existing per-variant rules stay — fallback for non-supporting browsers */
.badge[data-variant='success'] {
  background: var(--cascivo-color-success);
}
.badge[data-variant='warning'] {
  background: var(--cascivo-color-warning);
}
```

      Note: this requires the component to expose `--variant` as a custom property
      (`style={{ '--variant': variant }}` / a CSS var set from `data-variant`) — wire it
      minimally; if exposing the signal is more than a one-line change, keep the pilot to
      Button's `@function` only and record Alert/Badge as "deferred within v13" honestly.

- [ ] **Step 2:** Confirm the `if()` branches resolve to the **same tokens** the static rules
      use — no value drift. Fallback audit passes (static rules present). Component tests pass.
- [ ] **Step 3: Commit** — `feat(badge,alert): pilot if(style()) variant selection (fallback-guarded)`

## Tranche exit criteria

1. `functions.css` defines trivial, fallback-required `@function` helpers, documented as
   progressive enhancement.
2. The pilot (Button `@function`; Badge/Alert `if(style())` — or a recorded honest deferral
   of the `if()` part if signal exposure proved non-trivial) uses CSS-native logic, each
   enhanced declaration guarded by a static fallback for the same property.
3. The `css-fallback` audit fails on a bare `@function`/`if()` declaration and passes on the
   pilot; it runs in the gate.
4. The v9 theme-parity test stays green; a screenshot pair proves the supporting and fallback
   renders are visually identical (no regression).
5. Root gate + drift check exit 0.
