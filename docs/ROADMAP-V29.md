# cascivo — Roadmap v29: Blocks (`/blocks`)

**Last updated:** 2026-06-16
**Status:** 🚧 Planning
**Plan documents:** `docs/superpowers/plans/2026-06-16-v29-master-plan.md` + tranches 1–6

---

## Vision

v28 gave developers a theme in 30 seconds. v29 gives them a layout in 30 seconds.

The cascivo blocks page lives at `/blocks` on the landing. Eight production-ready UI sections —
two auth flows, two dashboard panels, two marketing sections, an app shell, and a settings form —
are built entirely from cascivo components and available to browse, preview, and copy. Each block
ships with its TSX and CSS source. `npx cascivo add block/auth-login` drops it directly into a
project.

The closest public reference is [ui.shadcn.com/blocks](https://ui.shadcn.com/blocks). Shadcn's
blocks page showcases composed UI sections (dashboards, login forms, sidebars) that users copy
into their projects. Our version adapts that concept for cascivo's model:

- Blocks render **live in-page** (no iframes) under a scoped `data-theme` container so the real
  components are always visible, not screenshots or stubs.
- The gallery uses **static screenshots** for the card thumbnails so the initial page load is fast.
- The detail page adds a **three-stop viewport selector** (320 / 768 / 1280 px) so developers
  verify mobile behaviour before copying.
- The code panel shows **two files** (TSX and CSS) in separate tabs — the exact output of
  `npx cascivo add block/<name>`.
- The install command uses the **same CLI** as components: `npx cascivo add block/auth-login`.

No new `@cascivo/*` npm packages. Blocks are implemented in `packages/components/src/blocks/` and
follow the same copy-paste model as components — users own what they add.

## What changes

### Feature 1 — `/blocks` gallery route

A new top-level route `/blocks` is added to the landing SPA. It is lazy-loaded like every other
non-home page. A "Blocks" link is added to the primary nav (between "Create" and the search icon).
The route is prerendered (head-only) for SEO.

The gallery layout is a CSS Grid of block cards. Above the grid, a category tab bar filters the
view: **All / Auth / Dashboard / Marketing / Shell**. A search `Input` below the tabs filters by
block name and description. Filtering and search are both driven by Preact Signals — no URL state
needed, filters reset on navigation away.

Each block card contains:
- A static screenshot of the block in the current site theme (light or dark), generated by the
  screenshot pipeline extended in T4.
- The block `displayName` as a heading.
- A category `Badge`.
- A "View block →" link to `/blocks/:name`.

Cards use `@container` (not `@media`) so the grid reflows cleanly at any container width.

### Feature 2 — `/blocks/:name` detail route

Clicking a block card navigates to `/blocks/<name>`. The detail page has two main regions:

**Preview region** — a full-width panel showing the block rendered live inside a
`[data-theme="<current>"]` container. Three controls sit above the preview:
- **Light / Dark** toggle (two-button group) — switches the preview theme without affecting the
  rest of the page.
- **Viewport selector** — three buttons (320 / 768 / 1280). Applies a `max-width` to the preview
  container so the block is seen at mobile, tablet, and desktop widths.
- **Open full tab** link — opens `/blocks/preview/<name>` in a new browser tab (a bare page with
  only the block and theme CSS, no landing chrome).

**Code region** — below the preview, a tab bar with two tabs:

- **TSX** — the block component source, syntax-highlighted (same CSS-only technique from v28
  CodePanel). Copy button top-right.
- **CSS** — the block module CSS source. Copy button top-right.

Below the code tabs, an install command row:

```
npx cascivo add block/auth-login
```

A copy-to-clipboard button beside the command. The command is static text — no server call.

### Feature 3 — 8 blocks in `packages/components/src/blocks/`

Each block is a directory with four files:

```
packages/components/src/blocks/<name>/
  <name>.tsx           # exported block component
  <name>.module.css    # block-specific styles (uses cascivo tokens only)
  <name>.meta.ts       # BlockMeta — displayName, category, description, tags, screenshot
  <name>.test.tsx      # renders without throwing, basic a11y assertions
```

Blocks import only from `@cascivo/react` (the prebuilt npm distribution). They do not import from
`@cascivo/components` (the registry source). This ensures the copied file works in any project that
has `@cascivo/react` installed.

**Auth blocks:**
- `auth-login` — email + password inputs, "Remember me" checkbox, "Forgot password?" link (plain
  anchor), "Sign in" primary button, "Create an account" ghost link. Centred Card layout.
- `auth-signup` — full name + email + password + confirm-password inputs, terms-of-service
  Checkbox with inline link, "Create account" primary button, "Sign in instead" ghost link.
  Centred Card layout, same width as login.

**Dashboard blocks:**
- `dashboard-overview` — welcome heading ("Good morning, {name}") with today's date, four KPI
  stat Cards (Revenue, Active Users, Orders, Conversion Rate) in a 2 × 2 grid on `md+` / 1-col
  on mobile. Each card: icon slot (SVG placeholder), metric value, percentage delta Badge
  (green/red), trend label.
- `dashboard-table` — section header with title + search Input (inline filter, signal-driven) +
  "Export" ghost Button. Sortable HTML table (click column heading → sort signal) with pagination
  controls (Previous / Next buttons, "Page N of M" label). Seven rows of placeholder data,
  columns: Name, Status (Badge), Email, Date, Amount.

**Marketing blocks:**
- `marketing-hero` — large headline, subtitle paragraph, two CTA Buttons (primary + ghost), a row
  of three "trust badges" (small Badges: "MIT licence" / "Zero dependencies" / "WCAG 2.1 AA").
  Full-width centred layout with generous vertical padding.
- `marketing-features` — section heading + subtitle, three-column feature grid (icon + title +
  description per cell, `@container`-driven). Six feature cells total (two rows of three on
  `lg+`).

**Shell blocks:**
- `app-shell` — two-column layout: a fixed-width left sidebar (logo slot, vertical nav list with
  five mock links, active-state styling via `[aria-current="page"]`), and a scrollable right
  content area (breadcrumb, page title, content placeholder). Sidebar collapses to a slide-over
  overlay on mobile via a `useSignal(false)` toggle.
- `settings-profile` — two-section form. Section 1 "Personal info": name, email, bio Textarea.
  Section 2 "Preferences": three Toggle rows (Email notifications, Push notifications, Weekly
  digest), a Select for language, a Select for timezone. Save Changes primary Button, Cancel ghost
  Button in a sticky footer bar.

All eight blocks pass `pnpm exec vp check` and `vp run @cascivo/components#test` before T4.

### Feature 4 — Block registry + CLI

`registry.json` gains a top-level `blocks` array alongside `components`. Each entry mirrors the
component shape but uses `"type": "block"`:

```json
{
  "name": "auth-login",
  "type": "block",
  "displayName": "Login Form",
  "description": "Email + password login with remember-me and forgot-password.",
  "category": "auth",
  "version": "0.0.0",
  "files": [
    "https://raw.githubusercontent.com/urbanisierung/cascivo/main/packages/components/src/blocks/auth-login/auth-login.tsx",
    "https://raw.githubusercontent.com/urbanisierung/cascivo/main/packages/components/src/blocks/auth-login/auth-login.module.css"
  ],
  "dependencies": ["@cascivo/react"],
  "tags": ["auth", "login", "form"],
  "screenshot": {
    "light": "/blocks/screenshots/auth-login-light.png",
    "dark": "/blocks/screenshots/auth-login-dark.png"
  }
}
```

The `scripts/registry/gen-registry.mjs` script is extended to scan
`packages/components/src/blocks/*/` and append entries to the `blocks` array.

The CLI `add` command already handles `type` correctly (it fetches `files` and writes them to the
output path). The user-facing syntax gains a `block/` prefix convention:

```sh
npx cascivo add block/auth-login   # installs auth-login.tsx + auth-login.module.css
npx cascivo add auth-login          # unchanged — still resolves to the component registry
```

The `block/` prefix is resolved in `packages/cli/src/utils/registry.ts`: if the name starts with
`block/`, the CLI queries the `blocks` array in `registry.json` instead of `components`.

The screenshot pipeline (`scripts/gen-demo-screenshots.mjs`) is extended with a
`captureBlocks()` pass that renders each block at 1280 × 900 (light) and 1280 × 900 (dark) and
writes PNGs to `apps/landing/public/blocks/screenshots/`.

### Feature 5 — Blocks data layer

`apps/landing/src/pages/blocks/blocks-data.ts` is the single import point for the landing. It:

- Imports all eight `*.meta.ts` files and re-exports a typed `BLOCKS` array.
- Provides lazy component loaders: `lazy(() => import('@cascivo/blocks/auth-login'))` (resolved
  via a source alias to the block TSX files in the landing vite config).
- Exports helpers: `findBlock(name: string)`, `blocksByCategory(cat: string)`.

The landing vite config gains a wildcard alias `@cascivo/blocks/*` →
`packages/components/src/blocks/*/<name>.tsx` resolved per-file at bundle time (via a custom
`resolve.alias` array of the eight known names).

## Why these changes and not others

**Blocks are the highest-value landing addition after the theme configurator.** The `/create` page
shows _how_ to customise cascivo; `/blocks` shows _what to build with it_. A developer who can
copy a production-ready dashboard panel in five seconds will reach for cascivo before trying
another library.

**Live in-page rendering beats screenshots for the detail view.** On the gallery thumbnails,
static screenshots are correct — fast load, no jank, no React hydration. On the detail page,
static screenshots would undermine the core cascivo claim: CSS-native, signal-driven, instantly
themeable. The developer switches light → dark and sees the real components update immediately.
That is the proof point.

**Viewport selector over responsive iframe.** Iframes introduce same-origin complexity, theme
inheritance issues, and scrollbar artefacts. A max-width constraint on the preview container is
simpler, more reliable, and exactly what the developer's CSS will do in production.

**Block/ prefix in the CLI, not a separate command.** `cascivo add block/auth-login` feels natural
alongside `cascivo add button`. It uses the same lock file, the same output-path resolution, and
the same version tracking. No new surface.

**Eight blocks at launch.** Three categories means breadth; two blocks per major category means
variety without overwhelming. The factory pipeline can add more blocks over time without changes
to the landing architecture.

## Decisions baked in

1. **Static screenshots on gallery cards, live render on detail page.** Gallery load performance
   wins over live render fidelity on the index. The detail page is where fidelity matters and the
   page loads only one block component.
2. **`[data-theme]` scoping on the detail preview.** The block preview sits inside a
   `<div data-theme="light">` or `<div data-theme="dark">` controlled by the toggle signal. The
   rest of the landing remains in the site theme. Same scoping model as the v28 configurator.
3. **Blocks reference `@cascivo/react`, not `@cascivo/components`.** Users who copy blocks must
   have `@cascivo/react` installed (the npm package). They do not get `packages/components/`
   source. This keeps the dependency model simple and consistent.
4. **`block/` prefix in the CLI, no new subcommand.** `add` already handles `RegistryItem`
   generically. The prefix routes the lookup to the `blocks` array in `registry.json`. No new
   command surface.
5. **Viewport selector applies `max-width`, not an iframe.** The preview container is a plain div
   with a CSS transition on `max-width`. Three widths (320 / 768 / 1280) match the canonical
   cascivo breakpoints (sm / md / xl loosely). The block itself is `width: 100%` so it always
   fills the container.
6. **No live code editor in the detail view.** The goal is "browse → copy → paste", not a
   playground. A code editor adds Prism/Monaco dependency, bundle weight, and complexity. Users
   who want to experiment have `/create`.
7. **Screenshot pipeline is a dev-time script, not a CI artefact.** Screenshots are committed to
   `apps/landing/public/blocks/screenshots/` and updated when blocks change. CI checks for drift
   via `git diff --exit-code` after running `pnpm regen`.
8. **`blocks-data.ts` is the single import seam.** The landing never imports block files
   directly. This keeps vite aliasing to one file and makes adding new blocks a one-file change.
