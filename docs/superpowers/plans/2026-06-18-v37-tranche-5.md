# v37 — Tranche 5: AppShell Layout + SideNav Polish

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship the missing "app shell" glue so consumers stop hand-rolling the `ShellHeader` + `SideNav`
scaffolding (#11) and get an **animated** show/hide with proper focus/`inert` handling (#12) — and fix the
`SideNav` `min-block-size:100%` footgun so its own internal scroll actually fires.

**Architecture:** `ShellHeader` and `SideNav` already exist as a matched pair but no component wires them
into the correct single-scroll-container layout. `ShellHeader` exposes `onMenuClick`/`menuExpanded`
(`shell-header.tsx:53-54`) but nothing binds it to the nav. `SideNav` sets `min-block-size:100%`
(`side-nav.module.css:7`) — a *min*, which lets a long list overflow its parent instead of scrolling — yet
also ships `overflow:hidden auto` and `transition: inline-size var(--cascivo-motion-emphasis)` for rail
collapse (line 14). We ship a new `AppShell` (Decision 4) that owns the scroll container, the burger↔nav
binding, the animated open/close (reusing `--cascivo-motion-*`, `prefers-reduced-motion`-aware), and
`inert`/focus management, and reconciles full-hide vs rail collapse. `SideNav`'s height constraint is
fixed in place. **No `SideNav`/`ShellHeader` API breaks.**

Authoring rules (CLAUDE.md) bind this component: no `useState`/`useEffect`/`useContext`/`useReducer`;
`open` is a controlled prop synced into a `useSignal` during render; DOM side effects (`inert`, focus) use
`useSignalEffect`; CSS `@function`/`if()` only as progressive enhancement with a static fallback. Avoid the
`grid 1fr→0fr` collapse trap (fails on an auto-width flex item, per the report) — animate explicit
`inline-size`/`transform`.

**Tech Stack:** React 18 component in `@cascivo/components`, `@cascivo/core` signals, vite+.
