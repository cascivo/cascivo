# v40 — Tranche 2 (core): Deep Conformance Validation

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Enforce what T1 asks the model to do. Today `validateView` checks that a node's `component` exists
and that `bind`/`events` use the right prefixes — but it does **not** check prop names, types, or enum
values, so an LLM can emit a real component with invented or wrong props and it renders silently. Extend the
validator to check **props against the manifests** so hallucinated props are caught with actionable errors.
This is the enforcement backstop for the bound vocabulary (T1) and hardens the dogfooded `CascadeView` demo.

**Architecture:** `packages/render/src/validate.ts` — `validateView(config)` recurses nodes, checks
`component in componentMap` (Levenshtein "did you mean"), validates `$data.`/`$actions.` prefixes, and
recurses `children`. The manifests carry per-prop `type`/`enum`/`default` (the same source T1 uses). Wire a
meta-lookup (component name → prop schema) so the validator can flag unknown props, type mismatches, and
out-of-enum values, reusing the existing Levenshtein helper for prop-name suggestions. `CascadeView`'s
`onInvalid` contract is preserved (validation returns structured errors, never throws).

**Tech Stack:** Hand-written TypeScript checks (no Zod — dependency policy); Vitest via vite+ (`vp`).
Independent of T1 and T3 (can run in parallel with T1).

### Task 2: Extend `validateView` with prop conformance

**Files:** `packages/render/src/validate.ts`

- [ ] **Step 1:** In `validateNode`, after the component-exists check, validate each entry of `props` against
      the component's prop schema: **unknown prop** (with a Levenshtein "did you mean" over that component's
      prop names), **type mismatch**, and **out-of-enum value**. Emit `ValidationError{path, message}` in the
      existing actionable style, `path` pointing to `…props.<name>`.
- [ ] **Step 2:** Do **not** validate `bind`/`events` *values* against types (they resolve at runtime from the
      host context) — keep the existing prefix checks. Do not weaken any existing check.
- [ ] **Step 3:** Confirm the function still **returns** `{ valid, errors }` and never throws, preserving the
      `CascadeView onInvalid` contract used by the landing/docs demos.
