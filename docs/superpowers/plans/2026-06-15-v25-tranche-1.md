### Task 2: Verify and fix "Count the re-renders" counters

The `SignalsDemo` section shows render counters for a signals form and a `useState` form. Both
use `renders.current++` + `queueMicrotask` to emit to a module-level signal. The counter for the
`useState` form should increment on every keystroke; the signals form counter should stay at 1.

- [ ] **Step 1: Test the demo manually**

  With the dev server running, scroll to the "Count the re-renders" section. Type characters into
  the **signals** form "Name" field. Observe the badge. Type characters into the **useState** form
  "Name" field. Observe the badge.

  Expected behaviour:
  - Signals form badge: stays at `1` regardless of typing (form never re-renders)
  - useState form badge: increments by 1 per keystroke (form re-renders on every state change)

  If the badges stay at 0 or do not increment, proceed to Step 2. If they work correctly, skip
  to Step 5 (just commit a no-op note).

- [ ] **Step 2: Diagnose the tracking gap**

  Open `apps/landing/src/sections/SignalsDemo.tsx`. Locate `StateForm`:

  ```tsx
  function StateForm() {
    const renders = useRef(0)
    renders.current++
    queueMicrotask(() => {
      stateFormRenders.value = renders.current
    })
    // ...
  }
  ```

  The issue with React 18 Strict Mode: components render twice on mount. This means on first
  mount `renders.current` goes to 2 before any user interaction. The badge shows `2` before any
  typing, which is confusing.

  Also confirm: does `stateFormRenders` start at 0? Does `RenderBadge` call `useSignals()`?
  Check that `RenderBadge` is the component that reads `count.value`.

- [ ] **Step 3: Fix initial count and Strict Mode issue**

  Replace the render-tracking in both `SignalForm` and `StateForm` to reset to 0 on mount and
  only count post-mount renders (so Strict Mode's extra render does not inflate the initial count):

  In `apps/landing/src/sections/SignalsDemo.tsx`, update `StateForm` and `SignalForm`:

  ```tsx
  function SignalForm() {
    const mounted = useRef(false)
    const renders = useRef(0)
    if (mounted.current) {
      renders.current++
      queueMicrotask(() => { signalFormRenders.value = renders.current })
    } else {
      mounted.current = true
    }
    // rest unchanged
  }

  function StateForm() {
    const [name, setName] = useState('')
    const [email, setEmail] = useState('')
    const [newsletter, setNewsletter] = useState(false)
    const mounted = useRef(false)
    const renders = useRef(0)
    if (mounted.current) {
      renders.current++
      queueMicrotask(() => { stateFormRenders.value = renders.current })
    } else {
      mounted.current = true
    }
    // rest unchanged
  }
  ```

  This counts 0 on the Strict Mode mount render and 1 on the first real render, then increments
  correctly from there.

- [ ] **Step 4: Verify counters work**

  With the dev server running (reload the page after file save), type into both forms. Confirm:
  - Signals form badge shows `0` (the form itself never re-rendered after initial mount)
  - useState form badge increments by 1 per character typed in any field

  If signals form shows `0` and useState form shows an incrementing count, the demo is correct.

### Task 4: Fix A11y matrix DataTable

The A11yMatrix on `/accessibility` renders a `DataTable` that may be blank. Two possible causes:
`A11Y_ROWS` is empty (no `meta.accessibility` blocks in the registry), or the signal subscription
is broken.

- [ ] **Step 1: Verify A11Y_ROWS is non-empty**

  ```sh
  node -e "
    const reg = JSON.parse(require('fs').readFileSync('registry.json', 'utf8'))
    const rows = reg.components.filter(c => c.meta?.accessibility)
    console.log('rows with accessibility:', rows.length)
    if (rows.length > 0) console.log('sample:', JSON.stringify(rows[0].meta.accessibility))
  "
  ```

  If `rows with accessibility: 0`, the registry entries lack `meta.accessibility`. The table
  will always be empty. Proceed to Step 2 to add an `EmptyState` fallback.

  If rows are non-zero, the bug is in the component — proceed to Step 3.

- [ ] **Step 2: Add EmptyState fallback when A11Y_ROWS is empty**

  In `apps/landing/src/pages/accessibility/A11yMatrix.tsx`, import `EmptyState`:

  ```tsx
  import { EmptyState } from '@cascivo/components/empty-state'
  ```

  Add a guard before the `DataTable`:

  ```tsx
  {rows.value.length === 0 ? (
    <EmptyState
      title="No accessibility data available"
      description="Run pnpm regen to regenerate registry.json with accessibility metadata."
    />
  ) : (
    <DataTable
      columns={COLUMNS}
      rows={rows.value}
      getRowId={(r) => r.name}
      density="compact"
      stickyHeader
      title="Keyboard and ARIA matrix"
      description="Role, WCAG level, and keyboard interaction per registry entry."
    />
  )}
  ```

- [ ] **Step 3: If rows are non-zero — check signal subscription**

  Open `apps/landing/src/pages/accessibility/A11yMatrix.tsx`. Confirm `useSignals()` is the
  first statement inside the `A11yMatrix` function (it already is, per the current file). Then
  confirm `rows.value` is passed to `DataTable` (it is — `rows={rows.value}`). If the table
  still shows nothing with non-zero rows, open DevTools → Console on `/accessibility` and look
  for errors.

- [ ] **Step 4: Verify the page**

  Navigate to `/accessibility` in the dev server. Confirm either:
  - The DataTable renders with rows for each component that has accessibility metadata, OR
  - The EmptyState renders with the message from Step 2

  Either outcome is correct. An empty table with no rows (the blank state the user reported)
