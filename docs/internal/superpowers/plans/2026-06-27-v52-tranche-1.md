# v52 — Tranche 1: Visual Enrichment — Gradients, Patterns & Curves

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Give cascivo charts the two cheapest, most-visible expressiveness wins that nivo and visx both have and
cascivo lacks: **SVG `<defs>` gradient + pattern fills** (N-2) and an **expanded curve-factory set** (N-3). Today fills
are solid `--cascivo-chart-*` tokens and `engine/shape.ts` offers only `'linear' | 'monotone'`. Lens: **nivo + visx**.

**Architecture:** gradients/patterns are SVG `<defs>` referenced by `fill="url(#id)"`. A new `chrome/defs.tsx`
exposes a `<ChartDefs>` that, given the resolved series colors, mints a `linearGradient` (same hue, vertical fade) or
a `pattern` (dots/lines/cross stroked in the series color) per series and returns stable ids — derived entirely from
the existing palette so they stay CVD-safe (Decision 6). Charts opt in with `fill="gradient" | "pattern" | "solid"`
(default `solid` — no visual change unless asked). Curves extend `engine/shape.ts`: the curve union grows and
`linePath`/`areaPath` dispatch to new factories (step/stepBefore/stepAfter, natural cubic, basis, cardinal,
catmull-rom) — same hand-rolled, dependency-free approach as the existing `monotonePath`.

**Tech Stack:** SVG `<defs>` + the existing palette tokens; `engine/shape.ts` math; no new dependency. The `<defs>`
render inside the existing `ChartFrame` `<svg>`.

### Task 1: Curve factories (engine/shape.ts)

**Files:** `packages/charts/src/engine/shape.ts`

- [ ] **Step 1:** Widen the curve type to
      `type Curve = 'linear' | 'monotone' | 'step' | 'stepBefore' | 'stepAfter' | 'natural' | 'basis' | 'cardinal' | 'catmullRom'`
      and export it.
- [ ] **Step 2:** Implement each factory as a `(points) => string` path builder (mirroring `monotonePath`): step
      variants emit `H`/`V` segments; `natural` solves the tridiagonal system for a natural cubic spline; `basis`/
      `cardinal`/`catmullRom` emit cubic `C` segments with the standard control-point formulas. Keep them pure +
      dependency-free.
- [ ] **Step 3:** Route `linePath(points, curve)` and `areaPath(points, baseline, curve)` to the right factory; keep
      `linear`/`monotone` behavior byte-identical (regression safety).
- [ ] **Step 4:** `shape.test.ts`: each curve produces a non-empty, `NaN`-free `d` for a known point set; step starts
      with `M` then alternating `H`/`V`; a single point degrades to a move; closed/loop inputs don't throw.

### Task 2: Gradient + pattern defs registry

**Files:** `packages/charts/src/chrome/defs.tsx`

- [ ] **Step 1:** Export `gradientId(seriesId)` / `patternId(seriesId, kind)` helpers that produce stable, collision-safe
      ids (prefix with a `useId` per chart instance so two charts on a page don't clash).
- [ ] **Step 2:** `ChartDefs({ series, fill, patternKind })` renders a `<defs>` block: for `fill="gradient"`, a
      `linearGradient` per series from the series color at full opacity → low opacity (vertical); for `fill="pattern"`,
      a `<pattern>` per series (dots / lines / cross-hatch via `patternKind`) stroked in the series color over a faint
      tint. Colors come from the resolved `--cascivo-chart-*` (or per-series `color`) — never hard-coded.
- [ ] **Step 3:** Export a `fillFor(seriesId, fill, color)` helper returning `url(#id)` or the solid color, so each
      chart maps a series to its fill in one call.
