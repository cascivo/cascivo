# v20 Tranche 3 — Interactive primitives

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development or
> superpowers:executing-plans. Steps use checkbox (`- [ ]`) syntax.
>
> Make the everyday controls consume the single height source and reach a comfortable tap target on touch
> devices — without changing desktop density. Button is the headline fix: it hardcodes `2/2.5/3rem` and
> ignores the existing `--cascivo-control-height-*` tokens. Then enroll the touched primitives in the
> sweep and turn on the touch-target assertion for them.
>
> Current-state assumptions — RE-VERIFY before starting:
>
> - T1 tokens shipped (`--cascivo-control-height-{sm,md,lg}` = 1.75/2.25/2.75rem;
>   `--cascivo-target-min-coarse` = 2.75rem). T2 sweep + lint shipped.
> - `button.module.css:32–46` hardcodes `height: 2rem / 2.5rem / 3rem` (NOT the control-height tokens) —
>   these differ from the tokens (2rem≠1.75rem etc.), so "consume the token" is a **value change**; decide
>   per Step 1 whether to adopt the token value or re-point the token. Default: keep Button's current
>   visual heights as the desktop truth, align the tokens/usage so there is one source.
> - Sibling controls that share the height story: `icon-button`, `menu-button`, `copy-button`,
>   `button-group`, `input`, `textarea`, `select`, `toggle`, `checkbox`, `radio` (re-verify the list and
>   which already reference the tokens).

**Goal:** Interactive primitives derive their height from `--cascivo-control-height-*` (one source), and
reach ≥44px effective under `@media (pointer: coarse)`. Desktop appearance is unchanged. The touched
primitives are enrolled in the sweep with the touch-target assertion enabled.

**Architecture:** CSS-only changes in `packages/components/src/{button,icon-button,menu-button,copy-button,
button-group,input,textarea,select,toggle,checkbox,radio}/*.module.css` (re-verify exact set), plus
enrolling them in the T2 sweep spec.

**Tech stack:** unchanged. Pure CSS. Any `@function`/`if()` keeps a static fallback.

### Task 1: Reconcile Button height to the token source

**Files:** Modify `packages/components/src/button/button.module.css`; possibly
`packages/tokens/src/index.css`

- [ ] **Step 1:** Decide the single source. **Default:** Button's shipped desktop heights are the truth
      (2rem/2.5rem/3rem). Either (a) update `--cascivo-control-height-*` to match Button and re-verify
      every other consumer + `target-size.test.ts`, or (b) keep the tokens and accept Button's slightly
      different heights, introducing `--cascivo-button-height-*` that map to the control-height scale.
      Pick the option that yields **one** source with **no desktop visual change** and document why.
- [ ] **Step 2:** Replace the hardcoded `height: 2rem|2.5rem|3rem` with the chosen token reference. Verify
      the computed desktop height is pixel-identical (snapshot from T1's approach).

### Task 2: Coarse-pointer comfortable targets

**Files:** Modify the touched control CSS modules

- [ ] **Step 1:** For each interactive control, add a `@media (pointer: coarse)` block raising the
      effective minimum to `var(--cascivo-target-min-coarse, 2.75rem)` — via `min-block-size` and, where a
      control is icon-only/small, `min-inline-size`. Keep it additive: the base (fine-pointer) rule is
      unchanged, so desktop density is frozen.
- [ ] **Step 2:** For controls smaller than 44px (e.g. `sm` Button, `checkbox`, `radio`, `toggle`,
      icon-only buttons), ensure the **hit area** reaches 44px on coarse pointers even if the visual box
      stays small (padding or a pseudo-element hit target), preserving the visual design. Re-verify
      `checkbox`/`radio` keep their documented spacing-exception note in `target-size.test.ts`.
- [ ] **Step 3:** Confirm no `@media` width literal was introduced (only `pointer: coarse`), so
      `breakpoint:check` is unaffected; confirm `forced-colors`/`reduced-motion` guards are untouched.

### Task 3: Enroll primitives in the sweep

**Files:** Modify the T2 sweep spec

- [ ] **Step 1:** Add each touched primitive to the enrolled set. Enable `assertTapTargets` (coarse
      context) for them now that they meet 44px.
- [ ] **Step 2:** Add `assertNoOverflow` cases for any control that can grow (e.g. Input/Select at full
      width inside a 320px container) — verify they don't force horizontal scroll.
- [ ] **Step 3:** Retire any allowlist entry in `breakpoint-allowlist.ts` that belonged to a primitive
      now migrated (likely none — primitives mostly lacked width media; confirm).

### Task 4: Verify + commit

- [ ] **Step 1:** Run the existing component tests + the sweep + `target-size.test.ts`. Desktop visual
      checks unchanged.
- [ ] **Step 2: Commit** — `feat(components): single control-height source + coarse-pointer 44px targets (v20-t3)`

## Tranche exit criteria

1. Button (and siblings) derive height from one token source; the desktop computed heights are unchanged.
2. Every touched control reaches ≥44px effective tap target under `@media (pointer: coarse)` while the
   fine-pointer (desktop) size is untouched.
3. The sweep enrolls the primitives with the touch-target assertion on; all pass at 320/360/390/414.
4. `target-size.test.ts` still passes; no off-scale width literal introduced.
5. No `useEffect`/`useState`; CSS-only; `fallback:check` clean.
