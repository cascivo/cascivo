# v55 — Tranche 3: Authoring & Publishing (GitHub-Only)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let third parties **author and publish** templates with **zero backend, everything in GitHub** (M-4): a
`cascivo template init` starter, template support in `cascivo registry build`, and a `docs/CONTRIBUTING-TEMPLATES.md`
that mirrors `docs/CONTRIBUTING-REGISTRY.md`. The whole loop is: author a `src/<name>/` folder + a manifest → build to
static JSON → host on GitHub raw / Pages → reference by `owner/repo/<name>` → submit to the directory by PR (T4). No
host beyond GitHub (ROADMAP-V55 Q3 / Master Decision 3).

**Architecture:** three pieces.
- **`template init`** (`commands/template-init.ts`): scaffold `src/<name>/<name>.tsx` (+ a `.css` module), an optional
  `fixtures/<name>.ts`, a `<name>.template.ts` manifest (the `TemplateMeta` source authors edit), and an entry in
  `cascivo-registry.json` — analogous to writing a component, plus the `registryDependencies` list + screenshots.
- **`registry build` (templates)** (`commands/registry-build.ts`): when an index item is `type:'template'`, resolve it
  (validate with T1's `validateTemplate`, inline the template's own file contents, verify each `registryDependencies`
  name is resolvable) and emit `public/r/<name>.json` — the same static artifact components produce.
- **`CONTRIBUTING-TEMPLATES.md`**: the contributor guide — file layout, manifest, build, host (raw / GitHub Pages /
  Vercel), reference, test-locally, submit-to-directory, update — mirroring the registry guide so the two read as one
  family.

**Tech Stack:** `packages/cli/src/commands/{template-init,registry-build}.ts`; T1 registry validator; a copy-paste
`apps/examples/template-starter/`. No new dependency.

### Task 1: `cascivo template init`

**Files:** `commands/template-init.ts`

- [ ] **Step 1:** Prompt (or accept flags) for name, category, framework, and the component list. Scaffold
      `src/<name>/<name>.tsx` (a minimal page composing the chosen components), `<name>.module.css` (`@layer
      components`), and an optional `fixtures/<name>.ts`.
- [ ] **Step 2:** Generate `<name>.template.ts` exporting a `TemplateMeta` (intent/framework/category/screenshots
      placeholders/fileRoles/pages) — the file the author fills in, the single source for the manifest.
- [ ] **Step 3:** Create or append the `cascivo-registry.json` item: `type:'template'`, `registryDependencies` = the
      chosen components, `files` = the scaffolded sources with `target`s, `meta` = the `TemplateMeta`, plus
      `license`/`author`/`version`.
- [ ] **Step 4:** Test: `template init` produces a folder + manifest + registry entry that `validateTemplate` accepts
      (modulo the screenshot placeholders, which it flags as TODO).

### Task 2: Template support in `registry build`

**Files:** `commands/registry-build.ts`

- [ ] **Step 1:** For each `type:'template'` item: run `validateTemplate`; resolve + **inline** the template's own
      `files` contents (read from disk / fetch the URL), exactly as components are inlined today.
- [ ] **Step 2:** Verify every `registryDependencies` name is resolvable **within the same index** or via a referenced
      registry; emit a clear build error if a component is missing (so a template can't ship a dangling dependency).
- [ ] **Step 3:** Write `public/r/<name>.json` (the resolved template artifact). Test: building a template index emits
      a valid artifact; a missing component dependency or a failed `validateTemplate` fails the build with a precise
      message.
