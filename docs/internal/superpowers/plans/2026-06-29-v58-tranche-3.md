# v58 — Tranche 3: Automatic Contrast with `contrast-color()` (post 2)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make "text on a colored surface" choose its own legible color instead of relying on a hand-picked literal.
Today `--cascivo-color-text-on-accent`/`-on-destructive` are hand-chosen per theme (`light.css:39–40` `oklch(1 0 0)`;
`dark.css:38–39` `oklch(0.145 0.005 250)` / `oklch(1 0 0)`); if an accent is re-tinted (a custom theme, a brand
override, T4's generator), those literals can silently fall below WCAG AA. Post 2's **`contrast-color()`** returns a
WCAG-passing black/white for a given background. Compute the text-on-* tokens with it — behind a static literal
fallback (the existing contract, extended in T2) so non-supporting browsers keep today's value. (M-3, Master
Decision 3.)

**Architecture:** a per-theme swap of two tokens (and the `*-content` aliases that point at them) from literal to
`contrast-color()`-with-fallback, plus a CI assertion that the **fallback** value (what non-supporting browsers
render) passes AA.

- **The pattern**, in each `[data-theme]` block:

  ```css
  /* text on accent — static fallback (all browsers) then auto-contrast */
  --cascivo-color-text-on-accent: oklch(1 0 0);                    /* fallback (today's literal) */
  --cascivo-color-text-on-accent: contrast-color(var(--cascivo-color-accent));
  ```

  Same for `--cascivo-color-text-on-destructive` against `--cascivo-color-destructive`. The `*-content` aliases
  (`--cascivo-color-accent-content`, `--cascivo-color-destructive-content`) already `var()` onto these
  (`light.css:118–119`, `dark.css:115–116`) — they inherit the improvement for free; no change needed there beyond
  confirming the chain.

- **Why fallback matters for AA.** Non-supporting browsers render the *fallback*. So the acceptance gate is: the
  **fallback literal** passes AA against its base in every theme (it does today — this tranche must not regress it),
  and where `contrast-color()` is supported it can only *improve* legibility. The CI contrast check
  (`scripts/checks/color/contrast.ts`) already validates AA; point it at the fallback value.

**Tech Stack:** the 12 `packages/themes/src/*.css`, the contrast CI check (`scripts/checks/color/contrast.ts`). Pure
CSS + a possible one-line check adjustment. No new dependency. Depends on T2 (the fallback-contract extension that
makes `contrast-color()` legal under `fallback:check`).
