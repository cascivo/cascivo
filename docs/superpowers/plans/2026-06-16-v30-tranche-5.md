# v30 — Tranche 5: SearchDialog Lazy + Resource Hints

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Remove `@cascivo/search` from the initial JS bundle by lazy-loading `SearchDialog`, and add resource hints (`preload`, `dns-prefetch`) to the HTML head to prime the browser for the assets that matter most.

**Architecture:**

**SearchDialog lazy loading:** Currently `SearchDialog` is imported unconditionally in `App.tsx` and the `<SearchDialog>` element is rendered on every route, loaded in the initial chunk. The fix: (1) convert to `React.lazy()`; (2) only render after the first time the dialog is opened, controlled by a `hasOpenedSearch` signal. Once mounted, the dialog stays in the tree so subsequent opens are instant.

**Resource hints:** Add `<link rel="preload">` for the Vite-generated critical CSS and font (if any), and `<link rel="dns-prefetch">` for any external origins the page contacts. These hints are in `index.html` and have zero JS cost — the browser acts on them during HTML parsing.

**Tech Stack:** `React.lazy()`, `useSignal`, `useSignalEffect`, `<link rel="preload">`, `<link rel="dns-prefetch">`.

### Task 1: Lazy-load SearchDialog

**Files:**

- Modify: `apps/landing/src/App.tsx`

- [ ] **Step 1: Read `apps/landing/src/App.tsx` in full**

  Locate:
  1. The import: `import { SearchDialog } from '@cascivo/search/SearchDialog'`
  2. The `search` JSX variable (or wherever `<SearchDialog>` is rendered)
  3. The `searchOpen` signal import from `'./search/state'`

- [ ] **Step 2: Replace the eager import with `React.lazy()`**

  Delete:

  ```ts
  import { SearchDialog } from '@cascivo/search/SearchDialog'
  ```

  Add alongside the other lazy declarations:

  ```tsx
  const SearchDialog = lazy(() =>
    import('@cascivo/search/SearchDialog').then((m) => ({ default: m.SearchDialog })),
  )
  ```

- [ ] **Step 3: Add the `hasOpenedSearch` signal**

  Inside the `App()` component function body, add (alongside the existing `useSignals()` call):

  ```tsx
  const hasOpenedSearch = useSignal(false)

  useSignalEffect(() => {
    if (searchOpen.value) hasOpenedSearch.value = true
  })
  ```

  `hasOpenedSearch` becomes `true` the first time `searchOpen.value` is set to `true`. After
  that it stays `true` forever (the dialog chunk stays mounted).

- [ ] **Step 4: Gate the SearchDialog rendering**

  Find the existing `search` JSX definition. It is currently something like:

  ```tsx
  const search = (
    <SearchDialog
      index={landingIndex}
      open={searchOpen.value}
      onClose={() => {
        searchOpen.value = false
      }}
      onNavigate={navigateToResult}
    />
  )
  ```

  Replace with:

  ```tsx
  const search = hasOpenedSearch.value ? (
    <Suspense fallback={null}>
      <SearchDialog
        index={landingIndex}
        open={searchOpen.value}
        onClose={() => {
          searchOpen.value = false
        }}
        onNavigate={navigateToResult}
      />
    </Suspense>
  ) : null
  ```

  The `search` variable is then used in the JSX return as before — no change to the return
  structure needed.

- [ ] **Step 5: Verify the search still works**

  Build and preview:

  ```bash
  pnpm exec vp run @cascivo/landing#build
  pnpm exec vp run @cascivo/landing#preview
  ```

  Open `http://localhost:4180`. Press Cmd+K (macOS) or Ctrl+K (Linux/Windows). The dialog
  must appear. Close it and reopen it — the second open must be instant (no chunk reload).
  Navigate to `/examples`, open search again — must still work.

- [ ] **Step 6: Verify `@cascivo/search` is not in the initial chunk**

  ```bash
  ls -la apps/landing/dist/assets/*.js
  ```

  Look for a chunk with `search` or `SearchDialog` in the name — it should exist as a
  separate file, not be part of the main `index-*.js` chunk. The main chunk size should
  be smaller than before T5.

### Task 2: Add resource hints to `index.html`

**Files:**

- Modify: `apps/landing/index.html`

- [ ] **Step 1: Read `apps/landing/index.html` in full**

  Note the current `<head>` contents. Identify which external origins the page contacts:
  - GitHub (for the GitHub link) — `github.com`
  - Any other CDN or API endpoint

  Also check whether any fonts are loaded (look for `@font-face` in `landing.css` or
  Google Fonts link tags in the HTML).

- [ ] **Step 2: Add `dns-prefetch` for external origins**

  For each external origin the page contacts (not for subresources on the same origin),
  add a `<link rel="dns-prefetch">` early in `<head>`:

  ```html
  <!-- DNS prefetch for external origins -->
  <link rel="dns-prefetch" href="https://github.com" />
  ```

  Add only origins that are actually used. Do not add speculative hints for origins that
  are never fetched.

- [ ] **Step 3: Add `preconnect` for origins needed in the critical path**

  If the page loads fonts or other critical resources from a third-party origin (e.g.,
  Google Fonts), add:

  ```html
  <link rel="preconnect" href="https://fonts.gstatic.com" crossorigin />
  ```

  If no third-party critical-path resources exist, skip this step.

- [ ] **Step 4: Add `modulepreload` for the main app chunk**

  Vite emits `<link rel="modulepreload">` tags for preloading JS chunks automatically when
  building. Verify this by inspecting `apps/landing/dist/index.html` after a build — if
  Vite already adds `modulepreload` hints, no manual addition is needed here.

  If Vite does NOT add them (check the dist output), add manually:

  ```html
  <link rel="modulepreload" href="/src/main.tsx" />
  ```

  In practice, Vite handles this automatically for `type="module"` scripts.

- [ ] **Step 5: Verify `index.html` is still valid HTML**

  All resource hint links must be self-closing (`/>`), have `href` values, and be placed
  before the async CSS link added in T4. Order in `<head>` matters: resource hints should
  come before stylesheets so the browser can act on them earlier.
