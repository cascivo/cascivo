# v39 — Tranche 1: Per-Theme Font Theming

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make a theme able to carry its own typeface — the lesson from studying RetroUI, whose identity is
as much its display/pixel font as its borders. Add a `--cascivo-font-display` token and make
`--cascivo-font-{sans,mono,display}` **per-theme overridable**, declared across **all 12** themes so token
parity holds. This is the foundation `arcade` (T2) needs, and it independently resolves the long-pending
per-theme-font **open question** in `docs/THEME-PROPOSALS.md` (current recommendation: "keep fonts global").
No component changes; CSS tokens only.

**Architecture:** `--cascivo-font-{sans,mono}` live in `packages/tokens/src/index.css :root` only. Themes
are `@import '@cascivo/tokens'` + `@layer cascivo.theme { [data-theme='<name>'] { … } }`.
`parity.test.ts` asserts every theme defines an **identical** `--cascivo-*` set. To make fonts per-theme
overridable without breaking parity, the font keys must be present in the **same** form in every theme file
(either all themes inherit via the token default and override only when desired — but parity counts *declared*
keys, so confirm the test's mechanism first and follow whichever form keeps the set identical).

**Tech Stack:** CSS-only; vite+ (`vp`) for the theme test.
