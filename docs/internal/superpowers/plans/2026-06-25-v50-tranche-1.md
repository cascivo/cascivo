# v50 — Tranche 1: Drawer Below the Navbar (the seed bug)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fix the reported bug — the opened burger menu starts *behind* the navbar — so the drawer slides in cleanly
*below* the header with its first link fully visible. While there, kill the latent header-height drift (the height is
assumed three different ways) and make the drawer clear the bottom safe-area. Lens: **architect + designer +
business** (the menu is the most-used mobile control; a clipped first row reads as broken on the majority device).

**Architecture:** `.mobile-nav-drawer` (`landing.css:2297`) is `position: fixed; inset-block: 0; inset-inline-end: 0;
z-index: 100; padding: var(--cascivo-space-8) var(--cascivo-space-6); padding-block-start: max(var(--cascivo-space-8),
env(safe-area-inset-top))`. The header `.landing-shell-header` (`:3398`) is `position: sticky; top: 0; z-index:
var(--cascivo-z-overlay)` = **200** (`tokens/src/index.css:264`), height `--cascivo-shell-header-block-size` = **3rem**
(`:270`). So the header paints over the drawer's top, and `space-8` (~2rem) < `3rem`, hiding the first link. The
correct idiom already exists at `.scroll-progress` (`:1143`): `top: var(--cascivo-shell-header-block-size, 3.5rem)`
(note: its `3.5rem` fallback disagrees with the `3rem` token).

**Tech Stack:** `landing.css` only; tokens `--cascivo-shell-header-block-size`, `env(safe-area-inset-bottom)`. No
markup or JS change. No component-library change.
