# v13 Tranche 2 — Context Surfaces (Token Catalog, Context Bundle, MCP Tools)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development
> (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use
> checkbox (`- [ ]`) syntax for tracking.
>
> Current-state assumptions — RE-VERIFY before starting:
>
> - T1 landed: manifests carry `intent`; published registry items mirror it.
> - `packages/tokens/src/index.css` is the single token source (primitive → semantic →
>   component custom properties); `packages/themes` overrides the semantic layer.
> - `scripts/llms/generate.ts` emits `apps/docs/public/llms.txt` + `/llms/<name>.md`.
> - `@cascivo/mcp` server (`packages/mcp/src/server.ts`) registers tools via
>   `server.registerTool(...)`; tests live beside it.
> - The regen pipeline is `registry:generate` + `readme:generate` + `llms:generate` then the
>   drift gate.

**Goal:** Publish the closed set and the consolidated context so an agent can read the WHY
through the channels it already speaks: a machine-readable token catalog, a `context.json`
bundle + per-component `context.md`, an enriched `llms.txt`, and three new MCP tools.

**Architecture:** New generators in `scripts/` (catalog + context) join the token CSS, the
registry, the intent blocks, and (later) the specs into static JSON/markdown under
`apps/docs/public/`, wired into the regen pipeline and the drift gate. The MCP server gains
read tools over those artifacts (read fixtures in tests, no network).

**Tech stack:** unchanged. CSS parsing reuses the v12 CSS-contract classifier approach.

### Task 1: Token catalog generator

**Files:**

- Create: `scripts/catalog/generate.ts`
- Create: `scripts/catalog/parse-tokens.ts` (+ test `parse-tokens.test.ts`)
- Output: `apps/docs/public/tokens.catalog.json`
- Modify: root `package.json` scripts (add `catalog:generate`; chain into the regen flow)

- [ ] **Step 1: Parser.** `parseTokens(css: string, resolved: Record<string,string>)` →
      `{ name, value, layer, group, resolvedDefault }[]`:
  - extract every `--cascivo-*: <value>;` declaration from `index.css`
  - `layer`: classify by name convention — primitive (`--cascivo-color-blue-500`,
    `--cascivo-space-4`), semantic (`--cascivo-color-accent`, `--cascivo-color-border`),
    component (`--cascivo-button-bg`) — document the classification rules in the file
  - `group`: the segment after `--cascivo-` up to the next `-` family (e.g. `color`,
    `space`, `radius`, `font`, `shadow`, `button`)
  - `resolvedDefault`: if `value` is `var(--other)`, follow the chain within the catalog to a
    concrete value using the **light theme** resolution map; if it only resolves per-theme,
    set `resolvedDefault: null` and `resolvesPerTheme: true`
- [ ] **Step 2: Test** the parser on a fixture CSS snippet covering all three layers, a
      `var()` chain, and a per-theme-only token. Assert layers, groups, and resolution.
- [ ] **Step 3: Generator.** Read `packages/tokens/src/index.css` + the light theme file,
      build the resolution map, write `tokens.catalog.json` with a top-level
      `{ generatedFrom, resolutionTheme: 'light', tokens: [...] }`.
- [ ] **Step 4: Run** `pnpm catalog:generate`; eyeball the output; confirm counts match the
      number of `--cascivo-*` declarations.
- [ ] **Step 5: Commit** — `feat(scripts): token catalog generator (closed set for AI)`

## Tranche exit criteria

1. `tokens.catalog.json` lists every `--cascivo-*` property with layer, group, and a
   light-theme `resolvedDefault` (or `resolvesPerTheme: true`); regenerating after a CSS edit
   changes it and the drift gate catches a stale copy.
2. `context.json` indexes all 72 components with intent; each `context/<name>.md` exists and
   carries that component's WHY; `llms.txt` links the bundle and summarizes intent per item.
3. The generators run inside the standard regen; `git diff --exit-code` is clean after regen.
4. MCP exposes `get_tokens`, `get_context`, `select_component`; tests cover filtering,
   lookup, deterministic ranking, and the unreachable-catalog path.
5. Root gate + drift check exit 0.
