# v43 — Tranche 7: `FlowStory` — Scripted Storyline Animation

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship the headline differentiator the static animated edge (T4) cannot give: a **scripted, sequenced,
looping flow animation** that walks the viewer through a graph step by step, fading in a label/description at
each step so the user *understands* the flow rather than watching one static loop. Ship `<FlowStory>` driven by
a **super-easy, serializable `script`** array.

**The motivating example.** Three nodes `A <--> B --> C`. The author wants: animate **A→B**, then **B→A**, then
**A→B**, then **B→C**, then loop from the start — each step fading in its own caption. The script is exactly:

```tsx
<FlowStory
  nodes={[{ id: 'A', data: { label: 'Client' } }, { id: 'B', data: { label: 'Gateway' } }, { id: 'C', data: { label: 'Service' } }]}
  edges={[{ id: 'ab', source: 'A', target: 'B' }, { id: 'bc', source: 'B', target: 'C' }]}
  script={[
    { from: 'A', to: 'B', label: 'Request sent' },
    { from: 'B', to: 'A', label: 'Acknowledged' },
    { from: 'A', to: 'B', label: 'Payload streamed' },
    { from: 'B', to: 'C', label: 'Forwarded to Service' },
  ]}
  loop
/>
```

A step is `{ from, to, label?, description?, duration? }`. The engine **resolves the edge** connecting
`from`↔`to` (so the one edge `ab` serves both `A→B` and `B→A`, animating in the step's direction — reversed for
`B→A`), animates a particle/dash traveling along it, **fades in the step's caption**, then auto-advances; after
the last step it loops (when `loop`). This reads like a storyboard and is plain JSON an agent can emit
(AI-first). A power-user alternative `{ edge: 'ab', reverse?: true }` is also accepted.

**Architecture (research → design).** Three concerns, split by CLAUDE.md's "HTML+CSS first, JS only where
required":

1. **Sequencing (the one place JS is genuinely required).** Stepping through an arbitrary script with arbitrary
   per-step durations and looping cannot be expressed in static CSS keyframes. Use the **established cascivo
   timing idiom**: a `currentStep` signal advanced by a `setTimeout` chain inside **`useSignalEffect`** (the
   `carousel` autoplay / v42 `relative-time` interval pattern — *not* `useEffect`), with an **injectable clock**
   so tests advance deterministically (fire ticks, never wall-clock). Loop by wrapping `currentStep` modulo
   `script.length`. Optionally model it as a `createMachine`/`useMachine` FSM (one state per step, timed
   auto-advance) — justified here because the component itself drives the transitions.
2. **Motion (CSS).** The active step sets `data-active` on its resolved edge → the edge's animated dash/particle
   (reuse the T4 `[data-animated]` `stroke-dashoffset` keyframe) runs; `data-direction="reverse"` flips the
   travel direction for `to→from` steps. The active caption fades in via opacity/transform transition
   (`@starting-style` enter). All movement is CSS; the signal only flips which edge/caption is active.
3. **Comprehension + a11y.** The current caption renders as real text in an `aria-live="polite"` region so
   each step is announced; a play/pause control and prev/next step controls make it keyboard-operable and
   user-paced; a step indicator (e.g. "2 / 4") orients the viewer. Under
   `@media (prefers-reduced-motion: reduce)` **no travel motion plays** — the storyline still advances on a
   dwell timer (or only via the manual prev/next controls) and highlights the active edge + caption statically,
   so the information is never lost.

`<FlowStory>` composes `<Flow>` (T6): it renders the graph, then overlays the storyline controller, the active
caption, and the controls; it drives the active edge via the `currentStep` signal. `currentStep` and `playing`
are controllable (`useControllableSignal`) with `onStepChange`.

**Tech Stack:** signal-driven TSX + CSS Modules; `@cascivo/core` (`useSignal`/`useComputed`/`useSignalEffect`/
`useControllableSignal`/`useSignals`, optionally `createMachine`/`useMachine`, `cn`); the T4 animated-edge CSS,
the T6 `<Flow>`; `@cascivo/i18n` for control labels; an injectable clock for deterministic tests; Vitest +
Testing Library.

### Task 2: `engine/script.ts` — resolve a step to an edge + direction

**Files:** `packages/flow/src/engine/script.ts`

- [ ] **Step 1:** Types: `StoryStep = { from: string; to: string; label?: string; description?: string;
      duration?: number } | { edge: string; reverse?: boolean; label?: string; description?: string;
      duration?: number }`; `ResolvedStep = { edgeId: string; direction: 'forward' | 'reverse'; label?; … }`.
- [ ] **Step 2:** `resolveStep(step, edges)`: for `{from,to}` find the edge whose endpoints match (either
      orientation); `direction='forward'` when `edge.source===from`, else `'reverse'`. For `{edge,reverse}`
      pass through. Throw a clear error if no matching edge (fail-fast on a bad script). Pure + deterministic.
- [ ] **Step 3:** `resolveScript(script, edges)` → `ResolvedStep[]`. Unit-testable without DOM.

### Task 3: `core/use-story.ts` — the sequencer

**Files:** `packages/flow/src/core/use-story.ts`

- [ ] **Step 1:** `useStory({ steps, loop, playing, currentStep, onStepChange, stepDuration, now? })`:
      `currentStep` + `playing` signals (controllable via `useControllableSignal`); `now?` is the injectable
      clock/timer factory (default real `setTimeout`).
- [ ] **Step 2:** In `useSignalEffect`: when `playing`, schedule advance after the active step's `duration ??
      stepDuration`; on fire, `currentStep = (currentStep + 1)`; wrap modulo `steps.length` when `loop` (stop
      at the end otherwise). Clear the timer in cleanup. SSR-guarded. **No `useEffect`/`setState`.**
- [ ] **Step 3:** Expose `play`/`pause`/`next`/`prev`/`restart` actions and the `activeStep` (`useComputed`).
      Under `prefers-reduced-motion` (read via `@cascivo/core` `useMediaQuery`), still advance on the dwell
      timer but signal `reducedMotion` so the view suppresses travel motion.
