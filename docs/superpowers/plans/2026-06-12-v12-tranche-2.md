> Current-state assumptions — RE-VERIFY before starting:

>   layouts/charts equivalents — confirm their CSS conventions before including them in
>   the sweep scope).
> - Every `component.meta.ts` carries `tokens: string[]` (the CSS custom properties the
>   component reads) — the theming-docs generator's input.
> - Docs `ComponentPage.tsx` renders manifest-driven sections; llms/conformance
>   generators (v11) established the generated-data → docs-page pattern.
> - MCP `generateThemeCss` was hoisted to a shared location in v11-T6 (find where);

>   reuse its prompt utility/style.

**Goal:** Pain #2 (styling lock-in, `!important` wars, rigid tokens) becomes structurally
impossible to regress: a repo-wide CI gate proves every visual property is token-driven
and `!important`-free; every component's docs page documents exactly which tokens to
override (generated, with a live editor); and `cascade theme create` gives CLI users the
same theme-building power the MCP has.

**Architecture:** The contract test is a vitest in the components package (pure
file-parsing, no DOM). The theming docs flow: generator script → JSON artifact →
ComponentPage section. `theme create` wraps existing shared theme-generation machinery.

**Tech stack:** unchanged. CSS parsing for the contract test: use a real CSS tokenizer if
one is already in the tree (check what vp/oxfmt expose); else a deliberately-simple
declaration-level regex parser with its own tests — no new heavyweight dep.

### Task 1: The contract test

**Files:**

- Create: `packages/components/src/css-contract.test.ts`
- Create: `packages/components/css-contract.allowlist.json` (committed, commented via a
  `_comment` convention or sibling md)

- [ ] **Step 1:** Define the rule set precisely (this list is the contract — copy it into
      the test header):
  - **Rule 1:** zero `!important` in any component/layout CSS file. No allowlist for
    this rule.
  - **Rule 2:** every declaration of a _visual property_ — `color`, `background*`,
    `border*` (color/width facets), `box-shadow`, `outline*`, `border-radius`,
    `margin*`, `padding*`, `gap`, `inset*`, `font-size`, `font-weight`, `font-family`,
    `line-height`, `transition-duration`, `animation-duration` — must resolve to
    `var(--cascade-…)`, `var(--_…)` (private knob), or a structural keyword
    (`0`, `none`, `auto`, `transparent`, `currentColor`, `inherit`, percentages and
    `em`-relative line-heights — finalize the keyword list while implementing against
    real files).
  - **Rule 3:** anything else needs an allowlist entry: `{ file, line-pattern, reason }`.
- [ ] **Step 2:** Implement the parser + test; failure output is `file:line: declaration
— rule violated`.
- [ ] **Step 3:** **The sweep.** First run will find real violations. Fix genuine ones
      (hardcoded values → tokens or `--_*` knobs with token defaults); allowlist only
      with a written reason. Keep fixes surgical — value moves to a token, nothing else
      changes. Eyeball any visually-affected component in docs/Storybook after.
- [ ] **Step 4:** Sanity-check the gate: introduce a deliberate violation, watch it fail
      with the right location, revert.
- [ ] **Step 5: Commit** (possibly several during the sweep) —
      `test(components): CSS customization contract — no !important, token-driven visuals`

   both tested.
