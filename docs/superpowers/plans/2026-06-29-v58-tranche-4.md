**Goal:** Make the agent-facing theme generator consistent with the all-OKLCH system and the T2/T3 derivation model.
Today `packages/mcp/src/theme.ts` (the MCP `create_theme` tool) derives shades with `color-mix(in oklab, â€¦,
black/white X%)` and **hardcodes hex status colors** â€” `--cascivo-color-destructive: #dc2626`,
`--cascivo-color-success: #16a34a`, `--cascivo-color-warning: #f59e0b` (`theme.ts:62â€“68`). In a system whose
hand-authored themes (after T2/T3) derive via `oklch(from var(--base) â€¦)` + `contrast-color()`, an agent that calls
`create_theme` should get a theme built the **same way** â€” OKLCH relative-color ramps, automatic contrast, no hex.
(M-4, Master Decision 5; post 1's "hold the hue and lightness, step the chroma.")

**Architecture:** rewrite the generator's color math, keep its public shape. `generateThemeCss(colors, name)` still
takes `{ primary, neutral, accent }` and emits a `@layer cascivo.theme { [data-theme='name'] { â€¦ } }` block â€” but
every derived value uses relative color syntax with a static fallback, and status colors are derived in-OKLCH rather
than pasted as hex.

- **Replace `darken`/`lighten` (`color-mix` helpers, `theme.ts:11â€“18`)** with relative-color emitters that produce a
  **literal-then-derived pair** per token, mirroring T2:

  ```ts
  // hover = base, one lightness step darker; fallback = a precomputed OKLCH literal
  `  --cascivo-color-accent-hover: ${oklchLiteral(primary, { l: -0.077 })};\n` +
  `  --cascivo-color-accent-hover: oklch(from ${primary} calc(l - 0.077) c h);\n`
  ```

  where `oklchLiteral(...)` computes the fallback by parsing the input to OKLCH and applying the same delta (a tiny
  pure helper â€” parse `oklch()`/hex/named input â†’ `{l,c,h}` â†’ apply delta â†’ serialize). The derived line is the
  progressive form; the literal is the fallback. Both are needed: the output must satisfy the same
  static-fallback contract as the hand-authored themes (and `fallback:check` if a consumer runs it over generated
  CSS).

- **Status colors in-OKLCH, derived from the inputs or fixed OKLCH constants** â€” not hex. Destructive/success/
  warning can be fixed OKLCH brand-neutral constants (the OKLCH equivalents of today's hex, computed once) so the
  generator stays deterministic, and their `-subtle`/`-hover` derive via relative color from those constants. No
  `#rrggbb` anywhere in the output.

- **`contrast-color()` for text-on-***, with an OKLCH literal fallback (`oklch(1 0 0)` or a computed contrast),
  matching T3. So a generated theme gets guaranteed-legible on-color text for free.

- **Surfaces (`neutral`-derived)**: keep deriving bg/surface/border/text from `neutral`, but via `oklch(from
  var(--neutral-as-token) â€¦)` / relative color rather than `color-mix(â€¦ white/black %)`, each with a literal
  fallback. (Functionally similar to today, but in the system's idiom.)
