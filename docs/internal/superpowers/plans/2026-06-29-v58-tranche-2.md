# v58 — Tranche 2: Derivable Color in the Shipped Themes — Relative Color Syntax + `@property` (post 2)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stop hand-authoring the same color *ladder* in 12 theme files. Today each theme restates
`accent/primary/secondary/destructive` `hover/active/subtle/muted` as literal OKLCH values; post 2's **relative color
syntax** derives them from a single base (`oklch(from var(--base) calc(l − .08) c h)`). Derive them in CSS — behind a
static fallback so non-supporting browsers keep today's literal — and register the themeable color custom properties
with **`@property`** so they interpolate on theme transitions and gain a typed substrate. (M-2, Master Decisions 2/4;
CLAUDE.md progressive-enhancement contract.)

**Architecture:** a per-base derivation pattern applied across all 12 selectable themes, plus one new
`@property`-registration file.

- **Relative-color derivation, with the literal as fallback.** For each base (`accent`, `primary`, `secondary`,
  `destructive`) and each state slot, emit two declarations in the theme's `[data-theme]` block — the existing
  literal first (the static fallback every browser uses), then the derived form (progressive). Example:

  ```css
  /* hover: darker — static fallback (all browsers) then derived (relative color) */
  --cascivo-color-accent-hover: oklch(0.546 0.224 250);            /* fallback */
  --cascivo-color-accent-hover: oklch(from var(--cascivo-color-accent) calc(l - 0.077) c h);
  ```

  The same shape for `-active` (further lightness step), `-subtle` (low chroma + high lightness, or an alpha:
  `oklch(from var(--cascivo-color-accent) l c h / 0.1)`), and `-muted` (`/ 0.2`). The exact deltas per slot come from
  diffing each theme's *current* literal against its base so the derived value lands on (or very near) today's value
  — derivation must not visually shift the themes.

- **`@property` registration.** Register the themeable color custom properties (`--cascivo-color-accent`,
  `-accent-hover`, …, `-primary*`, `-secondary*`, `-destructive*`, the `text-on-*` set T3 will touch) with
  `syntax: "<color>"; inherits: true; initial-value: …`. Registration is global (a `properties.css` imported by
  `all.css`), not per-theme; theme blocks still override the *values*.

- **Fallback contract extension.** `pnpm fallback:check` today enforces a preceding static fallback for
  `@function`/`if()` uses. Extend it to also require a static fallback when a declaration's value contains
  `oklch(from ` / `lab(from ` / `color(from ` / `contrast-color(` (T3). This keeps the progressive-enhancement
  invariant honest for the new constructs.

**Tech Stack:** all 12 `packages/themes/src/*.css`, a new `packages/tokens/src/properties.css` (+ `all.css` import),
the `fallback:check` script under `scripts/checks/*`, `packages/themes/src/parity.test.ts`. Pure CSS + a small
TypeScript check change. No new dependency.

### Task 1: Establish the derivation pattern on one theme (light)

**Files:** `packages/themes/src/light.css`

- [ ] **Step 1:** Enumerate the derivable variants in `light.css`: for bases `accent`, `primary`, `secondary`,
      `destructive`, the slots present today (`hover`, `active`, `subtle`, `muted` where they exist — `accent` has
      all; some bases have a subset). For each, read the **current literal** and the **base** value.
- [ ] **Step 2:** Compute the relative-color expression that reproduces each current literal from its base. For
      darken-style slots (`hover`/`active`): `oklch(from var(--base) calc(l - Δl) c h)` where `Δl` = base.L −
      literal.L (and adjust `c` only if the current literal also stepped chroma — prefer holding `c`/`h`, per post
      1's "step the chroma/lightness, hold the hue"). For tint-style slots (`subtle`/`muted`): prefer the alpha form
      already used in `dark.css:46–47` (`oklch(from var(--base) l c h / 0.1)`), which matches the existing
      subtle/muted semantics, unless the current literal is opaque (then derive via lightness+chroma steps).
- [ ] **Step 3:** Emit each as a **literal-then-derived pair** in the `[data-theme='light']` block (fallback first).
      Verify in a browser/devtools that the derived value resolves to (or within an imperceptible ΔE of) the original
      literal — derivation must be value-preserving, not a restyle. If a slot's literal cannot be expressed as a
      clean relative-color step (a bespoke hand-tuned value), **leave it as the literal only** and note it; not every
      value must be derived — only the mechanical ladders.
