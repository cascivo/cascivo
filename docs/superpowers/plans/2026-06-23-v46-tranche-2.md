**Goal:** Make external/controlled `value` updates **safe** â€” the second đź”´ blocker (gap-analysis **#4**). Today,
when the parent changes the controlled `value` (a remote pull, a programmatic reset, an external transaction),
the textarea re-renders with the new string and the **caret jumps to the end**; worse, the resulting change can
**echo-loop** with `onValueChange`. v46 internalizes the app's `syncDocPreservingSelection` + `isUpdatingFromStore`
pattern: diff the old/new text, **rebase the caret/selection** across the change, apply under an
`isApplyingExternal` guard so our own handler does not re-emit, and add a small imperative **`CodeEditorHandle`**
(`forwardRef`) for callers that already hold a transaction. Builds directly on T1's history (external sets seed,
not corrupt, the undo stack).

**Architecture:** The controllable `text` signal already mirrors the `value` prop. Add a `lastEmitted` ref so we
can tell our **own** echo (we just called `onValueChange`) from a genuine **external** change. On an external
change: compute a minimal `{ from, to, insert }` diff via common-prefix/common-suffix scan, map the current
`selectionStart/End` through it (offsets before `from` unchanged; inside the replaced span clamp to `from+insert.length`;
after shift by `insert.length - (to-from)`), then set `textarea.value` + restored selection under
`isApplyingExternal = true`. The imperative handle exposes the same primitive: `applyEdit({ from, to }, text)`
mutates text + selection + history coherently. `forwardRef<CodeEditorHandle, CodeEditorProps>` adds the handle
without changing the existing prop surface.

**Tech Stack:** owned TypeScript; `@cascivo/core` signals; `useRef` for the textarea + `lastEmitted` + handle;
`useImperativeHandle` is a React hook **for refs** (allowed â€” it is not state; if the project bans it, expose the
handle by assigning to the forwarded ref in a `useSignalEffect`). Confirm the project's stance in Task 1.
