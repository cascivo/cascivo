**Goal:** A security or a11y fix published upstream _reaches_ projects that vendored the
code: advisories ride the registry, `cascade audit` checks the lockfile like `npm audit`,
`cascade doctor --drift` tells users exactly which installed components are pristine,
customized, or conflicted — and a documented GitHub Action turns all of it into scheduled
PRs.

**Architecture:** Advisory types + matching live in `@cascade-ui/registry` (shared by CLI
and the directory CI). Drift classification lives in a CLI util consumed by both `doctor
--drift` and `audit` (provenance column). The automation is a composite action in
`.github/actions/cascade-update/` plus a copy-paste workflow recipe in docs.

**Tech stack:** unchanged. Semver-range matching needs a resolver — check what the repo
already uses (pnpm brings `semver` transitively; the CLI may add it as a direct dep).


```
combobox    1.2.0   high      CSA-2026-003  1.4.1    customized (12 lines)
```

      Footer: per-severity counts + the fix path (`cascade update combobox`). For
      affected+customized items, print the explicit caveat that `update` three-way merges

## Tranche exit criteria

1. Advisory published in a registry reaches `cascade audit` output in a project pinning
   an affected version (first-party AND community-fixture registries); clean projects
   report clean; offline registries are distinguishable from clean.
2. `doctor --drift` classifies the full scenario matrix correctly with per-file line
   counts; exit codes as specified.
3. `update --yes` merges clean items and skips conflicted ones machine-readably.
4. The composite action produced a real PR against a fixture repo with the specified
   body sections; re-run updated the same PR.
5. Root gate + drift check exit 0.
