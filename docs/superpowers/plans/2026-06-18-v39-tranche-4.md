# v39 — Tranche 4: shadcn-Registry-Compatible Export

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Adopt RetroUI's distribution reach without changing cascivo's own. RetroUI rides the **shadcn
CLI** (`npx shadcn add https://retroui.dev/r/button.json`) and its large install base. cascivo has its own
`registry.json` + `cascivo` CLI. This tranche **adds** a shadcn-schema-compatible emitter so anyone can
`npx shadcn add <cascivo-url>/r/<name>.json` and get a cascivo component — **purely additive interop**, no
change to cascivo's `registry.json` schema (consumed by the CLI, MCP, and docs) and no change to
`npx cascivo add`.

**Architecture:** `packages/registry` builds cascivo's `registry.json` from component metas
(`build.ts`, `validate.ts`, `directory.ts`, `types.ts`, `schema/`). T4 adds `shadcn.ts` that **maps** the
already-built cascivo registry entries into shadcn's `registry:component` item schema and writes a
per-component `/r/<name>.json`, hooked into `build.ts` so it emits alongside the existing output.

**Tech Stack:** TypeScript emitter; Vitest. No runtime dependency added to shipped packages — this is
build-time output.

### Task 1: Pin the shadcn registry item schema

**Files:** (research) `packages/registry/src/types.ts`, `packages/registry/schema/*`

- [ ] **Step 1:** Read cascivo's existing registry types/schema and `build.ts` to learn the in-memory shape of
      a built entry (`name`, `type`, `category`, `description`, `files: string[]` (raw GitHub URLs),
      `dependencies`, `tags`, `meta`).
- [ ] **Step 2:** Pin the target shadcn item schema fields the mapper will emit:
      `$schema`, `name`, `type: 'registry:component'`, `description`, `dependencies` (npm),
      `registryDependencies` (other cascivo items, by name/url), and `files: [{ path, content?, type:
      'registry:component' | 'registry:lib' }]`. Decide **inline `content`** (fetch/read the source at build
      time) vs **base-URL'd files** — _recommendation: inline content_ so `npx shadcn add` works without a
      second round-trip, matching how RetroUI ships. Record the decision in `shadcn.ts`'s header comment.
- [ ] **Step 3:** Note CSS-native caveats: cascivo components are `.tsx` + `.module.css` and rely on a
      `@cascivo/themes` import + `@cascivo/core` peer — there is **no** Tailwind `cssVars`/`tailwind` block to
      emit. The mapper leaves those shadcn fields empty/omitted and documents the prerequisite imports.

### Task 2: Implement the mapper

**Files:** `packages/registry/src/shadcn.ts`

- [ ] **Step 1:** Export `toShadcnItem(entry: RegistryEntry): ShadcnRegistryItem` — map cascivo fields to the
      pinned schema. `dependencies` = npm deps (e.g. `@cascivo/core`); `registryDependencies` = any cascivo
      components this one depends on (derive from the entry's component deps if present); `files` = each source
      file as `{ path: '<name>/<file>', content: <read source>, type }`.
- [ ] **Step 2:** Export `writeShadcnRegistry(entries, outDir)` — write one `r/<name>.json` per entry plus an
      `r/registry.json` index (array of item summaries) so the output is browseable. Use the repo's existing
      file-write/util conventions from `build.ts`.
- [ ] **Step 3:** Keep it pure/deterministic: same input registry → byte-identical output (sorted keys, stable
      ordering) so the drift gate stays green.

### Task 3: Hook into the build

**Files:** `packages/registry/src/build.ts`

- [ ] **Step 2:** Ensure `pnpm regen` triggers this path so the shadcn output regenerates with everything else
