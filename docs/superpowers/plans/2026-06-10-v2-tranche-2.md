Form is a typed signal store (`createForm`) plus a thin `<Form>` element. CommandMenu gains tests, nested pages, and a loading state, then mounts in the docs header as ⌘K search. **Tech stack:** as tranche 1 (signals via `@cascade-ui/core`, strings via `@cascade-ui/i18n` builtin catalog, Vitest + Testing Library, Playwright for the perf gate).

**Conventions (same as tranche 1):** `vp check` before every commit; no semicolons, single quotes; GPG signing is broken on this machine — commit with `git -c commit.gpgsign=false commit …`; feature branch, never `main`.

**The re-render contract (replaces the roadmap's literal "zero"):** with `@preact/signals-react`, a component that reads `signal.value` in render re-renders when that signal changes — so a sort click re-renders the table body **once**. True zero would require per-cell signal text bindings, which arbitrary `renderCell` JSX rules out. The enforced contract is:

1. Components **outside** the DataTable: **0 commits** on any table interaction (no state lifting, ever).
2. Table interactions (sort, filter, page, select, expand): **≤ 1 commit** each — one body re-render, no cascades, no commit-per-row.
3. Search keystrokes: **0 commits** until the debounce fires (the input is uncontrolled; only the debounced query write commits).
4. Paint cost of off-screen rows: eliminated via `content-visibility: auto` (CSS, not commits).

The 10k-row _felt_ performance is enforced by the Playwright latency budgets (Task 6), which is what the roadmap's killer argument actually needs.

**Context paths:** `packages/components/src/data-table/data-table.tsx` (read before Tasks 1–5), `packages/components/src/command-menu/command-menu.tsx` (before Tasks 7–9), `packages/components/src/pagination/` (i18n labels pattern), `packages/components/src/dropdown/` (open/close machine + keyboard pattern), `docs/superpowers/plans/2026-06-10-v2-tranche-1.md` (conventions precedent).

### Task 3: DataTable keyboard navigation + selection announcements

**Files:**

- Modify: `packages/components/src/data-table/data-table.tsx`

- [ ] **Step 1: Grid keyboard navigation (roving focus over interactive cells)**

Scope (deliberate): arrow-key navigation moves focus among the table's interactive controls (sort buttons in the header row, row checkboxes, expand buttons) — not a full spreadsheet grid. Native table semantics plus focusable controls already cover WCAG; arrows are the DX upgrade.

Add a keydown handler on the `<table>` element (`import type { KeyboardEvent } from 'react'` joins the existing react type imports):

```tsx
const onTableKeyDown = (e: KeyboardEvent<HTMLTableElement>) => {
  if (!['ArrowUp', 'ArrowDown', 'ArrowLeft', 'ArrowRight'].includes(e.key)) return
  const table = e.currentTarget
  const focusables = Array.from(
    table.querySelectorAll<HTMLElement>('th button, td button, td input[type="checkbox"]'),
  )
  const index = focusables.indexOf(document.activeElement as HTMLElement)
  if (index === -1) return
  e.preventDefault()
  const row = (el: HTMLElement) => el.closest('tr')
  let next: HTMLElement | undefined
  if (e.key === 'ArrowRight' || e.key === 'ArrowLeft') {
    const dir = e.key === 'ArrowRight' ? 1 : -1
    const candidate = focusables[index + dir]
    if (candidate && row(candidate) === row(focusables[index]!)) next = candidate
  } else {
    const dir = e.key === 'ArrowDown' ? 1 : -1
    const current = focusables[index]!
    for (let i = index + dir; i >= 0 && i < focusables.length; i += dir) {
      if (row(focusables[i]!) !== row(current)) {
        next = focusables[i]
        break
      }
    }
  }
  next?.focus()
}
```

Wire: `<table … onKeyDown={onTableKeyDown}>`. (RTL note: arrow semantics follow DOM order, which CSS logical properties already mirror — no extra handling.)

```ts
import { expect, test } from '@playwright/test'

// CI baseline budgets (ms). PERF_SCALE loosens them on slower machines (e.g. PERF_SCALE=2 locally).
const scale = Number(process.env.PERF_SCALE ?? 1)
const BUDGET = {
  initialRender: 3000 * scale,
  sort: 100 * scale,
  keystroke: 50 * scale,
}

test('10k-row DataTable stays within latency budgets', async ({ page }) => {
  const start = Date.now()
  await page.goto('/perf/data-table')
  await page.locator('tbody tr').first().waitFor()
  expect(Date.now() - start).toBeLessThan(BUDGET.initialRender)

  // Sort: click header, measure until the first row's text changes.
  const firstCell = page.locator('tbody tr').first().locator('td').nth(0)
  const before = await firstCell.textContent()
  const sortStart = Date.now()
  await page.getByRole('button', { name: /score/i }).click()
  await expect(firstCell).not.toHaveText(before ?? '')
  expect(Date.now() - sortStart).toBeLessThan(BUDGET.sort)

  // Keystroke feedback: typing must paint the character within budget.
  const search = page
    .getByRole('searchbox')
    .or(page.getByPlaceholder(/search/i))
    .first()
  const typeStart = Date.now()
  await search.pressSequentially('u')
  await expect(search).toHaveValue('u')
  expect(Date.now() - typeStart).toBeLessThan(BUDGET.keystroke)
})
```

### Task 8: CommandMenu nested pages + loading state

Roadmap 1.5 requires "nested pages, async items". Design decisions (locked):

- **Nested pages:** an item may carry `page: { placeholder?: string; groups: CommandGroup[] }` instead of `onSelect`. Selecting it pushes the page onto a stack signal; query resets; Backspace on an empty query (or Escape) pops one level before closing.
- **Async items:** host-driven. New props `onQueryChange?: (query: string) => void` and `loading?: boolean`. The host fetches on query change and swaps `groups`; `loading` renders a spinner row in place of the empty state. No built-in fetch/cache — JSON-serializable props stay the contract.

```tsx
'use client'
import { batch, cn, signal, useSignal, useSignals } from '@cascade-ui/core'
import type { Signal } from '@cascade-ui/core'
import type { FormHTMLAttributes, ReactNode } from 'react'
import styles from './form.module.css'

type Errors<T> = Partial<Record<keyof T, string>>
type Touched<T> = Partial<Record<keyof T, boolean>>

export interface FormConfig<T extends Record<string, unknown>> {
  initialValues: T
  /** Sync or async; return a (partial) field→message map. Empty map = valid. zod users: wrap safeParse here. */
  validate?: (values: T) => Errors<T> | Promise<Errors<T>>
}

export interface FormStore<T extends Record<string, unknown>> {
  values: Signal<T>
  errors: Signal<Errors<T>>
  touched: Signal<Touched<T>>
  submitting: Signal<boolean>
  field<K extends keyof T>(
    name: K,
  ): {
    value: T[K]
    onChange: (value: T[K]) => void
    onBlur: () => void
    error: string | undefined
  }
  setValue<K extends keyof T>(name: K, value: T[K]): void
  submit(onValid: (values: T) => void | Promise<void>): Promise<void>
  reset(): void
}

export function createForm<T extends Record<string, unknown>>(config: FormConfig<T>): FormStore<T> {
  const values = signal<T>({ ...config.initialValues })
  const errors = signal<Errors<T>>({})
  const touched = signal<Touched<T>>({})
  const submitting = signal(false)

  const setValue = <K extends keyof T>(name: K, value: T[K]) => {
    values.value = { ...values.value, [name]: value }
  }

  return {
    values,
    errors,
    touched,
    submitting,
    setValue,
    field(name) {
      return {
        value: values.value[name],
        onChange: (value) => setValue(name, value),
        onBlur: () => {
          touched.value = { ...touched.value, [name]: true }
        },
        error: errors.value[name],
      }
    },
    async submit(onValid) {
      submitting.value = true
      try {
        const result = (await config.validate?.(values.value)) ?? {}
        const allTouched = Object.fromEntries(
          Object.keys(config.initialValues).map((k) => [k, true]),
        ) as Touched<T>
        batch(() => {
          errors.value = result
          touched.value = allTouched
        })
        if (Object.keys(result).length === 0) await onValid(values.value)
      } finally {
        submitting.value = false
      }
    },
    reset() {
      batch(() => {
        values.value = { ...config.initialValues }
        errors.value = {}
        touched.value = {}
      })
    },
  }
}

/** Stable per-mount form store. Config is captured on first render (matches the prop-sync rules). */
export function useForm<T extends Record<string, unknown>>(config: FormConfig<T>): FormStore<T> {
  useSignals()
  const store = useSignal<FormStore<T> | null>(null)
  if (store.peek() === null) store.value = createForm(config)
  return store.peek() as FormStore<T>
}

export interface FormProps<T extends Record<string, unknown>> extends Omit<
  FormHTMLAttributes<HTMLFormElement>,
  'onSubmit'
> {
  form: FormStore<T>
  onValid: (values: T) => void | Promise<void>
  children: ReactNode
}

export function Form<T extends Record<string, unknown>>({
  form,
  onValid,
  children,
  className,
  ...props
}: FormProps<T>) {
  return (
    <form
      noValidate
      className={cn(styles['form'], className)}
      onSubmit={(e) => {
        e.preventDefault()
        void form.submit(onValid)
      }}
      {...props}
    >
      {children}
    </form>
  )
}
```

`useForm` justification (the authoring-rules tension): `useState`/`useMemo` are banned; `useSignal` is the sanctioned primitive and gives per-mount stability. The `store.peek() === null` lazy-init is the same render-time signal-sync pattern CLAUDE.md documents for controlled props. If review rejects it, the fallback is documenting `createForm` at module scope only — but the hook ships first.
