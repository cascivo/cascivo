# v14 Tranche 4 — WCAG 2.2 AA + APG Conformance + Media-Feature Audit

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development
> (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use
> checkbox (`- [ ]`) syntax for tracking.
>
> Current-state assumptions — RE-VERIFY before starting:
>
> - `AccessibilityMeta` in `packages/core/src/types.ts`: `{ role: string; wcag: 'AA' | 'AAA';
>   keyboard: string[] }`; every `*.meta.ts` sets `wcag: 'AA'`.
> - `prefers-reduced-motion` is handled in ~7 components only (spinner, dropdown, progress-bar,
>   skeleton, tooltip, progress-circle, status); no broad `forced-colors`/`prefers-contrast`.
> - The registry generator copies `accessibility` into published items; `data.ts` on the
>   landing a11y page reads `role/wcag/keyboard`.
>
> **Sequence rule (risk #7):** upgrade a component to meet a 2.2 criterion → run its check →
> THEN flip its manifest `wcag` value. Never flip the string before the behavior is true.

**Goal:** Move cascade from an asserted "WCAG 2.1 AA" string to a proven WCAG 2.2 AA system
with per-component APG pattern conformance and an audited forced-colors / contrast /
reduced-motion story.

**Architecture:** `AccessibilityMeta` gains a versioned WCAG value + `apgPattern` +
media-feature flags (in `@cascade-ui/core`). Components are upgraded to satisfy the applicable
2.2 additions. Three new `scripts/checks` tests enforce: APG pattern conformance,
media-feature coverage, and target-size. Manifests are migrated last.

**Tech stack:** unchanged. CSS parsing reuses the v12/v14 CSS-classifier approach for the
media-feature audit.

**Files:**

- Modify: `packages/core/src/types.ts`, `packages/core/src/index.ts` (+ test)

- [ ] **Step 1:** Extend the type (additive; keep `'AA'`/`'AAA'` accepted during migration):

```ts
export type WcagLevel = '2.1-AA' | '2.2-AA' | '2.2-AAA' | 'AA' | 'AAA' // 'AA'/'AAA' deprecated, migrate away

export interface AccessibilityMeta {
  role: string
  wcag: WcagLevel
  keyboard: string[]
  apgPattern?: string // e.g. 'tabs', 'dialog-modal', 'combobox', 'accordion', 'slider'
  forcedColors?: boolean // explicitly handles forced-colors (Windows High Contrast)
  reducedMotion?: boolean // respects prefers-reduced-motion (null/false ⇒ no motion to guard)
}
```

- [ ] **Step 2: Test** a manifest with the new fields type-checks; legacy `'AA'` still
      compiles. Commit — `feat(core): versioned WCAG + apgPattern + media-feature flags`

- Create: `scripts/checks/apg/patterns.ts` (the APG requirement map)
- Create: `scripts/checks/apg.test.ts`

- [ ] **Step 1:** `patterns.ts` — a map of `apgPattern → { requiredRoles: string[],

      alertdialog, combobox, listbox, menu, menubutton, accordion, disclosure, slider,
      checkbox, radiogroup, switch, tooltip, breadcrumb, …). Source the requirements from the

- [ ] **Step 2:** Test: for every component manifest with an `apgPattern`, assert its

      pattern). Components without `apgPattern` are skipped (risk #9).
- [ ] **Step 3:** Run; for each failure, either fix the component/keyboard handling or correct
      the manifest's documented keys (only if the behavior is actually present — verify in the
      component source, don't paper over).
- [ ] **Step 4: Commit** — `feat(checks): APG pattern conformance check`

### Task 5: Media-feature audit (forced-colors / contrast / reduced-motion)

**Files:**

- Create: `scripts/checks/media-features.test.ts`
- Modify: component CSS to fill gaps; set `forcedColors`/`reducedMotion` flags in metas

- [ ] **Step 1:** Parse each component's CSS. Assert:
  - any component with an animation/transition has a `@media (prefers-reduced-motion: reduce)`
    guard (or is allowlisted as non-animating);
  - every interactive component has a `@media (forced-colors: active)` treatment that
    preserves affordances (borders/outlines using system colors) — or is allowlisted as
    inherently safe (uses system-respecting properties only);
  - components where color conveys state (alert, badge, status) handle `prefers-contrast: more`.
- [ ] **Step 2:** Fill the gaps: add the missing guards (the ~7 with reduced-motion are the
      head start; extend to all animating components; add forced-colors outlines to interactive
      ones). Set `forcedColors`/`reducedMotion` in each affected manifest **after** the CSS is
      true.
- [ ] **Step 3:** Check green. Commit — `feat(a11y): forced-colors + contrast + reduced-motion coverage`

## Tranche exit criteria

1. `AccessibilityMeta` carries a versioned WCAG value + optional `apgPattern` +
   forced-colors/reduced-motion flags; legacy `'AA'` is migrated away.
2. The APG conformance check passes for every component declaring a pattern and fails on a
   missing required role/key (prove by temporarily removing one).
3. Applicable WCAG 2.2 AA additions (target size, focus not obscured, dragging) are met or have
   a documented exception; `docs/specs/wcag-2.2.md` records the approach per criterion.
4. The media-feature audit is green: reduced-motion on all animating components, forced-colors
   on interactive ones, contrast handling where color conveys state.
5. Manifests show `'2.2-AA'` where earned, `'2.1-AA'` where honestly not yet; the landing a11y
   page reflects the versioned value + APG column.
6. Root gate + drift check exit 0.
