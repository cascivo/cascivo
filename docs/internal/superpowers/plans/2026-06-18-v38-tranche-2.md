# v38 — Tranche 2: Brutalist/Cyberpunk CSS Animations

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Answer the brief's question — _can the usual brutalist animations be added, ideally just with
CSS?_ — with **yes**, by giving `cyberpunk` two layers of motion, both pure CSS: (1) an **ambient**
"hard-snap" motion character that existing component transitions inherit with zero component changes, and
(2) a small set of **opt-in, theme-scoped signature effects** (scanline/CRT overlay, glitch/RGB-split on
hover, neon flicker). Everything is progressive enhancement with static fallbacks and is fully disabled
under `prefers-reduced-motion: reduce`.

**Architecture:** Components already consume `--cascivo-motion-{enter,exit,emphasis}` for their
transitions; overriding those custom properties inside `[data-theme='cyberpunk']` retunes every
component's motion **character** without touching component code. The louder effects are theme-scoped
`@keyframes` attached to **opt-in markers** (a marker class such as `.cascivo-cyber-scanline` or a
`data-cyber` attribute) so they never fire on arbitrary components unasked. No component TSX/CSS changes;
all new CSS lives in `cyberpunk.css`.

**Tech Stack:** CSS-only (`@keyframes`, `transition`, `prefers-reduced-motion`); `fallback:check` +
`breakpoint:check` guards.

### Task 1: Ambient motion character — "hard snap"

**Files:** `packages/themes/src/cyberpunk.css`

- [ ] **Step 1:** Inside `[data-theme='cyberpunk']`, override the semantic motion tokens to a stepped/snap
      easing so transitions read brutalist (mechanical, no soft ease):
      `--cascivo-motion-enter`, `--cascivo-motion-exit`, `--cascivo-motion-emphasis` → a short duration with a
      sharp `steps(...)` or hard `cubic-bezier` (e.g. `cubic-bezier(0.2, 0, 0, 1)` for a snappy land, or

- [ ] **Step 2:** Confirm this changes only the theme's own custom properties (no new keys, no component edits),
      so parity is unaffected and existing components (Modal/Dropdown/Toggle) inherit the snap automatically.
