**Goal:** Every cascade component publishes _why_ it exists and _when not_ to use it, not
just its shape. `ComponentMeta` gains an `intent` block; all 72 first-party items are
backfilled; a CI completeness check makes intent non-optional in practice; the factory
template emits intent for new components.

**Architecture:** The type lands in `@cascade-ui/core` (additive, optional). Manifests
populate it. The registry generator passes it through into each published item's `meta`. A
vitest completeness check enforces presence per category.

**Tech stack:** unchanged. Pure TypeScript + vitest.

**Files:**

- [ ] **Step 1: Add the types** to `types.ts`:

```ts
export interface IntentAntiPattern {
  bad: string // what people do wrong (code or prose)
  good?: string // the corrected form
  why: string // why the bad form is wrong
}

export type IntentRelationship = 'alternative' | 'pairs-with' | 'contains' | 'contained-by'

export interface IntentRelated {
  name: string // another component's name, e.g. 'Toast'
  relationship: IntentRelationship
  reason: string // why/when this relationship matters for selection
}

export interface IntentFlexibility {
  area: string // e.g. 'token names', 'spacing', 'copy', 'a11y role'
  level: 'strict' | 'flexible'
  note: string // what is fixed vs adjustable, and why
}

export interface IntentContent {
  tone: string // voice for user-facing copy this component renders
  notes?: string // content constraints (length, casing, i18n)
}

export interface ComponentIntent {
  whenToUse: string[] // bullet scenarios this is the right choice for
  whenNotToUse: string[] // scenarios where another component is better
  antiPatterns: IntentAntiPattern[]
  related: IntentRelated[] // selection edges to neighbouring components
  a11yRationale: string // WHY the a11y decisions are what they are
  content?: IntentContent
  flexibility: IntentFlexibility[] // strict-vs-flexible boundaries
}
```

- [ ] **Step 2: Extend `ComponentMeta`** with `intent?: ComponentIntent` (optional — additive,
      no break to existing manifests):

```ts
export interface ComponentMeta {

  tags: string[]

}
```

### Task 2: Intent completeness check

**Files:**

- Create: `packages/components/src/intent-completeness.test.ts`

- [ ] **Step 1: Write the check.** Import every `meta` (glob the `*.meta.ts` exports — match
      how existing component aggregate tests import manifests; if none, import the registry
      source list). For each:
  - Components in `inputs | display | overlay | navigation | feedback | layout`: require
    non-empty `intent.whenToUse`, `intent.whenNotToUse`, `intent.related`,
    `intent.a11yRationale` (non-empty string), and `intent.flexibility`.
  - Components in `chart | block` (composites): relaxed subset — require `intent.whenToUse`,
    `intent.whenNotToUse`, `intent.related` only.
  - `antiPatterns` and `content` are encouraged but not required for any category.

```ts
import { describe, expect, it } from 'vitest'
import { allMetas } from './_all-metas' // see Step 2

const FULL = ['inputs', 'display', 'overlay', 'navigation', 'feedback', 'layout']

describe('intent completeness', () => {
  for (const meta of allMetas) {
    it(`${meta.name} has required intent`, () => {
      expect(meta.intent, `${meta.name}: missing intent`).toBeDefined()
      const i = meta.intent!
      expect(i.whenToUse.length).toBeGreaterThan(0)
      expect(i.whenNotToUse.length).toBeGreaterThan(0)
      expect(i.related.length).toBeGreaterThan(0)
      if (FULL.includes(meta.category)) {
        expect(i.a11yRationale.trim().length).toBeGreaterThan(0)
        expect(i.flexibility.length).toBeGreaterThan(0)
      }
    })
  }
})
```

- [ ] **Step 2:** Create `_all-metas.ts` if no aggregate exists — re-export an array of all
      manifests (auto-generate it in the regen pipeline if the project prefers; otherwise a
      small hand-maintained barrel is acceptable and matches how the registry is built).
- [ ] **Step 3: Run** — `pnpm exec vp run @cascade-ui/components#test -t "intent completeness"`.
      Expected: FAIL for all 72 (no intent yet). This is the backfill worklist.
- [ ] **Step 4: Commit** — `test(components): intent completeness check (currently red)`

### Task 3: Backfill intent — inputs + overlay

**Files:**

- Modify: each `*.meta.ts` in `packages/components/src/{button,input,textarea,select,checkbox,radio,toggle,slider,...}` (inputs) and overlay (`modal/dialog`, `dropdown`, `tooltip`, `toast`, `alert-dialog`, ...)

- [ ] **Step 1:** For each component, author `intent` grounded in the component's **actual**
      behavior (read its tsx + existing `examples` first — do not invent). Example for Button:

```ts
intent: {
  whenToUse: [
    'Triggering an action or navigation the user initiates by click/press',
    'Submitting a form or confirming a decision',
  ],
  whenNotToUse: [
    'Navigation between pages where a real link is semantically correct — use an anchor',
    'Toggling a binary setting — use Toggle; persistent selection — use Checkbox/Radio',
  ],
  antiPatterns: [
    {
      bad: '<Button onClick={() => navigate("/x")}>Home</Button>',
      good: '<a href="/x">Home</a>',
      why: 'Buttons are for actions, links for navigation — assistive tech and the browser treat them differently',
    },
    {
      bad: 'background: #3b82f6',
      good: 'background: var(--cascade-button-bg)',
      why: 'Hard-coded colors break theming and the customization contract',
    },
  ],
  related: [
    { name: 'Toggle', relationship: 'alternative', reason: 'Use for binary on/off state, not one-shot actions' },
    { name: 'Dropdown', relationship: 'pairs-with', reason: 'A button often triggers a menu' },
  ],
  a11yRationale:
    'Renders a native <button> so Enter/Space activation, focus, and role come from the platform; loading state uses aria-busy rather than removing the element so focus is preserved',
  content: { tone: 'Imperative verb-first label', notes: 'Sentence case, ≤3 words; avoid "Click here"' },
  flexibility: [
    { area: 'token names', level: 'strict', note: 'Visual props must resolve to --cascade-button-* / semantic tokens' },
    { area: 'label copy', level: 'flexible', note: 'Free, within tone guidance' },
  ],
}
```

- [ ] **Step 2:** Repeat for each inputs + overlay component. Keep entries specific and
      true; cross-reference `related` names against real component names.
- [ ] **Step 3: Run** the completeness check filtered to these categories; expected: those
      items PASS.
