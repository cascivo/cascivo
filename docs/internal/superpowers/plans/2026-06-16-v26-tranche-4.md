### Task 2: Fix "Why cascivo" href

The `/why` route does not exist in the landing SPA. The landing's `App.tsx` has no `'/why'`
entry in `ROUTES`. The "Why cascivo" content lives in the docs app at `docs.cascivo.com/why`.

- [ ] **Step 1: Update the href in Footer.tsx**

  In `apps/landing/src/sections/Footer.tsx`, in the `COLUMNS` array, find:

  ```ts
  { label: 'Why cascivo', href: '/why' },
  ```

  Change to:

  ```ts
  { label: 'Why cascivo', href: 'https://docs.cascivo.com/why' },
  ```

  Because `isExternal` checks for `https?://`, this link will now automatically get
  `target="_blank" rel="noopener noreferrer"` in the rendered `<a>`.

### Task 5: Serve `registry.json` from the landing public directory

The root `registry.json` (at the repo root) must be available at `/registry.json` on the
landing domain. The file must be copied into `apps/landing/public/` during the build so Vite
includes it in the output.

- [ ] **Step 1: Check if registry.json is already in public**

  ```sh
  ls apps/landing/public/registry.json 2>/dev/null && echo "exists" || echo "missing"
  ```

  Expected: `missing`. If it already exists, skip to Step 4.

- [ ] **Step 2: Add a prebuild copy step**

  Open `apps/landing/package.json`. Add a `prebuild` script that copies the root `registry.json`
  to the public folder:

  ```json
  "scripts": {
    "prebuild": "cp ../../registry.json public/registry.json",
    "build": "vp build",
    …
  }
  ```

  On Windows, use `node -e "require('fs').copyFileSync('../../registry.json', 'public/registry.json')"`.
  For cross-platform compatibility, use a Node one-liner instead of `cp`:

  ```json
  "prebuild": "node -e \"require('fs').copyFileSync('../../registry.json', 'public/registry.json')\""
  ```

- [ ] **Step 3: Also copy for dev server**

  The `vp dev` server also needs the file. Add a `predev` script:

  ```json
  "predev": "node -e \"require('fs').copyFileSync('../../registry.json', 'public/registry.json')\""
  ```

  Or handle it in the Vite config (see Step 4 alternative).

- [ ] **Step 4: Alternative — Vite plugin approach**

  If the package.json script approach causes issues with `vp`, add a tiny inline plugin to
  `apps/landing/vite.config.ts`:

  ```ts
  import { copyFileSync } from 'node:fs'
  import { resolve } from 'node:path'

  // In the plugins array:
  {
    name: 'copy-registry',
    configResolved(config) {
      const src = resolve(config.root, '../../registry.json')
      const dest = resolve(config.publicDir, 'registry.json')
      try { copyFileSync(src, dest) } catch { /* not fatal in CI if root differs */ }
    },
  }
  ```

  Pick whichever approach works with the existing `vp` toolchain. Prefer the `package.json`
  `prebuild`/`predev` approach as it requires no Vite config changes.

- [ ] **Step 5: Verify the file is served**

  Build and preview: `pnpm exec vp run @cascivo/landing#build && pnpm exec vp preview --port 4180`.
  Navigate to `http://localhost:4180/registry.json`. Confirm the JSON is returned.

- [ ] **Step 6: Add `registry.json` to `.gitignore` inside `apps/landing/public/`**

  The file is generated at build time and should not be committed (it will be stale on every
  regen). Add to `apps/landing/.gitignore` (or root `.gitignore`):

  ```
  apps/landing/public/registry.json
  ```

  Confirm the root `registry.json` and `apps/landing/public/registry.json` are different
  entries (the root file IS committed; the public copy is not).
