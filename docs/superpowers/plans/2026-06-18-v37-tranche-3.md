# v37 — Tranche 3: Token Canonicalization + Manifest

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** End the "grep `node_modules` to find the tokens" experience (#8) and resolve the
canonical-vs-alias ambiguity (#7) by publishing a machine-readable token manifest, a `.d.ts` for editor
autocomplete, and a generated reference table that marks each role's canonical name and flags aliases.

**Architecture:** No `tokens.json`/token `.d.ts` exists today (`packages/tokens/src` has only
`functions.css`, `index.css`, `screens.ts`). A single generator parses the token + theme CSS and emits
three outputs from one source of truth (Decision 5): `tokens.json` (role + value + canonical/alias flag),
`tokens.d.ts` (string-literal union), and a docs table. **No token is removed** — aliases are kept and
explicitly labeled (Decision 3), standardizing canonical names on the ones the report adopted
(`--cascivo-color-foreground`, `--cascivo-color-text-muted`, `--cascivo-color-destructive`).

**Tech Stack:** Node generator under `pnpm regen`; drift-checked output.

## Files

- Create: `scripts/tokens/generate-manifest.mjs` — parse CSS → manifest + `.d.ts` + docs table
- Create: `packages/tokens/tokens.json` — generated, committed

### Task 1: Define the canonical/alias mapping (#7)

**Files:** generator config or an inline table in `generate-manifest.mjs`

- [ ] **Step 1:** Enumerate the duplicated roles from the light theme: page bg
      (`color-background` canonical / `color-bg` alias), body text (`color-foreground` / `color-text`),
      danger (`color-destructive` / `color-error`), on-accent (`color-accent-foreground` /
      `color-accent-content`), on-success (`color-success-foreground` / `color-success-content`), and the
      `-muted` / `-subtle` / `-text-muted` family. Record the canonical pick + alias list per role.
- [ ] **Step 2:** Standardize canonical names on the report's choices (`foreground`, `text-muted`,
      `destructive`). This is a **documentation/labeling** decision — no CSS values change in T3.

### Task 2: Write the generator (#8, Decision 5)

**Files:** `scripts/tokens/generate-manifest.mjs`

- [ ] **Step 1:** Parse `packages/tokens/src/*.css` and the theme files for every `--cascivo-*` custom
      property: capture name, resolved/declared value (per theme where it differs), category
      (color/space/radius/typography/motion/…), and the canonical/alias flag from Task 1.
- [ ] **Step 2:** Emit `packages/tokens/tokens.json` — an array/object of
      `{ name, category, role, value, canonical: boolean, aliasOf?: string }`.
- [ ] **Step 3:** Emit `packages/tokens/src/tokens.d.ts` — a `type CascivoToken = '--cascivo-…' | …`
      union (plus a `CascivoColorToken` subset if cheap) so consumers get autocomplete on token names.
- [ ] **Step 4:** Emit/refresh `docs/TOKENS.md` with a table grouped by category, canonical names bolded
      and aliases marked `(alias of …)`.
