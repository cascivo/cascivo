# v40 ŌĆö Tranche 4: Deep Conformance Validation

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make validation as strong as OpenUI's schema-validated, hallucination-resistant output. Today
`validateView` checks that a node's `component` exists and that `bind`/`events` use the right prefixes ŌĆö but
it does **not** check prop names, types, or enum values, so an LLM can emit a real component with invented
or wrong props and it renders silently. Extend the validator to check **props against the manifests**, and
make the `cvl` parser tolerant of partial/invalid input (structured errors, never throw) so streaming (T3)
and untrusted LLM output degrade gracefully.

**Architecture:** `packages/render/src/validate.ts` ŌĆö `validateView(config)` recurses nodes, checks
`component in componentMap` (Levenshtein "did you mean"), and validates `$data.`/`$actions.` ref prefixes.
The manifests carry per-prop `type`/`enum`/`default` (the same source T2 uses). Wire a meta-lookup
(component name ŌåÆ prop schema) so the validator can flag unknown props, type mismatches, and out-of-enum
values, reusing the existing Levenshtein helper for prop-name suggestions.

**Tech Stack:** Hand-written TypeScript checks (no Zod ŌĆö dependency policy); Vitest via vite+ (`vp`).
Builds on T1's parser; supports T3's streaming.
