# v47 — Tranche 2: Persistent Per-Line State Index + `tokenizeRange`

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the two pure, framework-free engine pieces that make windowed tokenization possible: (1) a mutable
`LineStateIndex` that memoizes the `GrammarState` *after* each line so the start-state of any window is a read, not
a re-walk from line 0; and (2) a `tokenizeRange(grammar, lines, from, to, index)` engine entry that tokenizes only
`[from, to)`, threading state via the index. Prove they are **byte-for-byte correct** against the existing
whole-document `tokenizeDocument` oracle with a property test over adversarial cross-line documents. This is the
riskiest part of v47 (cross-line state) proven in isolation before any view wiring (T3) or edit integration (T4).

**Architecture:** `engine/tokenize.ts` already has `tokenizeDocument(grammar, text): Token[][]` — it splits on
`\n`, threads `GrammarState` from `grammar.initialState` line-to-line (so fenced code / block comments carry
across lines), and reuses a bounded (`MAX_CACHE = 5000`) per-line memo `tokenize(grammar, line, startState)`. The
start state of line `i` is the **end state of line `i-1`**; line 0 starts at `grammar.initialState`;
`grammar.tokenizeLine(line, startState)` returns `{ tokens, state }`. The new `LineStateIndex` (`engine/line-state.ts`)
holds an array of per-line end-states plus the grammar reference, with `ensure(lines, upto)` (lazily compute/
memoize end-states up to `upto`), `startStateOf(lines, i)` (threaded start state for line `i`), and
`invalidateFrom(line)` (truncate known end-states at/after `line`, for edits in T4). `tokenizeRange` ensures states
up to `from` via the index, tokenizes `[from, to)` reusing the per-line `tokenize()`, updates the index's
end-states for those lines, and returns a plain `Token[][]` of length `to - from` — the consumer (T3 `renderRows`)
knows it starts at `from`, keeping the alignment trivial. `tokenizeDocument` is **left unchanged**: it stays the
public whole-document API *and* the oracle for the property test.

**Tech Stack:** owned TypeScript in `@cascivo/editor` (no new runtime deps); `GrammarState`/`Token`/`Grammar` from
`engine/types.ts`; the existing per-line `tokenize()` memo; the `markdown` and `typescript` built-in grammars
(register via `'../grammars/builtins.ts'`, look up via `getGrammar`); Vitest for unit + property tests. No React,
no signals, no banned hooks — these are pure data + methods.

### Task 1: Study `tokenizeDocument` and the state-threading contract

**Files:** `packages/editor/src/engine/tokenize.ts`, `packages/editor/src/engine/types.ts`, `packages/editor/src/grammars/builtins.ts`

- [ ] **Step 1:** Read `tokenizeDocument` — confirm it seeds `state = grammar.initialState`, calls
      `tokenize(grammar, line, state)` per line, pushes `result.tokens`, and threads `result.endState` forward.
      The start state of line `i` is the end state of line `i-1`; line 0 starts at `initialState`. This is the
      exact threading the index and `tokenizeRange` must reproduce.
- [ ] **Step 2:** Read `engine/types.ts` for `GrammarState` (opaque string), `Token`, and the `Grammar` interface
      (`initialState`, `tokenizeLine(line, state) → { tokens, state }`). Read the per-line `tokenize()` memo and
      its `TokenizeResult { tokens, endState }` shape — `tokenizeRange` reuses it for token reuse within the window.
- [ ] **Step 3:** Confirm `grammars/builtins.ts` registers `markdown` and `typescript` and that `getGrammar(name)`
      returns them — these are the property test's adversarial grammars (fences, block comments crossing lines).

### Task 2: Write the `LineStateIndex` (`line-state.ts`)

**Files:** `packages/editor/src/engine/line-state.ts`

- [ ] **Step 1:** Define `class LineStateIndex` (or a `createLineStateIndex(grammar)` factory) holding the grammar
      reference and `endStates: GrammarState[]` (index `i` = state *after* line `i`). Keep it mutable, framework-free,
      no React, no signals — just data + methods so it unit-tests without a DOM.
- [ ] **Step 2:** `ensure(lines, upto)` — from the last known-good line, compute and memoize each end-state up to
      `min(upto, lines.length - 1)` by feeding `startStateOf` of each line through `grammar.tokenizeLine` (or the
      memoized `tokenize()`), appending to `endStates`. Idempotent: re-calling with a covered `upto` is a no-op.
- [ ] **Step 3:** `startStateOf(lines, i)` — return `grammar.initialState` for `i === 0`, else `ensure(lines, i-1)`
      then return `endStates[i-1]`. This is the threaded start state any window begins from.
- [ ] **Step 4:** `invalidateFrom(line)` — truncate `endStates` to length `line` (drop the end-state of `line` and
      everything after), so the next `ensure`/`startStateOf` recomputes from the edit. Used on edits in T4.

### Task 3: Add `tokenizeRange` to `tokenize.ts`

**Files:** `packages/editor/src/engine/tokenize.ts`

- [ ] **Step 1:** Add `tokenizeRange(grammar, lines: string[], from, to, index): Token[][]`. Clamp `from`/`to` to
      `[0, lines.length]`. Call `index.startStateOf(lines, from)` to get the window's seed state (ensuring states
      up to `from` as a side effect).
- [ ] **Step 2:** Walk `[from, to)`: for each line call the existing memoized `tokenize(grammar, line, state)`,
      push `result.tokens`, thread `result.endState` into the next line **and** write it into the index's
      `endStates[i]` so subsequent windows reuse it. Return the `Token[][]` of length `to - from`.
- [ ] **Step 3:** Document the return alignment in a doc comment: a plain `Token[][]` whose element `j` is line
      `from + j`; the consumer (T3) knows the window starts at `from`. Keep `tokenizeDocument` byte-for-byte
      unchanged — it remains the public whole-document API and the property-test oracle.

### Task 4: Property test — `tokenizeRange` equals the `tokenizeDocument` oracle

**Files:** `packages/editor/src/engine/tokenize-range.test.ts`

- [ ] **Step 1:** Import `'../grammars/builtins.ts'` (registration side effect), `getGrammar`, `tokenizeDocument`,
      `tokenizeRange`, and the index. Assemble adversarial fixtures: Markdown with **unclosed** fences, **nested/
      adjacent** fenced blocks, and TypeScript with a **block comment crossing** a window boundary.
- [ ] **Step 2:** For each fixture and a sweep of arbitrary `[from, to)` (including `from > 0` starting mid-fence,
      `to` past EOF, and empty ranges), assert
      `tokenizeRange(grammar, lines, from, to, freshIndex)` **deep-equals**
      `tokenizeDocument(grammar, text).slice(from, to)`. Use a fresh index per case.
- [ ] **Step 3:** Reuse-correctness: call `tokenizeRange` over several overlapping windows against one shared
      index, then assert each still deep-equals the oracle slice — proving the index's stored end-states are
      consistent with full-document threading, not just first-touch.

### Task 5: Unit tests for the index incl. `invalidateFrom`

**Files:** `packages/editor/src/engine/line-state.test.ts`

- [ ] **Step 1:** `startStateOf(lines, 0)` returns `grammar.initialState`; `startStateOf(lines, i)` equals the
      end-state threaded by `tokenizeDocument` through line `i-1` (cross-check against an oracle walk).
- [ ] **Step 2:** `ensure(lines, upto)` is lazy and idempotent — `endStates` grows only to `upto`; re-calling with
      a smaller or equal `upto` does not recompute or change length.
- [ ] **Step 3:** `invalidateFrom(line)` truncates `endStates` to length `line`; a subsequent `startStateOf`/`ensure`
