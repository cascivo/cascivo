# v8 Master Plan — Assembly Included (Sections, Layout Primitives, Plain Charts)

> **For agentic workers:** This is the umbrella document. Implement tranche by tranche:
> `2026-06-11-v8-tranche-1.md` … `2026-06-11-v8-tranche-6.md`, in order. Each tranche uses
> superpowers:subagent-driven-development or superpowers:executing-plans.

**Goal:** Execute `docs/ROADMAP-V8.md` — ship ready-to-use composition: a `plain` mode for
every chrome-bearing chart in `@cascade-ui/charts` (micro charts everywhere), two new plain-CSS
layout primitives (`Masonry`, `AutoGrid`), a new `section/` registry family (hero, feature
grid, CTA, masonry gallery, stats band, footer — all static, copy-paste, token-only), and the
docs `/layouts` gallery that finally surfaces the whole layouts family.

**Architecture:** Sections and primitives live in `packages/layouts` (private, copy-paste
registry source — unchanged distribution model). Plain charts are a feature of the published
`@cascade-ui/charts` package. The registry generator's typed source roots gain one root
(`sections/`); CLI, MCP and llms generation are registry-driven and pick the new entries up
with minimal or zero changes (re-verify per tranche). Docs gain one gallery page and a nav
entry.

**Tech stack:** unchanged — React 18+ components, CSS Modules under `@layer cascade.component`
with `--cascivo-*` tokens and private `--_*` knobs, vitest + @testing-library/react, vp (vite+)
toolchain, Playwright in `apps/docs/test/`.

## Research findings (ground truth for all tranches)

### Current state (verified 2026-06-11)

**Registry** (`registry.json`, generated by `scripts/registry/generate.ts`): 97 entries =
60 `component` + 16 `chart` + 11 `layout` + 10 `block`. The generator scans typed
`SourceRoot`s (`dir`, `type`, `prefix`, `skipDirs`) — currently `packages/components/src`
(`''`), `packages/layouts/src` (`layout/`, skips `blocks`), `packages/layouts/src/blocks`
(`block/`), `packages/charts/src/charts` (`chart/`). Chart entries carry an `install` field
(npm package) instead of copyable files; `packages/cli/src/commands/add.ts` prints install
instructions for charts and strips type prefixes (`layout/app-shell` → `app-shell`) before
copying files for everything else.

**Layouts package** (`packages/layouts`, private, "copy-paste only, no build"): primitives
`stack`, `grid` (12-col span grid), `columns`, `center`, `spacer`, `split-view`; app layouts
`app-shell`, `auth-layout`, `dashboard-layout`, `settings-layout`, `page-header`; 10 blocks
(`console-app`, `dashboard-charts`, `login-page`, `stats-cards`, …). House style per
`grid/grid.tsx` + `grid.module.css`: `'use client'`, `cn` from `@cascade-ui/core`, props
extend `HTMLAttributes`, knobs passed as private custom properties
(`['--_grid-cols' as string]: String(cols)`), CSS under `@layer cascade.component` with
`container-type: inline-size` + `@container` breakpoints. Each entry ships `<name>.tsx`,
`<name>.module.css` (primitives), `<name>.meta.ts`, `<name>.test.tsx`. Blocks hardcode demo
copy via props with defaults (see `login-page.tsx`) — content scaffolding, not i18n surface.
`package.json#exports` lists only entries other packages import (`./app-shell`,
`./shell-state`, one block) — extend only as needed.

**Charts package** (`packages/charts`, published npm): 16 charts. Shared core:
`ChartFrame` (responsive SVG wrapper with `role="img"`, `aria-label`=title, `<desc>`, table
`fallback`, `className`, `data-state` passthrough), `use-chart.ts` (`useChartSize`,
`DEFAULT_MARGINS`), chrome `Axis`/`GridLines`/`Legend`. Cartesian charts (e.g.
`area-chart.tsx`) hardwire `GridLines` + two `Axis` renders inside the frame and default
`legend: series.length > 1`, `height: 300`. Already-plain charts: `sparkline` (120×32,
no frame, no chrome), `meter`, `kpi`. Series colors come from `--cascivo-chart-1..8`.

**Docs app** (`apps/docs`, Preact): nav built in `src/nav.ts` from the five component
categories (`inputs`/`display`/`overlay`/`navigation`/`feedback`) — `layout/*`, `block/*`
entries never appear; `Layout.tsx` hand-adds top links `AI / MCP`, `Charts`, `Playground`.
Routes in `App.tsx`: `/`, `/ai`, `/charts`, `/playground`, `/perf/data-table`,
`/components/:name`, default → ComponentPage. **Caveat:** registry names with slashes
(`layout/grid`) likely don't match `/components/:name` — the gallery must not depend on
per-entry detail routes (re-verify in T5). Demos in `src/demos.tsx` import component sources
directly via `@cascade-ui/components/<name>` subpath exports (Preact compat). Playwright
suite exists at `apps/docs/test/` (`visual.spec.ts`, `perf.spec.ts`), port 4173.

**Storybook** (`apps/storybook/stories/`): hand-written `*.stories.tsx` per component —
new entries need stories added the same way.

**Generated artifacts:** `pnpm registry:generate && pnpm readme:generate &&
pnpm llms:generate` regenerate registry.json/README/llms outputs from manifests; CI's drift
job fails on any uncommitted diff.

### External reality (verify at implementation time)

**CSS masonry (as of early 2026):** two competing syntaxes have existed — Firefox's
`grid-template-rows: masonry` (behind a flag since 2020) and the newer `display: masonry`
direction championed by WebKit/Chrome. Shipping status across the "last 2 versions of
Chrome/Firefox/Safari" target is in flux — **re-verify on MDN/caniuse before writing the
CSS**. Plan assumes: dual `@supports` blocks for both syntaxes + a universal CSS
multi-columns fallback (`column-count`/`column-gap`, `break-inside: avoid`). The fallback
orders items column-major (top-to-bottom per column) instead of row-major — acceptable for
galleries; documented in the manifest, not hidden.

**`auto-fill` + `minmax(min(X, 100%), 1fr)`:** baseline CSS, no support risk; `min()`
prevents overflow when the container is narrower than the minimum track.

**shadcn precedent:** shadcn distributes "blocks" (sections + pages) through the same
registry/CLI pathway as components, with a gallery page — the model v8 mirrors with cascade's
typed-registry twist. No new distribution machinery needed.

## Cross-cutting rules (every tranche)

1. **Tokens only.** All new CSS reads `--cascivo-*` custom properties; knobs are private
   `--_*` properties set from props. No raw colors, no px font sizes, no new fonts.
2. **Logical properties only.** `inline-size`, `margin-block-end`, `inset-inline-start`, …
   — grep for `left|right|top|bottom` physical properties before each commit (T6 formalizes).
3. **Container queries over media queries** for component-level responsiveness, mirroring
   `grid.module.css`.
4. **CLAUDE.md hook rules bind everything new**: no `useState`/`useContext`/`useEffect`/
   `useLayoutEffect`/`useReducer`. Sections additionally ban `useSignal`/`useMachine` —
   they are static (decision 3).
5. **Manifests are not optional.** Every new entry ships `<name>.meta.ts` with real props,
   tokens, a11y and examples — registry, MCP, llms and docs all derive from it.
6. **Gate before committing** (CLAUDE.md): `pnpm exec vp check` → `pnpm build` →
   `pnpm exec vp run -r check` → `pnpm test` → regenerate
   (`pnpm registry:generate && pnpm readme:generate && pnpm llms:generate`) →
   `pnpm exec vp check --fix` → `git diff --exit-code`.
7. **Surgical changes.** Don't refactor existing charts/layouts beyond what `plain`/sections
   require; match house style even where you'd deviate.
8. **Eyeball every visual change**: docs dev server, light AND dark, 1440 and 390 widths,
   plus `dir="rtl"` for sections.

## Edge cases / risks registry

1. **Masonry syntax flux**: `display: masonry` vs `grid-template-rows: masonry` — ship both
   `@supports` blocks; if a browser supports both, the later-specified block must win
   (order them deliberately) — verify cascade order in the module CSS. If at implementation
   time neither syntax is in any stable target browser, the columns fallback IS the
   implementation and the `@supports` blocks are progressive enhancement; that's fine.
2. **Columns fallback ordering**: column-major item order differs from native masonry's
   row-major fill. Document in the Masonry manifest description and gallery caption. Don't
   attempt JS reordering.
3. **`plain` on charts with interactive layers**: some charts may render hover/nearest-point
   layers (re-verify per chart against `engine/nearest.ts` usage). Plain removes visual
   chrome only — keep interaction layers; if an interaction layer depends on axis ticks,
   note it and disable just that piece with a manifest caveat.
4. **Stacked/multi-series plain charts lose the legend** — series remain distinguishable by
   `--cascivo-chart-N` colors only. Acceptable: plain charts are glanceable, not analytical.
   The docs micro-charts section shows a multi-series plain example so the tradeoff is
   visible.
5. **`height` default change in plain mode**: don't mutate `ChartFrame`'s default; resolve
   the default in each chart (`height = plain ? 48 : 300`) so non-plain behavior is
   byte-identical.
6. **Generator `skipDirs`**: forgetting to add `sections` to the layouts root produces
   duplicate entries (`layout/sections` garbage). The T3 registry diff must be reviewed
   entry-by-entry.
7. **CLI/MCP assumptions about types**: `add.ts` special-cases `chart` (npm install path) and
   strips prefixes generically — re-verify there's no exhaustive type switch that throws on
   `section` (also in `list.ts`, MCP `registry.ts`, llms/readme generators, docs `data.ts`
   `Category` type).
8. **Slash names vs docs routing**: `/components/:name` won't match `section/hero` — the
   gallery must render inline previews and NOT link to detail pages for prefixed entries
   (decision 10). Verify existing behavior didn't silently regress for `layout/*` either.
9. **Preact compat for section previews**: docs renders React components via preact/compat —
   sections are static so risk is low, but StatsBand pulls `@cascade-ui/charts` (signals) —
   the existing `/charts` docs page proves charts render under compat; mirror its import
   pattern.
10. **Charts bundle budget**: `plain` adds branches, not bytes of dependency — but verify
    the 80KB gz budget check still passes (it's part of the charts package CI).
11. **`exactOptionalPropertyTypes`**: conditional props (`plain`, `headingLevel`) must use
    `| undefined` or conditional spread per house style — copy the patterns in
    `chart-frame.tsx` (`'data-state'?: string | undefined`).
12. **Section demo defaults vs i18n rule 6**: CLAUDE.md's builtin-catalog rule targets
    component chrome strings; sections are content scaffolding (decision 3). Record this in
    each section meta description ("replace demo content") so the factory doesn't "fix" it.
