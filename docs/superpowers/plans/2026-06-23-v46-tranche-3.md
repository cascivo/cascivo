# v46 — Tranche 3: In-Document Find & Replace

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add **find & replace** — the third 🔴 blocker (gap-analysis **#2**). A transparent textarea cannot use
the browser's `Ctrl-F` against its visible text, and long notes need in-document search. Build a small,
signal-driven **find panel** (find field, match count, next/prev, replace, replace-all, `Esc` to close) opened by
`Mod-F`, with **matches highlighted in the overlay layer** via a decoration mechanism in `renderRows`, and
replacements routed through the T2 history-aware edit path (undoable). This tranche also **introduces the
decoration seam** that T4 formalizes as public and T5 reuses for brackets.

**Architecture:** Search is a pure scan: given the document text + query (+ case/word flags), produce a list of
`{ start, end }` offset matches; the "current" match is an index signal. Matches become **decorations** — the
view layer maps offset ranges to a CSS class and renders them as extra spans. Extend `renderRows` to accept a
per-line decoration list (default empty → unchanged for `Highlight`): for each line, split token spans at
decoration boundaries and tag the covered slice with the decoration class. Next/prev moves the current index
(wrapping), scrolls the match into view, and sets the textarea selection to it. The panel is a signal-driven
React subcomponent; its open state, query, replacement, flags, matches, and current index are signals on the
editor. `Mod-F` (find) / `Mod-Alt-F` or `Mod-H` (replace) come from the T1 keymap; `Esc` closes and returns
focus to the textarea.

**Tech Stack:** owned TypeScript scan; `@cascivo/core` signals + `cn`; `@cascivo/i18n` `builtin.editor` strings;
CSS module for the panel; Vitest + Testing Library. No `useState`/`useEffect`; `useSignalEffect` for the
scroll-into-view side effect.

### Task 1: Decoration support in the view layer

**Files:** `packages/editor/src/editor/view.tsx`, `packages/editor/src/editor/highlight/highlight.module.css`

- [ ] **Step 1:** Define `type Decoration = { line: number; start: number; end: number; className: string }` (offsets are **column** within the line). Change `renderRows(lines, start, end, decorations?)` — default `undefined` keeps `Highlight` byte-identical.
- [ ] **Step 2:** For each rendered line, if it has decorations, walk its tokens accumulating a column cursor and **split** any token span overlapping a decoration range, wrapping the covered slice in a `<span>` that adds the decoration class (compose with the token-kind class via `cn`). Keep it allocation-light; lines without decorations take the existing fast path.
- [ ] **Step 3:** Add a `.match` and `.matchCurrent` class in the highlight CSS module (background tint via an editor token, e.g. `--cascivo-editor-match` / `--cascivo-editor-match-current`; add those tokens with sensible fallbacks). `aria-hidden` already covers the layer.

### Task 2: The search scan (`find.ts`)

**Files:** `packages/editor/src/editor/code-editor/find.ts`

- [ ] **Step 1:** `scan(text, query, opts: { caseSensitive?: boolean; wholeWord?: boolean }): { start: number; end: number }[]` — literal substring scan (no regex by default; keep it simple and safe). Empty query → no matches. Deterministic, O(n·m) acceptable for notes; can switch to `indexOf` loop.
- [ ] **Step 2:** `toDecorations(matches, text, currentIndex): Decoration[]` — convert absolute `{start,end}` offsets to per-line `{line,start,end,className}` (split matches that cross a newline; in practice matches are single-line for literal queries without `\n`). Tag the current match with `.matchCurrent`.
- [ ] **Step 3:** `replaceAt(text, match, replacement)` and `replaceAll(text, matches, replacement)` returning the new text + the edit range(s) so the component can route them through `applyEdit` (undoable). Replace-all applies right-to-left so offsets stay valid.

### Task 3: The find panel (`find-panel.tsx`)

**Files:** `packages/editor/src/editor/code-editor/find-panel.tsx`, `find-panel.module.css`

- [ ] **Step 1:** Props: the find/replace query signals, match count + current index, and callbacks (`onNext`, `onPrev`, `onReplace`, `onReplaceAll`, `onClose`, `onToggleCase`, `onToggleWord`, `onToggleReplace`). `useSignals()` first. No internal state beyond the passed signals.
- [ ] **Step 2:** Render a labelled region (`role="search"` or a labelled `<div>`), the find input (autofocus on open), a "N of M" count from i18n (`t(builtin.editor.findCount, { current, total })`), prev/next buttons, a replace row (find input + replace input + replace/replace-all), and a close button. All labels from `builtin.editor`. `Esc` closes.
- [ ] **Step 3:** CSS module: a compact bar anchored top-end inside the editor (`position: absolute`), token-themed, ≥44px coarse tap targets on the buttons (`min-block-size: var(--cascivo-target-min-coarse, 2.75rem)` under `@media (pointer: coarse)`), reduced-motion safe, no off-scale breakpoint literals, never `display:none` that loses a control.

### Task 4: Wire find into the editor

**Files:** `packages/editor/src/editor/code-editor/code-editor.tsx`

- [ ] **Step 1:** Add signals: `findOpen`, `findQuery`, `replaceQuery`, `caseSensitive`, `wholeWord`, `replaceMode`, `currentMatch`. Add keymap entries: `Mod-f` → open find + focus its input; `Mod-Alt-f` (and/or `Mod-h`) → open in replace mode; `Esc` (when open) → close + refocus textarea.
- [ ] **Step 2:** Compute `matches = scan(highlightText.value, findQuery.value, {…})` and feed `toDecorations(...)` into `renderRows` (merge with any other decorations). Clamp `currentMatch` when matches change.
- [ ] **Step 3:** `onNext`/`onPrev` move `currentMatch` (wrap), set the textarea selection to the match, and in a `useSignalEffect` scroll it into view. `onReplace`/`onReplaceAll` call `find.replace*` and apply via `applyEdit` (undoable), then re-scan.
- [ ] **Step 4:** Add `openFind()` to the T2 `CodeEditorHandle`. Render `<FindPanel … />` inside the editor root when `findOpen.value`.
