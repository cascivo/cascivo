# v41 — Tranche 2: `user` + `avatar-group` Components

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship the two identity-display pieces HeroUI has that cascivo lacks, both **reusing the existing
`avatar`**: **`user`** (avatar + name + description + optional trailing action slot — HeroUI's `User`) and
**`avatar-group`** (overlapping stack of `avatar`s with a `max` cap and a `+N` overflow chip — HeroUI's
`AvatarGroup`). Pure-display components: no machine (no internal transitions). Each ships the full kit, is
exported from `@cascivo/react`, added to `_all-metas.ts`, and picked up by `pnpm regen`.

**Architecture:** Both compose `packages/components/src/avatar/avatar.tsx` rather than duplicating its load
FSM. `avatar-group` follows the `button`/`button-group` and `radio`/`radio-card` precedent (its own
directory wrapping the primitive). Per CLAUDE.md: no banned hooks; pure display means no machine and no
signals unless something is genuinely driven internally; CSS handles layout/overlap; strings default from
`@cascivo/i18n`.

**Tech Stack:** signal-driven-where-needed TSX + CSS Modules; `@cascivo/core` (`cn`); `@cascivo/components`
`avatar`; `@cascivo/i18n` for the overflow label; Vitest + Testing Library.

### Task 2: Implement `user.tsx` (avatar + text composite)

**Files:** `packages/components/src/user/user.tsx`, `user.module.css`

- [ ] **Step 1:** Props extending `HTMLAttributes<HTMLDivElement>`: `name: ReactNode` (required),
      `description?: ReactNode`, `avatarProps?: AvatarProps` (forwarded to `<Avatar>`), optional `children` as a
      trailing action slot, `className`/`...rest`. Render `<div>` → `<Avatar {...avatarProps} />` + a text column
      (`name` + `description`) + optional action slot. Pure display — **no machine, no signals**.
- [ ] **Step 2:** `user.module.css`: `@layer cascade.component`; flex row, gap via `--cascivo-space-*`; name uses
      a text/label token, description a muted/secondary color token; logical properties; truncation with
      `text-overflow: ellipsis`; ≥44px target only if the action slot is interactive.
- [ ] **Step 3:** A11y: the composite is a labelled group; `name` is the accessible label for the avatar context
      (avatar already handles its own `role="img"`/`aria-label`). No redundant roles.

### Task 3: Implement `avatar-group.tsx` (overlap stack + overflow)

**Files:** `packages/components/src/avatar-group/avatar-group.tsx`, `avatar-group.module.css`

- [ ] **Step 1:** Props extending `HTMLAttributes<HTMLDivElement>`: `max?: number` (cap visible avatars),
      `total?: number` (override count for the `+N` chip), `spacing?: 'sm'|'md'|'lg'` (overlap amount),
      `isGrid?: boolean` (HeroUI parity — wrap into a grid), `children` (the `<Avatar>`s), `className`/`...rest`.
- [ ] **Step 2:** Render up to `max` children; compute overflow `= (total ?? childCount) - max`; when `> 0`
      render a final overflow avatar/chip showing `+N` with an i18n-defaulted `aria-label`
      (e.g. `'{count} more'`). Overlap via CSS negative `margin-inline-start` on all but the first (logical
      property, RTL-safe), driven by `data-spacing`.
- [ ] **Step 3:** `avatar-group.module.css`: `@layer cascade.component`; overlap via logical negative margin +
      `z-index` stacking so earlier avatars sit on top (or reverse, per HeroUI); a subtle ring/border using a
      token to separate overlapping avatars; `isGrid` switches to a wrapping grid (no `display:none`).
- [ ] **Step 4:** No machine, no banned hooks. If you read a derived count as a signal for any reason, call
      `useSignals()` — but a plain computed during render is fine here.
