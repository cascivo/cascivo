# v40 — Tranche 1 (core): Library-Derived Prompt + Grammar Generation

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** The single biggest value of the OpenUI study — make agent-generated UI **reliable** by bounding
the model's vocabulary. Generate, from the `component.meta.ts` manifests, the **system prompt** + a
machine-checkable **allowed-vocabulary grammar** (components → props → enums/sizes/variants) an LLM needs to
emit valid **`ViewConfig` JSON** restricted to cascivo's real components. This is OpenUI's "generate the
system prompt from the component library" anti-hallucination mechanism, applied to the format cascivo
already ships and dogfoods (landing `AgentLayer`/`JsonPlayground`, docs playground, CLI `generate`, example
apps). It upgrades today's keyword-matching `scaffoldView`.

**Architecture:** Manifests carry per-prop `type`/`enum`/`default` + `variants`/`sizes`.
`packages/mcp/src/scaffold-view.ts` currently picks components by keyword and fills required/default props —
it does **not** describe the schema or bound the vocabulary. The generator reads the registry/manifests and
emits (a) a system prompt documenting the `ViewConfig` JSON shape (`view.regions` → `ComponentNode`s with
`props`/`bind: $data.*`/`events: $actions.*`/`children`) and (b) a compact allowed-vocabulary table. Both
stay in sync with the components because they are **derived**, not authored. No new wire format — this works
on JSON `ViewConfig`.

**Tech Stack:** TypeScript generator in `@cascivo/mcp`; Vitest via vite+ (`vp`). No deps.

### Task 1: Derive the allowed-vocabulary grammar from manifests

**Files:** `packages/mcp/src/registry.ts` (read), `packages/mcp/src/grammar.ts`

- [ ] **Step 1:** Read the registry/manifest surface (`registry.ts`, the `component.meta.ts` shape) and build
      a descriptor: per component → its props (name, type, enum values, default, required), plus `variants`/
      `sizes`. This is the bound vocabulary.
- [ ] **Step 2:** Emit it in a compact, model-friendly form (terse, not verbose JSON): one line per component
      listing allowed props and enum choices. Keep it deterministic (sorted) for testability.

### Task 2: Generate the system prompt

**Files:** `packages/mcp/src/prompt.ts`, `packages/render/schema/view.v1.json` (read)

- [ ] **Step 1:** Compose a system prompt that (a) documents the **`ViewConfig` JSON** shape (top-level
      `view.layout?` + `view.regions`; each node `{ component, props?, bind?, events?, children? }`;
      `bind` values are `$data.*`, `events` values are `$actions.*`), (b) embeds the allowed-vocabulary
      descriptor from Task 1, and (c) instructs the model to output **only** valid `ViewConfig` JSON using
      registered components/props. Keep it concise.
- [ ] **Step 2:** Make the prompt parameterizable by an optional component subset (scope it to, e.g., the 6
      components relevant to a task), defaulting to the full registry.
- [ ] **Step 3:** Leave a clearly-marked extension point for an optional `cvl` section (added by T3 if built) —
      a comment, not code, so T1 stands alone.

### Task 3: Boundedness test

**Files:** `packages/mcp/src/grammar.test.ts`

- [ ] **Step 1:** Assert every component name in the generated grammar exists in the registry, and every
      prop/enum it lists matches that component's manifest (no invented props/values).
- [ ] **Step 2:** Snapshot a small generated prompt for a 2–3 component subset to lock the format (update
      deliberately).

### Task 4: Expose via MCP

**Files:** `packages/mcp/src/server.ts`, `packages/mcp/src/scaffold-view.ts`

- [ ] **Step 1:** Surface the prompt/grammar to agents — a new tool (e.g. `get_view_grammar` /
      `get_generation_prompt`) or an extension of the existing `scaffold_*` tool returning the prompt + grammar
      context. Follow the tool-registration pattern in `server.ts`.
- [ ] **Step 2:** Optionally have `scaffoldView` also return the grammar context so a caller gets both a
      starter scaffold and the bounded vocabulary in one call.
