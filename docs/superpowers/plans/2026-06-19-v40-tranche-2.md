# v40 — Tranche 2: Library-Derived Prompt + Grammar Generation

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Teach an LLM to emit valid `cvl` (T1) bounded to cascivo's **real** components, props, enums,
sizes, and variants — the OpenUI mechanism of "generate the system prompt from the component library" so
the model has a *bound vocabulary* and cannot hallucinate components or props. Generate this prompt + a
machine-checkable grammar from the `component.meta.ts` manifests, upgrading today's keyword-matching
`scaffoldView`.

**Architecture:** Manifests carry per-prop `type`/`enum`/`default`, plus `variants`/`sizes` (CLAUDE.md
manifest schema). `packages/mcp/src/scaffold-view.ts` currently picks components by keyword and fills
required/default props — it does **not** describe the format or bound the vocabulary. The generator reads
the registry/manifests and emits (a) a system prompt that documents the `cvl` grammar (from
`packages/render/src/lang/grammar.ts`, T1) and (b) an allowed-vocabulary table. Both stay in sync with the
components because they are derived, not authored.

**Tech Stack:** TypeScript generator in `@cascivo/mcp`; Vitest via vite+ (`vp`). Depends on T1's `cvl`
grammar.

### Task 1: Derive the allowed-vocabulary grammar from manifests

**Files:** `packages/mcp/src/registry.ts` (read), `packages/mcp/src/grammar.ts`

- [ ] **Step 1:** Read the registry/manifest surface (`registry.ts`, `component.meta.ts` shape) and build a
      descriptor: per component → its props (name, type, enum values, default, required), plus `variants`/
      `sizes`. This is the bound vocabulary.
- [ ] **Step 2:** Emit it in a compact, model-friendly form (mirroring `cvl`'s terseness, not verbose JSON):
      one line per component listing allowed props and enum choices. Keep it deterministic for testability.

### Task 2: Generate the system prompt

**Files:** `packages/mcp/src/prompt.ts`

- [ ] **Step 1:** Compose a system prompt that (a) explains the `cvl` syntax by referencing the grammar
      contract from `packages/render/src/lang/grammar.ts` (Task 1 of T1), (b) embeds the allowed-vocabulary
      descriptor, and (c) instructs the model to output **only** `cvl` using registered components/props, with
      `bind`/`events` for data/actions. Keep it concise — the point is token efficiency end-to-end.
- [ ] **Step 2:** Make the prompt parameterizable by an optional component subset (so a caller can scope it
      to, e.g., the 6 components relevant to a task) — defaulting to the full registry.

### Task 3: Boundedness test

**Files:** `packages/mcp/src/grammar.test.ts`

- [ ] **Step 1:** Assert every component name in the generated grammar exists in the registry, and every prop/
      enum it lists matches that component's manifest (no invented props/values).
- [ ] **Step 2:** Snapshot a small generated prompt for a 2–3 component subset to lock the format (update
      deliberately).

### Task 4: Expose via MCP

**Files:** `packages/mcp/src/server.ts`, `packages/mcp/src/scaffold-view.ts`

- [ ] **Step 1:** Surface the prompt/grammar to agents: either a new tool (e.g. `get_view_grammar` /
      `get_generation_prompt`) or extend the existing `scaffold_*` tool to optionally return the prompt +
      grammar context. Follow the existing tool-registration pattern in `server.ts`.
- [ ] **Step 2:** Optionally have `scaffoldView` also return the scaffolded view rendered as `cvl` (via T1's
      `encode`) so the scaffold and the generation format match.
