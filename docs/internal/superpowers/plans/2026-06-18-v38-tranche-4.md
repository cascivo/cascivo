# v38 — Tranche 4: Landing-Page Integration (every surface)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Honor the explicit ask — _"make sure it is considered everywhere in the landing page"_ — by
wiring `cyberpunk` into **every** landing surface that enumerates themes. The landing page has **five**
distinct theme touchpoints (and two of them are independent hard-coded arrays), so this gets its own
tranche and ends with a grep sweep that proves none was missed.

**Architecture:** The landing app does not read the themes package's list; each surface duplicates it:
1. `apps/landing/src/theme.ts` — the persisted theme switcher's source of truth (`THEMES`).
2. `apps/landing/src/sections/ComponentField.tsx` — a **separate, local** `THEMES` array driving the live
   "theme-tile wall" (`bg-field-tile` elements each set `data-theme`).
3. `apps/landing/src/sections/Features.tsx` — the "Ten themes" feature card (count + enumerated list copy).
4. `apps/landing/src/landing.css` — per-theme `@import` of the theme CSS.
5. `apps/landing/src/pages/create/presets.ts` — the `/create` preset gallery (swatch + config per theme).

**Tech Stack:** TSX/TS + CSS edits; `vp` build/check; the landing app reads `signal.value` so any new
reactive read must keep `useSignals()` discipline (no new components needed here — only list edits).
