# v57 — Tranche 1: Registry Dependency-Graph Resolution (the blocker)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Remove the report's one hard blocker — `cascivo add shell-header`/`side-nav` install a component that
imports `../popover/use-popover`, but the installer never ships that hook. Fix the **three** broken links in the
declare→generate→resolve chain so a bare-name install pulls its internal (registry) dependencies, and the five
affected components build standalone (M-1, Master Decisions 1/2).

**Architecture:** four small edits across the registry pipeline + CLI; the v2 schema field already exists.
- **`ComponentMeta.registryDependencies?: string[]`** (`packages/core/src/types.ts`) — lets a manifest declare "I need
  registry component X." The `RegistryItem` schema already has the field (`packages/registry/src/types.ts:85`); this
  adds the *authoring* surface.
- **Generator** (`scripts/registry/generate.ts`): (a) `isSourceFile` includes non-test, non-`.meta` `.ts` files so
  shared hooks/utils (`use-popover.ts`) land in `files[]`; (b) `buildEntry` copies `meta.registryDependencies` onto
  the emitted entry.
- **Manifests**: the five importers declare `registryDependencies: ['popover']`. (Installing `popover` brings
  `popover.tsx` + `popover.module.css` + — after the `.ts` fix — `use-popover.ts`, so the relative import resolves.)
- **CLI bare-name path** (`packages/cli/src/commands/add.ts:102–127`): walk `registryDependencies` transitively
  (de-duped), mirroring the multi-registry path (`add.ts:69` + `resolve.ts` queue) which already does this.

**Tech Stack:** `packages/core` (types), `scripts/registry/generate.ts`, `packages/components/**/*.meta.ts`,
`packages/cli/src/commands/add.ts` (+ `resolve.ts`), `registry.json`. Pure TypeScript. No new dependency.
