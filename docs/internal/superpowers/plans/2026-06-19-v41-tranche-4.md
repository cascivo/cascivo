# v41 — Tranche 4: Interaction Polish + State Tokens

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Consume T3's hooks and adopt HeroUI's remaining UX/theming polish — all **opt-in, additive**, no
behavior change to existing call sites. Four threads:

1. **Draggable overlays** — opt-in `draggable` Modal (drag by header) and drag-to-dismiss Drawer, via
   `useDraggable` applied as a CSS `translate` (no Framer Motion).
2. **ScrollShadow fade** — an opt-in `mask`/fade variant on `scroll-area` (HeroUI's `ScrollShadow`), with the
   existing box-shadow edges as the static fallback.
3. **Infinite scroll** — a documented integration path wiring `useInfiniteScroll` into `data-table` (or
   `select`) via a sentinel row.
4. **Interaction-state tokens** — the one genuine theming gap: add `--cascivo-disabled-opacity` and
   `--cascivo-hover-opacity` to `packages/tokens` and **all 12** theme files (parity), and route ≥1
   component's state opacity through the token to prove the mechanism. (cascivo's `radius` one-knob
   `--cascivo-radius-base` already covers HeroUI's `radius` layout token — not re-done.)

**Architecture:** Reuse T3's hooks; `scroll-area` already renders box-shadow edges from `data-*` attributes —
the mask is a CSS-only addition gated by a prop. Token parity is a hard gate. Per CLAUDE.md: no banned hooks;
CSS for visual states; progressive enhancement with static fallbacks; reduced-motion-safe; no off-scale
breakpoint literals.

**Tech Stack:** signal-driven TSX (consuming `@cascivo/core` hooks) + CSS Modules; CSS custom-property token
rows; vite+ (`vp`) check/build/test; `parity.test.ts`, `breakpoint:check`, `fallback:check`.

### Task 2: ScrollShadow fade variant on `scroll-area`

- [ ] **Step 2:** `scroll-area.module.css`: gate the `mask-image` declarations on the `'mask'` variant; precede

- [ ] **Step 3:** No behavior change for existing callers (default stays `'shadow'`). Add a test asserting the

### Task 3: Draggable Modal (opt-in)

**Files:** `packages/components/src/modal/modal.tsx`, `modal.module.css`

- [ ] **Step 1:** Add an opt-in `draggable?: boolean` prop (default `false` — no change). When true, call
      `useDraggable({ ... })`, attach `handleRef` to the modal header and `targetRef` to the dialog, and apply
      `offset` as a CSS `translate` custom property (e.g. `--modal-x`/`--modal-y`) on the dialog. `useSignals()` as
      needed. No `useEffect` (the hook owns the pointer listeners via `useSignalEffect`).
- [ ] **Step 2:** `modal.module.css`: apply `translate: var(--modal-x,0) var(--modal-y,0)`; any snap/settle
      transition is reduced-motion-safe with a static fallback. The header shows a grab cursor only when
      `draggable`. Keep focus-scope/dismiss behavior intact.
- [ ] **Step 3:** Test: with `draggable`, a synthetic pointer drag on the header updates the dialog's translate
      custom property; without it, no drag listeners attach and behavior is unchanged.

### Task 4: Drag-to-dismiss Drawer (opt-in)

**Files:** `packages/components/src/drawer/drawer.tsx`, `drawer.module.css`

- [ ] **Step 1:** Add an opt-in `swipeToDismiss?: boolean` (default `false`). When true, use `useDraggable`
      constrained to the drawer's edge axis; past a threshold on `onDragEnd`, trigger the existing close path
      (call the same `onOpenChange`/close used today — do not add a new state machine). Below threshold, `reset()`

- [ ] **Step 2:** `drawer.module.css`: apply the drag offset as a translate; snap-back transition reduced-motion-safe
      with a static fallback; do not let drag interfere with inner scroll (exclude the scroll body from the handle,
      per HeroUI's approach).
- [ ] **Step 3:** Test: a synthetic drag past threshold calls the close handler; a short drag snaps back (offset
      resets); default (`swipeToDismiss={false}`) attaches nothing.
