Produce a baseline JSON report from a pre-T1 build (use `git stash` or a comparison branch if one exists) and compare it against the post-T5 build. **Tech Stack:** `@lhci/cli`, `pnpm exec vp run @cascivo/landing#preview`, `lhci collect`, `lhci assert`.

### Known limitation: `react-dom/profiling`

The `vite.config.ts` alias maps `react-dom/client` → `react-dom/profiling` unconditionally.
This is intentional — the `SignalsDemo` uses `<Profiler onRender>` and needs profiling hooks
to fire in production builds (the standard prod build compiles away `onRender`). The profiling
build adds marginal overhead to every React commit across the entire app, not just the demo.

If Performance reaches 90–94 but not 95, consider extracting the `<Profiler>` usage into
its own mechanism that does not require the whole app to run on `react-dom/profiling`. Options:

1. Keep SignalsDemo's profiling but use a custom render counter that does not rely on
   `onRender` (e.g., a signal incremented in a `useSignalEffect`).
2. Restrict the profiling alias to the SignalsDemo chunk only via a Rolldown chunk-level
   plugin (advanced, not recommended unless performance is the blocker).

This optimization is **not in scope for v30**. Surface it as a v31 candidate only if the
Performance score is 90–94 after all five tranches.
