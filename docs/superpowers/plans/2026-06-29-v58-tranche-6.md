### Task 2: `THEMING.md` — the three corrections

**Files:** `docs/THEMING.md`

- [ ] **Step 1: "Why `[data-theme]`, not `light-dark()`" (C-1).** Explain: `light-dark()` resolves a **two-way**
      light/dark pair off computed `color-scheme`; cascivo ships **12** themes (`light, dark, warm, flat, minimal,
      midnight, pastel, brutalist, corporate, terminal, cyberpunk, arcade`) scoped to **any container** via the
      `[data-theme]` attribute. `light-dark()` cannot express `warm` vs `brutalist` and keying on `color-scheme`
      collapses container-scoped multi-theming to a binary — so cascivo keeps `[data-theme]`. Note that `light-dark()`
      *could* be used narrowly inside the light/dark pair, but it buys little over the attribute system and v58 does
      not adopt it.
- [ ] **Step 2: "`@container style()` palette branching — deferred" (C-2).** Describe the technique (a component's
      text palette branches by the surface it sits on, via a registered `--contrast-color` + `@container style()`),
      note it's powerful micro-theming, and record it as a **future-roadmap candidate** — orthogonal to v58's
      "derive + disambiguate" core, and gated on broader `@container style()` support. The `@property` registrations
      from T2 are the substrate a future implementation would build on.
- [ ] **Step 3: "OKLCH is the floor" (C-3).** State plainly that every cascivo color is already OKLCH (primitives +
      semantics), so the common "use OKLCH for machine-derivable color" advice is **already satisfied** — v58 added
      the *derivation* layer (relative color, `contrast-color()`) on top, it did not "adopt OKLCH."
