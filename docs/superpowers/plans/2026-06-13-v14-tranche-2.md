> Current-state assumptions — RE-VERIFY before starting:
>
> - 17 chart dirs under `packages/charts/src/charts/*`; shared frame at
>   `packages/charts/src/core/chart-frame.tsx` (no tooltip support today).
> - Only `line`, `heatmap`, `histogram` declare a `tooltip` prop; `line-chart.tsx` mutates a
>   `tooltipRef` div on pointer move (mouse-only, no keyboard, no aria) — this is the pattern
>   to replace.
> - cascade has a `Tooltip` _component_ (`packages/components/src/tooltip`) — evaluate whether
>   to reuse it or build a chart-specific overlay (charts need datum-anchored positioning).
> - House rules: signals only, no `useEffect`; `useSignalEffect` for DOM side effects;
>   `useSignals()` in React apps that read signals.

**Goal:** Every chart reveals its underlying data on interaction, to mouse **and** keyboard,
with a screen-reader announcement — via a single tooltip primitive, not 17 bespoke ones.

**Architecture:** A `chart-tooltip` primitive + a `data-point provider` adapter pattern: each
chart supplies its focusable data points (`{ id, x, y, label, value, seriesId }[]` with pixel
positions). The shared `chart-frame` renders a focusable point layer + the tooltip overlay,
handles pointer hit-detection (nearest datum) and keyboard traversal (arrow keys move the
focused datum index), and exposes the focused datum to an `aria-live` region. Charts opt in by
providing points; single-value charts (meter/kpi) opt out with a documented reason.

**Tech stack:** unchanged. Positioning math is pure (testable); geometry-dependent visuals
verified via Playwright/screenshot (jsdom can't lay out — risk #4).

### Task 1: Data-point provider type + frame plumbing

**Files:**

- Create: `packages/charts/src/core/data-point.ts` (types + helpers)
- Modify: `packages/charts/src/core/chart-frame.tsx` (+ test)

- [ ] **Step 1:** Define the contract:

```ts
export interface ChartPoint {
  id: string
  cx: number // pixel x within the plot area
  cy: number // pixel y
  label: string // category / x label
  value: number | string // y / measure
  seriesId?: string
}
export interface TooltipModel {
  points: ChartPoint[] // all focusable points, in traversal order
  format?: (p: ChartPoint) => string // overrides default "label: value"
}
```

- [ ] **Step 2:** `chart-frame` accepts an optional `tooltip?: TooltipModel`. When present it
      renders: a transparent focusable layer (`tabindex=0`, `role="application"` or
      `img`+description — decide and document), an absolutely-positioned tooltip overlay, and a
      visually-hidden `aria-live="polite"` region. State: a `useSignal<number | null>` focused
      index.
- [ ] **Step 3: Test** the frame renders the point count it's given and the aria-live region
      exists; no tooltip prop → no tooltip layer (backward compatible).
- [ ] **Step 4: Commit** — `feat(charts): chart-frame data-point + tooltip plumbing`

## Tranche exit criteria

1. A single `chart-tooltip` primitive in `@cascivo/charts` handles pointer hit-detection,
   keyboard data-point traversal (Arrow/Home/End/Escape), and an `aria-live` datum
   announcement.
2. All 17 chart types either use the primitive or document why they opt out (meter/kpi); the
   ad-hoc line/heatmap/histogram tooltips are removed.
3. Pure logic (nearest-point, traversal, announced text) is unit-tested; geometry-dependent
   visuals are verified via a recorded Playwright/docs check.
4. Each adopting chart's meta documents the keyboard interaction.
5. Root gate + drift check exit 0.
