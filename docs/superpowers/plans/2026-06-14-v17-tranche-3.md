> do I make this match my brand without fighting the system?" The answer is the three-tier token model

>
> Current-state assumptions — RE-VERIFY before starting:
>
> - Three-tier tokens (from `CLAUDE.md` "Token Architecture"): primitive (`--cascivo-color-blue-500`)
>   → semantic (`--cascivo-color-accent`) → component (`--cascivo-button-bg`). Themes override the
>   SEMANTIC layer; users override COMPONENT tokens for per-component brand. RE-VERIFY real token
>   names exist in `packages/tokens/src/` before putting them in copyable snippets.
> - `data-theme="light|dark|warm|…"` applies a theme to ANY element (scoped). 10 themes in
>   `packages/themes/src/*.css`.
> - Skill `skills/cascivo-create-theme/` generates a full brand theme; MCP `create_theme` tool in
>   `@cascivo/mcp` does the same for agents. RE-VERIFY the skill folder name and the MCP tool name.
> - `skills/cascivo-extend/` covers customizing a component beyond tokens. Mention it for "I need to go
>   further than colors."
> - `apps/docs/src/pages/WhyCascadePage.tsx` — claim 32 lands here.

**Goal:** A `BrandCustomization` section (anchor `#customize`) on `/guides` that:

1. Explains the three-tier token model in one diagram-like block (primitive → semantic → component).
2. Shows the **three-line** "change a button's brand color" override as the headline example — real
   tokens, copyable.
3. Shows `data-theme` scoping (brand one subtree without touching the rest of the app).
4. Routes "I want a whole brand theme" to the `cascivo create-theme` skill + MCP `create_theme` tool,
   and "I need to change behavior, not just color" to the `cascivo-extend` skill.
5. Renders with zero horizontal overflow at 320–414px.

**Architecture:** New `apps/landing/src/pages/guides/BrandCustomization.tsx`, content in
`guides/data.ts`, styles in `landing.css`, claim 32 in `WhyCascadePage.tsx`. No package changes.

**Tech stack:** unchanged. Mobile-first CSS. `useSignals()` only if a signal is read.
