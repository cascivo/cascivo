# v33 — Tranche 7: Bench Render-Commit Repair + Regenerate Results

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the re-renders benchmark produce real data. `apps/bench/results/results.json` → `renders` currently holds all zeros (`{cascade:0, shadcn:0, carbon:0}` for all eight scenarios), which is why the Performance "Re-renders" chart had no bars. Diagnose why `window.__commits` never increments, repair any wiring defect, re-run the render suite, and regenerate `results.json` so cascivo reads far lower commit counts than shadcn/ui for keystroke/toggle scenarios.

**Architecture:** The render suite (`apps/bench/runner/src/renders.ts`) launches each bench app, reads `window.__commits` before and after a scenario op, and records the difference. Each app wraps its tree in a React `Profiler` whose `onRender` does `window.__commits += 1` (`apps/bench/app-*/src/harness.tsx`). All-zero means either the Profiler isn't wrapping the interactive subtree, the op doesn't trigger a commit in the harness route, or the committed data is simply stale (the suite was never re-run after a regression).

**Tech Stack:** Playwright (existing bench runner), React Profiler, Vite dev servers per app. Heavy: builds/serves three apps and drives them headlessly.

> **Environment caveat:** If the full Playwright render suite cannot run in this execution environment (no browser, no dev server), deliver the harness diagnosis + any wiring fix and **document the hand-off** — do not hand-edit `results.json`. The T6 graceful empty state remains the user-facing behaviour until the suite runs in CI. Mark the relevant Definition-of-Done item as "deferred to CI" in the roadmap.

### Task 1: Diagnose the all-zero commit count

- [ ] **Step 1: Confirm the data.** Inspect `apps/bench/results/results.json` → `renders`; verify every scenario is `{cascade:0, shadcn:0, carbon:0}`.

- [ ] **Step 2: Read the harness wiring.** In each `apps/bench/app-*/src/harness.tsx`, confirm `window.__commits = 0` at module load and the `<Profiler onRender={() => { window.__commits += 1 }}>` wrapper.

- [ ] **Step 3: Confirm the Profiler wraps the _interactive_ subtree.** Read each app's entry/`main.tsx` and the scenario routes. The bug is almost always that the Profiler wraps a static shell while the scenario op mutates a subtree mounted _outside_ the Profiler, OR each scenario navigates to a fresh route where the harness/Profiler isn't applied. Verify the component the scenario op interacts with is a descendant of the `<Profiler>`.

- [ ] **Step 4: Confirm the runner's read timing.** In `apps/bench/runner/src/renders.ts`, the runner waits `body[data-bench-ready="1"]`, runs `scenario.setup`, waits 100 ms, reads `before`, runs `scenario.op`, waits 200 ms, reads `after`. Confirm `scenario.op` actually triggers React state changes in the harnessed app (not a no-op selector miss) and that 200 ms is enough for the commits to flush. Check whether `window.__commits` is reset by any per-route remount between `before` and `after`.

- [ ] **Step 5: Record the root cause** in the commit message / a short note: stale-data-only (re-run fixes it) vs. a structural wiring bug (Profiler placement / route remount / op selector).
