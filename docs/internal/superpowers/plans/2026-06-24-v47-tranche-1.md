# v47 — Tranche 1: Benchmark Fixture + Perf Budget + Regression Guard

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Lock the performance target with a test **before** touching any tokenizer code. (1) Extract the
large-document generator from `scripts/bench-large-doc.mjs` into a reusable TS fixture (`makeMarkdownDoc(lines)`)
so the bench and the test share **one** doc generator producing mostly-unique Markdown lines (a real doc is not a
repeated block). (2) Add a **test-only** module-level counter to `engine/tokenize.ts` that counts per-line
`tokenize()` calls, plus a reset — zero cost in production. (3) Add a **deterministic** Vitest perf-regression
test that renders a `CodeEditor` over a 50,000-line doc, stubs line height, resets the counter, triggers a render,
and asserts lines-tokenized-per-render is **window-bounded** (`≤ visibleRows + OVERSCAN*2 + k`) — **not** a
wall-clock millisecond assertion. (4) Document the enforced budget in `PERFORMANCE.md`.

**Architecture:** Today `code-editor.tsx` calls `tokenizeDocument(getGrammar(language), highlightText.value)` over
the **full** document every render; `Highlight` does the same. Windowing constants are `VIRTUALIZE_THRESHOLD =
1000` and `OVERSCAN = 12`; the per-line memo is bounded `MAX_CACHE = 5000`. Measured keystroke re-tokenize is
~106 ms at 10k lines and ~587 ms at 50k. So this test **must FAIL against today's code** (counter ≈ 50000 over a
50k-line doc) and is expected to pass after **T3** switches the view path to `tokenizeRange`. The counter is the
deterministic proxy for "is per-render work O(viewport)?" — Decision 8 of the master plan. The fixture is the
single source of truth for the doc shape so the bench numbers and the test budget describe the same document.

**Tech Stack:** owned TypeScript in `@cascivo/editor` (no new runtime deps); the real `tokenizeDocument` + built-in
`markdown` grammar; `@cascivo/core` signals (the editor is signals-only — no `useState`/`useEffect`); Vitest +
@testing-library/react; the existing `stubLineHeight` helper pattern from `perf.test.tsx`; vite+ (`vp`) for
check/test.

### Task 1: Study the current bench + the counter seam

**Files:** `packages/editor/scripts/bench-large-doc.mjs`, `packages/editor/src/engine/tokenize.ts`, `packages/editor/src/editor/code-editor/perf.test.tsx`

- [ ] **Step 1:** Read `makeDoc(lines)` in the bench — note the `BLOCK` template, the unique `#${n}` suffix per
      line, and that blank lines / ```` ``` ```` fence delimiters are kept **verbatim** (they are state-significant;
      padding them would change cross-line grammar state). The extracted fixture must preserve this exactly so the
      output is byte-identical to today's bench doc.
- [ ] **Step 2:** Read `tokenize()` in `tokenize.ts` — it is the **single** per-line entry (`tokenizeDocument`
      loops over it). A counter incremented at the top of `tokenize()` counts exactly the lines tokenized this
      pass. Note the `MAX_CACHE = 5000` memo: a cache hit still calls `tokenize()`, so the counter measures
      *invocations*, which is what "lines tokenized per render" means for the budget.
- [ ] **Step 3:** Read `stubLineHeight(px)` in `perf.test.tsx` (sets a fixed line-box height so the windowing math
      is deterministic in jsdom) and confirm the test reads `VIRTUALIZE_THRESHOLD` / `OVERSCAN` from the source,
      not magic numbers. Confirm there is no existing tokenized-line counter.

### Task 4: Write the deterministic perf-regression test

**Files:** `packages/editor/src/engine/tokenize-perf.test.ts`

- [ ] **Step 1:** Import `makeMarkdownDoc` from the fixture, `__tokenizeCount`/`__resetTokenizeCount` +
      `clearTokenizeCache` from `tokenize.ts`, `VIRTUALIZE_THRESHOLD`/`OVERSCAN` from their source, and the
      `stubLineHeight` pattern from `perf.test.tsx`. Build a 50,000-line doc once.
- [ ] **Step 2:** Render `<CodeEditor value={doc} language="markdown" />`, `stubLineHeight(20)`, `clearTokenizeCache()`,
      then `__resetTokenizeCount()`, and trigger a render/scroll so the view path runs (scroll the viewport a few
      rows, or re-render). Read `__tokenizeCount()` after.
- [ ] **Step 3:** Compute `visibleRows` from the stubbed viewport height / line height and assert
      `__tokenizeCount() <= visibleRows + OVERSCAN * 2 + k` where `k` is a small constant (e.g. 8) for boundary
      threading slack. **No millisecond assertion** — deterministic across machines/CI.
- [ ] **Step 4:** Document in a comment that this test **fails today** (today the view path is `tokenizeDocument`
      over the full doc → count ≈ 50000) and is expected to go **green after T3** swaps in `tokenizeRange`. If the
      tranche is landed test-first, mark it `it.fails(...)` or `test.skip` with a `// UNSKIP IN T3` note so the
      suite stays green until the fix lands — choose per the executing-plans sub-skill's test-first convention.
