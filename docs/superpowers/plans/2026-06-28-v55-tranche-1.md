# v55 — Tranche 1: Template Item Type & Schema

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Establish the contract every later tranche reads: a **`template` registry item type** and a `TemplateMeta`
shape, modeled so a template is *just a `RegistryItem`* (M-1/M-2). A template references its components via
`registryDependencies` and ships its own page/fixture source via `files[].target`; gallery metadata lives in `meta`.
**No new transport, no parallel manifest** — this is the firm decision that keeps templates from forking the registry
(ROADMAP-V55 Q2 / Master Decision 1).

**Architecture:** three pieces, all inside `@cascivo/registry`.
- **Type** (`src/types.ts`): add `'template'` to `RegistryItemType`. A template is a `RegistryItem` where
  `registryDependencies` lists its components, `files` holds its *own* novel source (page/fixtures/assets), and
  `meta` carries a `TemplateMeta`.
- **TemplateMeta** (`src/types.ts` + `src/template.ts`): `{ intent, framework, category, screenshots: {light,dark,alt}[],
  demoUrl?, fileRoles: Record<targetPath, 'page'|'fixture'|'asset'|'config'>, pages?: {name,target}[] }` — enough for
  the resolver (T2) and the gallery (T5) without a new file format.
- **Validation** (`src/template.ts`): a type guard + a validator that, for `type:'template'`, requires a `TemplateMeta`,
  ≥1 `registryDependencies` **or** ≥1 page file, ≥1 screenshot, and a `license`. Extend the JSON schema to match.

**Tech Stack:** `packages/registry/src/{types,template,index}.ts`, `packages/registry/schema/registry-item.v2.json`.
Pure TypeScript + JSON Schema. No runtime dependency.

### Task 2: Validation + type guard

- [ ] **Step 1:** `isTemplateItem(item: RegistryItem): boolean` → `item.type === 'template'`. `asTemplateMeta(meta:
