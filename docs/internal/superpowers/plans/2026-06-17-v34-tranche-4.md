# v34 — Tranche 4: Trusted-Publishing GitHub Actions Workflow

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Enable and convert the release workflow to **npm OIDC trusted publishing** — no `NPM_TOKEN`, automatic provenance. Document the one-time npmjs.com trusted-publisher setup and the first-publish bootstrap in a release runbook.

**Architecture:** The existing `.github/workflows/release.yml.disabled` already uses `changesets/action@v1` (open/update a "Version Packages" PR while changesets accumulate; publish when merged). It is converted to trusted publishing by: upgrading npm to ≥ 11.5.1, adding `registry-url`, removing the `NPM_TOKEN` env, and ensuring provenance. The OIDC token is minted by GitHub via `id-token: write`; `changeset publish` shells out to `npm publish`, which auto-detects OIDC.

**Tech Stack:** GitHub Actions, `changesets/action@v1`, `actions/setup-node`, npm ≥ 11.5.1, pnpm 11.5.2.

### Task 1: Enable and convert the workflow

**Files:** rename then edit `.github/workflows/release.yml`

- [ ] **Step 1:** `git mv .github/workflows/release.yml.disabled .github/workflows/release.yml`.
- [ ] **Step 2:** Keep the existing structure (trigger `push: branches: [main]`, `concurrency`, the `changesets/action@v1` step running `version: pnpm run version-packages` / `publish: pnpm run release`). Confirm permissions are exactly:

  ```yaml
  permissions:
    contents: write # changesets commits + tags
    pull-requests: write # the Version Packages PR
    id-token: write # OIDC for trusted publishing
  ```

- [ ] **Step 3:** Pin the npm registry on `actions/setup-node` and keep pnpm caching:

  ```yaml
  - uses: actions/setup-node@v4
    with:
      node-version: 22
      cache: pnpm
      registry-url: 'https://registry.npmjs.org'
  ```

- [ ] **Step 4:** Add an npm-upgrade step **before** the changesets action (Node 22 ships npm 10.x; OIDC + provenance need ≥ 11.5.1):

  ```yaml
  - name: Upgrade npm for trusted publishing
    run: npm install -g npm@latest
  ```

- [ ] **Step 5:** Remove the `NPM_TOKEN` line from the `changesets/action@v1` `env:`. The env should be only:

  ```yaml
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
    NPM_CONFIG_PROVENANCE: true
  ```

  Do **not** add `NODE_AUTH_TOKEN`/`NPM_TOKEN` — trusted publishing supplies credentials via OIDC.

- [ ] **Step 6:** Add a top-of-file comment block summarizing the trusted-publishing contract and pointing to `docs/RELEASING.md` (npmjs.com per-package setup + first-publish bootstrap are manual prerequisites).

- [ ] **Step 2:** Document the **first-publish bootstrap** (Decision 7): all ten names are unpublished. If npm requires a package to exist before attaching a trusted publisher, perform a one-time bootstrap publish per package using a short-lived **granular automation token** (least privilege, delete immediately after), then attach the trusted publisher and run all future releases tokenless. If npm allows pre-creation trusted-publisher config for the org, skip the bootstrap. The workflow file never contains a token either way.
- [ ] **Step 3:** Document the steady-state flow:
  1. PRs land on `main` carrying `.changeset/*.md` entries (authored via `pnpm changeset`).

  3. Merging that PR triggers the workflow's publish path (`changeset publish`) → packages publish to npm with provenance, tags + GitHub release notes are created.
- [ ] **Step 4:** Note verification: after a release, `npm view <pkg>` shows the new version and a provenance/attestation badge appears on the npm package page.
