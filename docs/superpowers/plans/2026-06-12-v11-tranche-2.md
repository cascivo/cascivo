> Current-state assumptions — RE-VERIFY before starting:
>
> - T1 is merged: `@cascade-ui/registry` exists; per-item JSON (incl. `@<version>` pins) is
>   published under `/r/`; the CLI has a `registry build` command.
> - `packages/cli/src/utils/config.ts`: `CascadeConfig = { registry, outputDir, theme }`,
>   env overrides `CASCADE_REGISTRY` / `CASCADE_OUTPUT_DIR`.
> - `commands/add.ts` fetches one URL, `findComponent` exact+suffix match, writes to
>   `outputDir/<name>/`, auto-installs deps. `commands/update.ts` = two-way `diffLines`
>   against latest + confirm. No lockfile anywhere.
> - CLI tests exist (`add.test.ts`, `add.integration.test.ts`, `list.test.ts`,
>   `generate.test.ts`) — learn their fixture/mocking style and follow it.

**Goal:** `cascade add @ns/thing`, `cascade add <url>`, and
`cascade add owner/repo/thing#v1` all work, registry dependencies resolve recursively across
registries, every install is recorded in `cascade.lock`, and `cascade update` three-way
merges upstream changes into locally-edited files instead of overwriting them.

**Architecture:** All resolution logic lives in a new `packages/cli/src/utils/resolve.ts`
(addressing → registry → item) on top of one injectable fetch wrapper
(`utils/http.ts` — cache, retries, offline errors). Lockfile module `utils/lock.ts`.
Merge module `utils/merge.ts` wrapping a diff3 dependency (decision 9). Commands `add`,
`update`, `list` are extended; new `search` and `view` commands added.

**Tech stack:** unchanged; `node-diff3` (or equivalent tiny diff3 lib — evaluate at
implementation) added to the CLI package only.

### Task 4: Three-way-merge `update`

**Files:**

- Create: `packages/cli/src/utils/merge.ts` (+ exhaustive tests)
- Modify: `packages/cli/src/commands/update.ts`
- Modify: `packages/cli/package.json` (diff3 dep)

- [ ] **Step 1:** Evaluate and pin a minimal diff3 lib (criteria: no transitive deps,
      typed or trivially typeable, conflict-marker output). Wrap it:
      `merge(base, local, upstream)` → `{ text, conflicts: number }` with standard
      `<<<<<<< local / ======= / >>>>>>> upstream` markers.
- [ ] **Step 2:** `update <item…>` flow: read lock → fetch
      `r/<name>@<lockedVersion>.json` files as **base**, latest as **upstream**, disk as
      **local** → if base fetch 404s (pruned third-party registry, risk #2) degrade to
      today's two-way diff with an explicit warning → show per-file summary (clean merge /
      N conflicts / unchanged) → confirm → write + update lock (version + new hashes; on
      conflicts, lock records upstream version but flags `"conflicted": true` until a
      subsequent clean update or re-add).
- [ ] **Step 3:** `update --check`: list outdated items (locked vs latest version), exit 1
      if any — CI-friendly.
- [ ] **Step 4:** Merge tests (the heart of pitch claim #1):
  - local edit + upstream edit in different regions → clean merge containing both
  - same-region collision → conflict markers, file count reported
  - no local edits → fast-forward without diff3
  - no lock entry → falls back to legacy two-way behavior with warning
- [ ] **Step 5: Commit** — `feat(cli): version-aware three-way merge update via cascade.lock`

## Tranche exit criteria

1. All four address forms install from fixture registries; closure resolution crosses
   registries; `--dry-run` prints the plan.
2. `cascade.lock` written on add; `list --installed` uses it; pre-v11 projects unaffected.
3. The three-way merge scenario matrix (Task 4 Step 4) is green — including the
   locally-customized + upstream-changed clean merge that shadcn cannot do.
4. `search`/`view` work across ≥2 fixture registries offline (injected fetch).
5. No live-network calls in CI tests; root gate + drift check exit 0.
