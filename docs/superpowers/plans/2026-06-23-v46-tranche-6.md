# v46 — Tranche 6: Markdown Hardening, Perf/Wrap Validation, Docs & Gate

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Finish the gap-analysis closure. (1) **Harden the Markdown grammar** for a notes surface (finding
**#1** — the grammar exists but is basic; add the constructs notes actually use, with tests). (2) **Validate the
items already shipped** — line **windowing/perf** (#7) with a large-doc test and an honest note on the wrap-mode
windowing hole, and the **`wrap`** control (#9). (3) **Land all integration + docs:** EditorPage demos, component
meta, registry entry, i18n finalization, README/CHANGELOG/version bump, roadmap status — then run `pnpm regen`,
the drift gate, the full CI gate, and a grep sweep proving every new surface is wired.

**Architecture:** `grammars/markdown.ts` uses `createRuleGrammar` with a `default` and a `fence` state (fenced
code already carries state across lines). Hardening adds rules for the constructs a notes/wiki app shows. The
tokenizer already memoizes per line and windows large docs (`VIRTUALIZE_THRESHOLD = 1000`); validation is a test
+ a documented limitation, not a rewrite. Embedded-language highlight *inside* fenced blocks (delegating the fence
body to another grammar) is a **documented stretch goal**, not required for v46 — the rule-grammar model colors
the fence body uniformly today, which is acceptable for the surface.

**Tech Stack:** `createRuleGrammar` rules; Vitest for grammar + perf tests; the docs app (`EditorPage`, Preact);
`code-editor.meta.ts`; `registry.json` (generated where applicable); `builtin.editor` i18n; `pnpm regen` + the
full `pnpm ready` gate.

### Task 1: Harden the Markdown grammar

**Files:** `packages/editor/src/grammars/markdown.ts`, `packages/editor/src/grammars/grammars.test.ts`

- [ ] **Step 1:** Add rules (order matters — earlier wins) in the `default` state for: **task lists**
      (`^\s*[-*+]\s+\[[ xX]\]\s` → `operator`/`boolean` for the checkbox), **setext headings**
      (a line that is all `=` or `-` under text — best-effort within the line-based model), **horizontal rules**
      (`^\s*(?:[-*_])(?:\s*\1){2,}\s*$` → `comment`/`punctuation`), and **strikethrough** (`~~[^~]+~~` → a
      suitable kind, e.g. `comment` or `operator`). Keep existing heading/list/quote/inline-code/strong/emphasis/
      link rules intact.
- [ ] **Step 2:** Verify fenced code still carries state across lines (the `fence` push/pop) and that the closing
      fence pops. Leave the fence body uniformly colored (embedded-language delegation is a documented stretch
      goal — add a `// NOTE:` comment, do not implement here).
- [ ] **Step 3:** Add `grammars.test.ts` cases: a heading, a `**bold**`/`*italic*`/`` `code` ``/`~~strike~~` line,
      an unordered + ordered list item, a `- [ ]`/`- [x]` task item, a blockquote, an HR, and a fenced block (open
      → body → close) — asserting the expected token kinds and that values concatenate back to the line
      (losslessness, the grammar contract).

### Task 2: Validate windowing/perf and the wrap control

**Files:** `packages/editor/src/editor/code-editor/perf.test.tsx`, `packages/editor/readme.body.md`

- [ ] **Step 1:** Add a test that renders a 5,000-line document and asserts that **only a windowed slice** of rows
      is in the DOM (count `.line` spans ≪ 5,000) while the textarea holds the full text — proving windowing holds
      at scale. (Set the measured `lineHeight`/viewport via the component's normal path or a deterministic stub.)
- [ ] **Step 2:** Confirm `wrap` toggles soft wrap and that windowing correctly **disables** under `wrap` (rows are
      variable-height) — assert the non-windowed path renders all rows when wrapping. Document this trade-off
      (wrap disables windowing) in the README so large **wrapped** Markdown is a known limitation, not a surprise.
- [ ] **Step 3:** No perf regression in the highlight path (rAF debounce still in place); the test must be
      deterministic (no real `requestAnimationFrame` timing dependence — use the synchronous fallback path).
