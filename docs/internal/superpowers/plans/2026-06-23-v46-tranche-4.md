# v46 — Tranche 4: Save Keybinding + Extensibility Seam (Keymap + Decorations)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Land the `Mod-S` **save** hook (gap-analysis **#5**) and turn the two seams T1/T3 built internally — the
keymap and decorations — into a small, **documented public extension API** (gap-analysis **#10**). Add an
`onSave` prop bound to `Mod-S` (suppressing the browser's save dialog), a public `keymap` prop merged over the
built-ins, and a public `decorations` provider prop (the exact shape search already emits). This is deliberately
**just two seams** — no plugin lifecycle, no transaction filters; that is engine territory and out of scope.

**Architecture:** T1's keymap is already a `Map<chord, Command>`. Public extension = merge a user `keymap` over
the built-in map (user wins on conflict) before dispatch. Save is a built-in command that calls `onSave(text)`
and returns handled (so `preventDefault` stops the OS dialog) — a no-op when `onSave` is absent. T3 already
threads decorations into `renderRows`; public extension = accept a `decorations` provider
`(text: string) => Decoration[]` (or a static `Decoration[]`), merged with the internal search/bracket
decorations. Both props are optional and additive — default render is unchanged.

**Tech Stack:** owned TypeScript; `@cascivo/core` signals; `@cascivo/i18n` (no new strings expected — save is
keyboard-only, host app renders any "Saved" UI); Vitest + Testing Library. No banned hooks.
