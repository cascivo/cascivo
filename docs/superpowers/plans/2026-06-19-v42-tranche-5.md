# v42 — Tranche 5: Web Awesome Study Doc + Intent-Router + Final Gate

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Land the docs, adopt Web Awesome's one fresh AI idea, and run the full gate. Specifically:

1. **Document** the study (this roadmap) and the new surfaces (the three components, the three core hooks, the
   `formatBytes` formatter).
2. **Adopt the `choosing-components` decision tree** — Web Awesome's v3.9.0 AI feature that helps agents "pick
   the right component by intent instead of guessing from names." cascivo **already** has richer data: a typed
   **`intent`** block on every `component.meta.ts` (`whenToUse`/`whenNotToUse`/`related`, enforced by
   `intent-completeness.test.ts`). The work is to **surface** that existing data as a consolidated
   **intent-router** resource (an `@cascivo/mcp` tool + a skill section / generated artifact) — not to author a
   new taxonomy. cascivo is ahead (it also has an MCP server, which Web Awesome lacks); this closes the one
   presentation gap.
3. **Gate:** `pnpm regen`, drift, the full CI suite, and a grep sweep proving `comparison`/`qr-code`/
   `relative-time` reached every registration surface.

**Architecture:** Read `@cascivo/mcp` to see how existing tools enumerate the registry/metas (e.g.
`list_components`/`get_component`), then add or extend a tool that maps an **intent phrase → ranked
components** from the existing `intent` fields. Keep it a thin read over the metas — single source of truth
stays `component.meta.ts`. Mirror the data into a skill section / generated artifact if the existing skills are
generated from the registry.

**Tech Stack:** Markdown docs; `@cascivo/mcp` (existing registry-reading tools); the `intent` data already in
`_all-metas.ts`; `pnpm regen` + the full `vp` gate.

> v3.9.0 `choosing-components` skill. So this task **enhances** rather than builds: `select.ts` now returns
> `whenNotToUse` + `related` in each result so agents can disqualify wrong picks (the plan's intent). The three
> new components carry `intent` and flow into the tool automatically via `context:generate` (`context.json`).
